# household-ledger — 제품 요구사항 정의서 (PRD)

> 생성: `/forge` 무인 파이프라인. 시작 1회 인테이크로 박은 전제 + 시니어 PM 자문자답.
> **이 PRD의 빈칸은 에이전트가 사용자 대신 내린 결정이다 — 본문은 결정으로 쓰되, 검증 안 된 가정은 §11에 모았다.**

---

## 1. 배경과 문제 정의

부부·가족은 돈을 **함께** 쓰는데 가계부는 **따로** 쓴다. 각자 앱·엑셀·메모로 기록하니 "이번 달 **우리 집**이 얼마 썼나"가 어디에도 합산되지 않는다. 모임통장·공유 엑셀은 (a) 입력이 번거롭고 (b) 카테고리·통계가 약하며 (c) 누가 무엇에 썼는지 멤버 단위로 안 보인다.

핵심 문제: **여러 사람이 한 가계부를 빠르게 같이 기록하고, 한 화면에서 합쳐 보는 경험의 부재.**
- 입력이 느리면(앱 켜고 여러 화면 거치면) 안 적고, 안 적으면 가계부는 죽는다 → **입력 마찰이 1순위 적.**
- 합산·멤버별 가시성이 없으면 그냥 "각자 개인 가계부 2개"와 다를 게 없다 → **공유 가시성이 차별점.**

대상은 외부 연동(은행·카드) 없이 **수동 입력**으로 성립하는 가벼운 공유 가계부다(인테이크 확정).

> **시장 신호(시장조사 근거 → `plan-decisions.md`):** 뱅크샐러드가 2025.8 부부 공동 자산관리('우리집 돈 관리')를 종료해 *신뢰형 부부 가계부*의 메이저 공백이 생겼다(기회). 동시에 "공유 자산관리는 리텐션이 어렵다"는 경고이기도 하다 — 채택의 1순위 동기는 **투명한 공동 지출 기록**이지 자산 통합·예산이 아니다. 경쟁(Shareroo·함쓰·BuBoo)의 공통 핵심은 ① "누가 썼나" 멤버 귀속 ② 실시간 공유 ③ 빠른 입력이다.

## 2. 목표 및 성공 지표

**제품 목표:** 가족이 *마찰 없이 같이* 기록하고, 가구 단위 지출을 *한눈에* 본다.

| 지표 | 정의 | 목표(가설) |
|---|---|---|
| 입력 마찰 | 거래 1건 입력 완료까지 탭 수 / 소요 시간 | **핵심경로 ≤ 3 탭 / ≤ 5초** (업계 통념: 한 건 로깅 5초 넘으면 1주 내 이탈) |
| 입력 활성 | 주 4회 이상 거래를 입력하는 가구 비율 | 활성 가구 40%+ |
| 공유 가시성 | 홈 진입 시 "이번 달 가구 총지출"이 0클릭에 보임 | 100%(설계 보장) |
| 리텐션 | 가구 단위 D30 재방문 | 35%+ |
| 통계 도달 | 월 1회 이상 통계 화면 도달 멤버 비율 | 50%+ |

> 절대 수치는 **가설**이다(런칭 후 보정). 측정 가능한 *형태*로 못박는 데 의미가 있다 — "많이 쓰면 성공"은 거부.

## 3. 타깃 사용자와 사용 시나리오

**핵심:** 2~5인 가족/부부(맞벌이·외벌이). 한 명이 가계부를 만들어 **소유자(owner)**, 나머지는 초대받은 **멤버**. 전원 모바일로 *그때그때* 입력.

- **시나리오 A(즉시 입력):** 마트에서 장 보고 나오며 폰으로 3만원 지출을 10초 안에 적는다.
- **시나리오 B(돌아보기):** 주말에 배우자가 "이번 달 외식 얼마 썼지?" → 통계에서 카테고리·멤버별 확인.
- **시나리오 C(고정비):** 월세·구독을 매달 적기 귀찮다 → 반복 거래로 한 번 등록해 자동 반영.

## 4. 기능 요구사항

우선순위 근거: **입력(F1)·내역(F2)이 없으면 제품이 성립 안 함 → 코어. 통계(F3)·반복(F4)은 "공유 가계부"의 차별·지속 동력 → MVP 필수. 멤버(F6)는 '가족 공유' 정체성 자체 → 최소형으로 MVP. 예산·한도는 인테이크에서 안 골랐고 코어 검증 후로 미뤄도 제품이 선다 → 나중.**

### MVP (이번 범위)

- **F1 거래 입력 (빠른 입력이 생사)** — 사용자가 금액·유형(수입/지출)·카테고리·날짜·입력 멤버·**공동/개인 구분**·(선택)메모·(선택)반복을 넣고 저장하면, 시스템은 거래를 추가하고 홈/내역/통계 합계를 갱신한다. **핵심 경로 3탭**(금액 숫자패드 → 카테고리 아이콘 1탭 → 저장); 날짜=오늘·멤버=나로 프리필, 메모는 선택. 입력 마찰이 이탈 1순위라 최우선.
  - **빠른 재입력(MVP):** 최근 거래·즐겨찾기(자주 쓰는 항목)를 1탭 복제 입력 — 수동입력 MVP에서 SMS 자동입력의 부재를 메우는 핵심 장치(시장조사 근거).
  - **공동/개인 구분(MVP, 경량):** 거래마다 '공동 지출 / 개인 지출' 토글 1개. 부부의 "공동비 vs 내 용돈" 정서에 직접 부응(저비용·고적합). 통계·내역에서 필터.
  - 엣지: 금액 0 또는 빈값 → 저장 비활성. 카테고리 미선택 → 저장 비활성. 미래 날짜 허용(예정 지출). 음수 입력 차단(유형 토글로 부호 결정).
- **F2 거래 내역 목록** — 사용자가 기간/카테고리/멤버로 필터하면, 시스템은 해당 거래만 최신순으로 보여준다. 항목을 누르면 상세로 이동, 상세에서 수정/삭제.
  - 엣지: 필터 결과 0건 → 빈 상태 안내. 삭제 → 확인 후 합계 갱신. 같은 날짜 다건 → 날짜 그룹 헤더.
- **F3 월간 통계** — 사용자가 월을 선택하면, 시스템은 그 달의 총수입·총지출·잔액, 카테고리별 비중(도넛), 멤버별 지출(바)을 보여준다. 월을 좌우로 이동.
  - 엣지: 데이터 없는 달 → 빈 차트 안내. 수입/지출 탭 전환.
- **F4 반복·고정 거래** — 사용자가 금액·카테고리·주기(매월/매주)·시작일·멤버로 규칙을 등록하면, 시스템은 다음 발생일을 계산해 보여주고 발생일에 거래를 자동 생성(자동 생성분은 내역에서 '반복' 배지). 규칙 수정/삭제/일시중지.
  - 엣지: 일시중지 → 다음 발생 안 함. 말일 처리(31일 규칙을 짧은 달에) → 해당 월 말일로. (자동 생성의 실제 스케줄링은 개발 영역, 디자인은 규칙 CRUD·다음 발생일 표시까지.)
- **F5 홈 대시보드** — 진입 즉시 이번 달 가구 총지출/총수입/잔액 + 최근 거래 몇 건 + 빠른 입력(FAB). 하단 탭으로 내역·통계·반복 이동.
- **F6 멤버 귀속·공유 컨텍스트 (귀속은 코어)** — **거래마다 작성자(멤버) 귀속은 코어다.** 공유 가계부의 존재 이유가 "누가 썼나"이고(시장조사: BuBoo '객체'·함쓰 멤버표기), 이게 없으면 1인 가계부와 차별이 0이다. 멤버를 색 아바타로 표시, 입력 시 멤버 선택, 통계에서 멤버별 비교. **단 멤버 초대·권한·온보딩은 나중으로 미룬다 — '최소형'은 *귀속*이 아니라 *초대/권한*을 미루는 것이다.**

### 나중 (Next) — 로드맵 우선순위 순
1. **정산·더치페이 (로드맵 1순위)** — "각자 카드로 결제 후 나중에 정산"은 부부/동거의 매우 흔한 흐름(시장조사: Shareroo·토스·카뱅의 핵심 무기). 균등/금액 정산만이라도 빠른 후속. *MVP 보류이되 최상단.*
2. **실시간 푸시 알림** — 상대가 입력하면 즉시 반영·알림(공유의 "함께 쓴다" 체감). 단 알림 피로 제어 필요.
3. 카테고리별 **예산·한도**와 초과 경고 / 멤버 **초대·권한** 관리 / 영수증 사진 첨부 / 자산·계좌 잔액 관리 / 데이터 CSV 내보내기 / 사용자 정의 카테고리 / 프라이버시 공개범위(용돈 가리기).

## 10에 대응 — 범위 외 (Out of Scope)
은행·카드 **자동 연동** / 영수증 **OCR** 자동 입력 / 투자·대출·보험 관리 / 다국어(한국어 단일) / 웹 데스크톱 전용 레이아웃(모바일 우선, 데스크톱은 나중).

## 5. 데이터 모델 (개념 — 물리 스키마는 범위 밖)

핵심 엔터티와 관계(텍스트 ERD):

- **Household(가구)** — id, name, base_currency(기본 KRW)
- **Member(멤버)** — id, household_id, name, role(owner|member), avatar_color
- **Category(카테고리)** — id, name, icon, kind(income|expense), is_system, sort_order
- **Transaction(거래)** — id, household_id, **member_id(작성자/귀속 — 멤버별 통계의 근거, 필수 FK)**, category_id, amount(양수), type(income|expense), date, memo, **scope(shared|personal)**, recurring_rule_id(nullable), source(manual|auto), created_at
- **RecurringRule(반복규칙)** — id, household_id, member_id, category_id, amount, type, interval(monthly|weekly), anchor_day, **end_rule(none|count:N|until:date)**, next_date, memo, status(active|paused)

관계:
- Household **1—N** Member / Household **1—N** Transaction / Household **1—N** RecurringRule
- **Member 1—N Transaction (payer/귀속) — 이 직접 관계가 "멤버별 비교"의 데이터 근거다(시장조사: 이게 없으면 공유 가치 성립 불가).**
- Member **1—N** RecurringRule / Category **1—N** Transaction / Category **1—N** RecurringRule
- RecurringRule **1—N** Transaction (발생분 생성, `source=auto`)

**시스템 기본 카테고리 세트(시장조사 근거):** 지출 13 — 식비🍚 · 카페·간식☕ · 교통🚆 · 주거·공과금🏠 · 통신📱 · 생활용품🧴 · 쇼핑·의류👕 · 건강·의료💊 · 문화·여가🎬 · 구독🔁 · 모임·경조사🎁 · 교육·자기계발📚 · 기타📦 / 수입 3 — 월급💰 · 용돈·상여🪙 · 기타수입➕. **'기타'는 미분류 흡수·통계 묶음용 필수.** 사용자 추가는 나중.

> **물리 스키마(컬럼 타입·인덱스·정규화·마이그레이션)는 개발 단계 결정 — 여기서 확정하지 않는다.** 금액은 정수 최소단위(원) 가정. **이체(transfer)는 계좌 엔터티가 범위 밖이라 MVP 제외(나중)** — type은 income|expense만.

## 6. 사용자 흐름

- **입력:** 홈 → FAB(빠른 입력) → 거래 입력 → 저장 → 홈(요약 갱신)
- **돌아보기:** 홈/하단탭 → 내역 → (필터) → 거래 항목 → 상세 → 수정/삭제 → 내역
- **통찰:** 하단탭 → 통계 → 월 이동 / 수입·지출 탭 / 카테고리·멤버 드릴
- **고정비:** 하단탭 → 반복 → 추가(폼) → 저장 → 목록 / 항목 일시중지·삭제
- **멤버 선택:** 거래 입력 화면 안에서 멤버 칩으로(별도 화면 없음)

## 7. 화면 목록 및 화면별 요구사항

대상 뷰포트: **모바일 390px**(가정). 하단 탭 4개: 홈 / 내역 / 통계 / 반복.

1. **index.html (홈 대시보드)** — 이번 달 총지출·총수입·잔액 카드 / 최근 거래 3~5건 / FAB(→입력) / 하단탭 / 멤버 아바타 줄. 상태: 정상 / 빈(거래 0).
2. **add.html (거래 입력)** — 수입·지출 세그먼트 토글 / **금액(숫자패드)** / 카테고리 **아이콘 그리드**(단일선택) / 날짜(기본=오늘) / 멤버 칩(단일선택, 기본=나) / **공동·개인 토글** / 메모(선택) / 반복 토글(켜면 주기·시작일·종료규칙 노출) / **최근·즐겨찾기 빠른입력** / 저장(유효성 충족 시 활성). 상태: 작성중 / 유효성 미충족(저장 비활성).
3. **list.html (거래 내역)** — 필터 칩(기간 단일 + 카테고리·멤버 다중 토글) / 날짜 그룹 / 거래 항목(→상세) / 반복 배지. 상태: 정상 / 필터 결과 0(빈) / 로딩.
4. **detail.html (거래 상세)** — 금액·카테고리·날짜·멤버·메모·반복출처 표시 / 수정(→입력) / 삭제(→확인→내역).
5. **stats.html (월간 통계)** — 월 이동(◀ 2026.06 ▶) / 요약(수입·지출·잔액, **전월 대비**) / 수입·지출 탭 / **카테고리 도넛(상위 5 + '기타', 최대 6슬라이스) + 리스트(금액·%)** / **멤버별 가로막대**(도넛 아님 — 누가 얼마). 색만으로 구분 금지(아이콘 병행). 상태: 정상 / 데이터 없는 달(빈).
6. **recurring.html (반복 거래)** — 규칙 목록(다음 발생일·금액·카테고리·멤버) / 활성·일시중지 토글 / 추가 폼(인라인 열기/닫기) / 삭제. 상태: 정상 / 빈(규칙 0).

인터랙션 모델(구조 결정): 세그먼트 단일선택(수입/지출) · 카테고리/멤버 단일선택 · 필터 칩 다중토글 · 활성 토글(aria-pressed) · 인라인 폼 토글 · 월 페이저(값 갱신). 다중선택 일괄액션은 이번 범위 아님.

- **톤:** 미니멀·깔끔, 신뢰감(금융앱). 레퍼런스: **토스·뉴뱅크**(인테이크 확정). 흰 여백 + 차분한 액센트 1색 + 숫자 가독성 우선.
- **대상 뷰포트:** 모바일 **390px** 우선(가정). 데스크톱·반응형은 나중.
- **다크모드:** 이번 MVP는 **라이트 우선, 다크모드 미포함**(가정 — 인테이크에서 안 받음. 토큰 2단 구조라 추후 다크 추가는 semantic 재매핑으로 가능).
- **언어:** 한국어 단일. 통화 ₩(원), 정수 표기.
- **액센트색:** 토스풍 **블루 계열**(가정) — primitive 팔레트로 두고 semantic `--color-primary`에 매핑(리스킨 가능하게).

## 9. 비기능 요구사항

- **성능/입력성:** 거래 입력 화면 즉시 반응, 합계 갱신 지연 체감 0.
- **접근성:** WCAG AA — 텍스트 대비 4.5(큰 글씨 3.0), 포커스 가시성, 역할·라벨. (토큰 대비 + axe 자동 검사로 게이트.)
- **로컬·독립:** 산출물은 외부 의존성 없는 독립 HTML(상태용 인라인 JS 허용), 브라우저로 바로 열림.
- **확장성:** 토큰 2단 구조로 추후 다크모드·리브랜드 흡수. 데이터 모델은 멤버/카테고리 증가에 선형.

## 11. 미해결 이슈와 가정 (가정 원장 — /forge가 사용자 대신 내린 결정)

> **검수 우선순위: 아래 가정이 틀렸으면 해당 섹션을 고치고 재생성하라.** 사용자가 인테이크에서 답한 것(타깃=가족공유 / 입력=수동 / MVP=입력·내역·통계·반복 / 톤=미니멀 토스풍)은 *전제*, 아래는 *에이전트 결정*이다. **시장조사로 근거가 확보된 항목은 "근거 있는 결정"으로, 미확보는 "가정"으로 구분**한다(상세 근거·출처는 `plan-decisions.md`).

**근거 있는 결정(시장조사로 승격 — `plan-decisions.md` 참조):**
- **결정 1 — 멤버 귀속을 코어로 격상(거래별 작성자).** 근거: 공유 가계부 이탈·차별의 핵심이 "누가 썼나"(Blind·BuBoo·함쓰). 기존 'F6 최소형'에서 *귀속*을 빼내 코어로.
- **결정 2 — 입력 마찰 목표 3탭/5초 + 빠른 재입력(최근·즐겨찾기).** 근거: 이탈 1순위가 입력 귀찮음, SMS 자동입력 부재를 재사용으로 보완.
- **결정 3 — 공동/개인 구분 토글을 MVP 경량 포함.** 근거: 부부 "공동비 vs 용돈" 정서, 저비용.
- **결정 4 — 통계는 도넛 상위5+기타·멤버별 가로막대.** 근거: 도넛 슬라이스 ≤6 베스트프랙티스, 멤버별은 막대가 적합.
- **결정 5 — 정산/더치페이는 MVP 보류·로드맵 1순위.** 근거: 분리재정 흐름 보편(Shareroo·토스). 예산·OCR·연동 보류는 적절 확인.
- **결정 6 — type은 income/expense만(이체 제외).** 근거: 계좌 엔터티 범위 밖.

- **가정 A — 뷰포트 = 모바일 390px.** 근거: 가계부는 폰 즉시 입력이 지배적. (데스크톱/반응형 원하면 §8·전 화면 레이아웃 변경.)
- **가정 B — 다크모드 미포함(라이트 우선).** 근거: 인테이크 미요청, MVP 범위 압축. 토큰 2단이라 추후 추가 용이.
- **가정 C — 액센트 = 블루 계열.** 근거: 레퍼런스(토스·뉴뱅크)의 신뢰형 톤. (브랜드 컬러가 있으면 `--color-primary` primitive만 교체.)
- **가정 D — (개정됨, 결정 1 참조) 멤버 *귀속*은 코어, *초대·권한·온보딩*만 나중.** 시장조사로 귀속이 코어로 승격됨.
- **가정 E — 카테고리는 시스템 기본 세트 제공, 사용자 추가는 나중.** 근거: 입력 마찰↓ 우선, 커스텀은 코어 이후.
- **가정 F — 예산·한도 기능 제외(나중).** 근거: 인테이크에서 안 고름. '절약 초점' 사용자가 아니라 '가족 공유' 사용자.
- **가정 G — 반복 거래의 실제 자동 생성 스케줄링은 개발 영역.** 디자인은 규칙 CRUD + 다음 발생일 표시 + 자동분 배지까지. 근거: 백엔드 잡 스케줄은 물리 구현.
- **가정 H — 통화 단일 ₩(원), 정수.** 근거: 국내 가족 가계부. 다통화는 범위 밖.
