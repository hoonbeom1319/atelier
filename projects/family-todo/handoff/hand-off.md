# family-todo — 인계 프롬프트 (Claude Code에 붙여넣으세요)

> 이 문서는 **대상 프로젝트의 Claude Code에 그대로 붙여넣는 브리프**다.
> 아래 디자인 산출물(chosen final)을 너의 프로젝트 컨벤션으로 구현해라.
> **디자인을 새로 발명하지 말고, 이미 확정된 흐름·토큰·컴포넌트·모션 계약을 지켜라.**
> 스택은 **Next.js(App Router) + Supabase + Vercel** 기준이지만, 컴포넌트 분해·파일 구조·스타일링 방식은 너의 프로젝트 관습을 따른다.

---

## 0. 무엇을 만드나

부부 2명이 쓰는 **공유 할 일 앱**. "흩어진 우리 둘의 할 일을 한 곳에서". 모바일 온리(아이폰 세로, 390px 기준). 파란색 계열·다크모드·**애니메이션 비중 큼**.

화면은 **3개뿐**(잔인하게 작은 MVP):
- **S1 PIN 입장** — 가족 공통 PIN 하나로 입장(개인 로그인 아님).
- **S2 할 일 목록(메인)** — 진행 중 + 완료됨(흔적) + 하단 빠른 추가.
- **S3 추가·편집** — **바텀시트**로 목록 위에 올라옴(별도 페이지 아님).

## 2. 반드시 지킬 계약

### 2-1. 디자인 토큰 (구조를 지켜라)
- **2단 구조 유지**: primitive(`--blue-500`, `--gray-0..900` 웜그레이 램프, `--amber-*`) → semantic 역할(`--color-primary`, `--color-text`, `--color-surface`, `--color-border`, `--color-accent-warm`, `--color-success`, `--color-danger`, `--gradient-bg`, `--gradient-primary` …). **컴포넌트는 semantic만 참조.**
- **다크모드**: `prefers-color-scheme` + 수동 `[data-theme]` 오버라이드 **둘 다**. (원본 tokens.css가 두 경로를 다 정의해 둠.)
- **그라데이션 2개**: `--gradient-bg`(앱 바탕 상단 글로우, `background-attachment:fixed`) / `--gradient-primary`(primary 버튼·FAB 대각). 절제된 세기 — 더 진하게 만들지 마라.
- 매핑 방법은 `token-mapping.md` 참조. Tailwind든 CSS Modules든 vanilla-extract든 **너의 방식**으로 옮기되 역할명은 보존.

### 2-2. 기능·플로우 (동작을 지켜라 — `screens.spec.js`가 기준)
- **S1**: 최초=PIN 설정 / 이후=PIN 입력. 틀리면 같은 화면에서 에러+재시도, 맞으면 S2. (실제 PIN 저장·검증은 너의 백엔드 몫.)
- **S2 카운트**: 항상 `진행 중 N · 완료됨 M` 형태로 노출.
- **추가는 1경로**: 하단 빠른 입력(텍스트만, 즉시). **날짜 단축 버튼 없음.**
- **완료 = 토글 + 보존 + 접기**: 체크하면 삭제가 아니라 "완료됨" 영역으로 이동. 다시 체크 → 진행 중 복귀. **완료됨은 기본 접힘(개수만)** + **완료 비우기**로 정리.
- **날짜는 편집에서**: 항목 탭 → S3 시트 → 날짜 토글 on/off로 붙임(있어도/없어도 같은 목록에 공존).
- **S3 검증**: 내용 비면 저장 비활성.
- **상태 망라**: 빈 / 로딩 / 오프라인(끊겨도 안 깨짐) 모두 화면으로 표현.
- 동작 계약 셀렉터(role/aria/텍스트, `.item`·카운트 텍스트·"완료됨"·"완료 비우기"·"삭제"·"날짜 추가" 등)는 `screens.spec.js`에 박혀 있다. **이식해서 green 유지**하면 회귀를 막을 수 있다.

### 2-3. 모션 (느낌을 지켜라 — PRD 1순위 중 하나)
- 추가=위에서 스프링 등장, 삭제=옆으로 밀려 접힘, 체크=스프링 팝, 시트=아래에서 emphasized 슬라이드업, 저장=토스트, 빈 상태 아이콘 바운스, PIN 오류 shake.
- ease/dur 토큰(`--ease-spring`·`--ease-emphasized`·`--dur-sheet` 등)을 그대로 옮겨 써라. `prefers-reduced-motion: reduce` **존중**(원본이 dur를 1ms로 죽임).

### 2-4. 데이터 (PRD §4-1)
- 엔터티 2개: **Household**(name, pin) 1:N **Task**(text, date *nullable*, done, completed_at). 개인 User 엔터티 없음.
- 동시편집 = **LWW(마지막 동작이 이긴다)**. PIN 공통 입장. 물리 스키마·PIN 보안은 너의 구현 판단.

## 3. 만들 순서 (권장)

1. **토큰 이식** — `token-mapping.md`대로 semantic 토큰 + 다크모드 + 그라데이션을 너의 스타일 시스템에 올린다. 색·타이포·간격·모서리·그림자·모션.
2. **공용 컴포넌트** — `component-inventory.md`의 버튼·입력·체크박스·항목행·토글·접이식·배지·빈상태·스켈레톤·배너·토스트·바텀시트를 컴포넌트화(`app.css`가 스타일 정답지).
3. **S1 PIN 입장** — 입력→검증→이동, 에러 shake. (검증은 Supabase Household.pin 대조.)
4. **S2 목록** — 데이터 바인딩(Supabase 실시간/폴링). 진행중/완료됨 접이식, 빠른 추가, 체크 토글, 삭제, 카운트, 상태(빈/로딩/오프라인).
5. **S3 바텀시트** — 항목 탭 진입, 내용·날짜 편집, 저장/삭제, 빈내용 검증.
6. **동기화** — Supabase 구독으로 두 기기 반영(원본에선 `[범위 밖]`이던 부분). LWW.
7. **검증** — `screens.spec.js`의 시나리오를 네 라우팅/셀렉터에 맞춰 이식해 green. 라이트/다크 둘 다 눈으로.
