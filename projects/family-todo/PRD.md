# PRD — family-todo (우리 가족 전용 TO-DO 앱)

> 작성: /plan (기획 파이프라인) · 상태: MVP 1차 범위 확정 · 다음: /design 인테이크
> 한 줄 요약: **부부 두 사람의 흩어진 할 일을 "우리집" 한 곳에 모으는, 우리만의 작고 따뜻한 공유 투두.**

---

## 1. 배경과 문제 정의

부부 두 사람의 할 일·메모·일정이 **여러 곳에 흩어져 있다** — 내 폰 캘린더, 내 폰 메모, 와이프 폰, 또 다른 메모…. "우리 둘의 할 일"을 한눈에 보는 **단 하나의 장소가 없다.**

- 핵심 통증은 *공유 실패로 인한 큰 사고*라기보다 **"흩어짐" 그 자체의 찝찝함**이다(중복 입력, 어디 적었는지 헷갈림, 한 곳에 안 모임).
- 애플 기본 앱(공유 캘린더·미리알림·공유 메모)으로도 *기능적으로는* 가능하지만, 이 프로젝트의 가치는 효율 경쟁이 아니라 **"우리 가족만의 것을 갖는다"는 애착·소유감**에 있다(아래 목표 참조).

## 2. 목표 및 성공 지표

이 앱의 가치 비중은 **실용(효율) 20 : 우리만의 것(애착) 80**이다. 따라서 일반적 사용자 수(DAU) 지표가 아니라 다음으로 성공을 가늠한다.

| # | 성공 지표 | 측정 방법 |
|---|---|---|
| G1 | **한 곳으로 모인다** — 흩어 적던 걸 이제 이 앱에 적는다 | 새 할 일을 캘린더·메모 대신 이 앱에 적기 시작(정성적: "더 이상 메모장에 안 적는다") |
| G2 | **둘 다 본다** — 한 명이 추가·체크하면 상대 폰에 반영된다 | 한 기기의 변경이 다른 기기에 나타나는지(공유 동작 성공) |
| G3 | **우리 거 같다** — 열었을 때 "이건 우리집 앱" 하는 느낌 | 정성적 — 디자인 방향(§8)의 판단 기준 |

> 리텐션("계속 켠다")은 별도 지표로 박지 않는다 — G2(공유)+G3(애착)이 충족되면 자연히 따라온다고 본다.

## 3. 타깃 사용자와 사용 시나리오

- **사용자: 부부 2명.** 둘 다 아이폰. 권한·역할 구분 없음(둘 다 동등). 한 명은 개발자(본인)지만 사용 맥락은 일반 사용자와 같다.
- **비사용자: 16개월 아기.** 앱을 쓰지 않는다 — 아기는 "사용자"가 아니라 **할 일의 소재**(예방접종, 어린이집 준비물 등)다.
- **사용 시나리오**
  - *(집)* 저녁에 "내일 분리수거", "기저귀 주문" 같은 걸 떠오를 때 바로 적는다.
  - *(밖)* 마트에서 장 볼 항목을 미리 적어두고, 담으면서 체크한다. 상대가 추가한 것도 같이 보인다.
  - *(공유)* 한 명이 "관리비 25일" 같은 날짜 있는 일을 적으면 상대 폰에도 떠서 둘 다 안다.

## 4. 기능 요구사항

표기: 동작은 **"사용자가 X를 하면 시스템은 Y를 한다"** 형태. 우선순위 — **[MVP]** 1차 / **[다음]** 2차 이후.

### F1. 가족 공간 입장 (PIN) — [MVP]
- 사용자가 **앱을 처음 열면**, 시스템은 **가족 PIN을 설정**하게 한다(이때 "우리집" 가족 공간이 1개 생성된다).
- 사용자가 **올바른 PIN을 입력하면**, 시스템은 **그 가족 공간의 할 일 목록으로 입장**시킨다.
- 한 번 입장한 기기는 시스템이 **입장 상태를 기억**해, 매번 PIN을 다시 묻지 않는다.
- **우선순위 근거:** 공개 URL(Vercel) 웹앱이라 잠금이 없으면 누구나 가족 데이터를 보게 된다. "우리만 들어온다"를 지키는 최소 장치. 개인 계정(신원 구분)은 불필요하므로 PIN 하나로 충분.
- **엣지 케이스:** ① 틀린 PIN → 입장 거부 + 재시도 안내. ② 빈/형식 오류 PIN 설정 시도 → 막는다. ③ PIN 분실 → MVP 미해결(§11).

### F2. 할 일 추가 — [MVP]
- 사용자가 **할 일 내용을 입력하고 추가하면**, 시스템은 **목록 맨 위(또는 정해진 순서)에 새 할 일을 추가**하고 상대 기기에도 반영한다.
- 사용자가 **날짜를 붙이면**(선택), 시스템은 그 할 일에 **날짜를 함께 저장·표시**한다. **날짜는 선택사항**이다(메모형 기본, 일정형은 날짜 추가).
- **우선순위 근거:** 흩어진 입력을 한 곳에 모으는 핵심 동작. 날짜 nullable 설계로 "캘린더에 적던 것 + 메모에 적던 것"을 한 목록에서 모두 흡수.
- **엣지 케이스:** ① 빈 내용 저장 → 막는다. ② 과거 날짜 → 허용(지난 일도 적을 수 있음). ③ 아주 긴 텍스트 → 말줄임/줄바꿈 처리(표시 규칙은 디자인 단계).

### F3. 완료 체크 / 해제 — [MVP]
- 사용자가 **할 일을 체크하면**, 시스템은 그 할 일을 **완료 상태로 바꾸고 완료 시각을 기록**하며, **"완료됨" 영역으로 이동**시킨다.
- 사용자가 **완료된 할 일을 다시 누르면**, 시스템은 **미완료로 되돌린다**.
- **우선순위 근거:** 투두의 기본. 완료 시각 기록이 "흔적"(F6)의 토대.
- **엣지 케이스:** 두 사람이 거의 동시에 같은 항목을 토글 → **마지막 동작이 이긴다(LWW)**.

### F4. 삭제 — [MVP]
- 사용자가 **할 일을 삭제하면**, 시스템은 **목록에서 제거**하고 상대 기기에도 반영한다.
- **엣지 케이스:** ① 한 명이 보고 있는 항목을 상대가 삭제 → LWW(사라짐). ② 실수 삭제 복구(undo)는 **MVP 범위 밖**(§10), 삭제 전 확인 여부는 디자인 단계 결정.

### F5. 두 기기 동기화 — [MVP]
- 한 기기에서 **추가·완료·삭제가 일어나면**, 시스템은 **다른 기기의 목록에도 그 변화를 반영**한다(근실시간 또는 앱 진입 시 최신화).
- **우선순위 근거:** "둘 다 본다"(G2)의 본체. 이게 없으면 그냥 개인 투두다.
- **엣지 케이스:** ① 오프라인에서 변경 → 재연결 시 동기화. ② 동시 변경 충돌 → **LWW**.

### F6. 완료 흔적 — [MVP]
- 할 일을 완료하면, 시스템은 그것을 **즉시 지우지 않고 "완료됨"으로 남겨 쌓이게** 한다(완료 시각 기준).
- **우선순위 근거:** B(애착) 80짜리 프로젝트의 정서 포인트 — "우리가 이만큼 해냈네"가 보임. 단, MVP에서는 **목록 보존 수준까지만**.
- **엣지 케이스:** 완료 항목이 무한정 쌓일 때의 정리(보관/숨김)는 디자인·구현 단계에서.

### F7. 완료 통계·성취 화면 — [다음]
- "이번 주 N개 완료" 같은 누적 성취를 보여주는 화면. **MVP 아님**(F3의 완료 시각을 그대로 재사용해 나중에 확장).

## 5. 데이터 모델 (개념)

> 물리 스키마(컬럼 타입·인덱스·정규화·마이그레이션)는 **범위 밖** — 개발 단계(Supabase)에서 확정한다. 여기서는 출발점만.

**엔터티 2개**

- **가족 공간 (Household)** — 공유의 울타리
  - 주요 속성: 공간 이름("우리집"), **PIN**, 생성일
- **할 일 (Task)** — 앱의 주인공
  - 주요 속성: 내용(텍스트), **날짜(nullable)**, 완료 여부, **완료 시각(nullable — 흔적)**, 생성 시각

**관계 (텍스트 ERD)**

```
가족 공간 (Household) ──< 할 일 (Task)     (1:N)
```

- **개인 사용자(User) 엔터티 없음** — PIN 공통 입장 방식이라 개인 신원을 구분하지 않는다. "누가 만들었나/누가 할까"를 따지지 않기로 한 결정(담당자 제외)과 일치.
- **날짜 nullable**이 핵심 — 메모형(날짜 없음)과 일정형(날짜 있음)이 같은 Task로 공존.
- **완료 시각** 하나로 흔적(F6)과 미래 통계(F7)를 모두 커버.

## 6. 사용자 흐름

```
[최초]  앱 열기 → 가족 PIN 설정(우리집 공간 생성) → 할 일 목록
[합류]  (상대 기기) 앱 열기 → 같은 PIN 입력 → 같은 할 일 목록 입장
[일상]  앱 열기 → 목록(최신 동기화) → 추가 / 체크 / 삭제
[추가]  목록에서 내용 입력 → (원하면) 날짜 붙임 → 저장 → 목록에 뜸 → 상대 폰에도 반영
[완료]  항목 체크 → '완료됨'으로 내려가 흔적으로 쌓임 (다시 누르면 복귀)
```

## 7. 화면 목록 및 화면별 요구사항

**대상 뷰포트: 모바일(아이폰 세로, 약 390px 폭) 단일.** 데스크톱·태블릿은 범위 밖.

| 화면 | 역할 | 주요 요구사항 | 상태(빈/로딩/에러) |
|---|---|---|---|
| **S1. PIN 입장** | 우리집 문 | 최초=PIN 설정 / 이후=PIN 입력. 입장 기억. 틀린 PIN 처리 | 에러: 틀린 PIN |
| **S2. 할 일 목록(메인) ★** | 앱의 거의 전부 | 진행 중 목록 + **완료됨 영역(흔적)**, 빠른 추가 입력, 각 항목 체크·삭제, 날짜 표시 | **빈**: 첫 화면("우리집 첫 할 일을 남겨보세요" 류 따뜻한 카피) / **로딩**: 동기화 중 / **에러**: 오프라인 |
| **S3. 할 일 추가·편집** | 입력 | 텍스트(필수) + 날짜(선택). 인라인 또는 작은 시트 | — |

> S2가 사실상 앱의 본체다. S3는 S2 안의 인라인/시트로 흡수될 수 있음(디자인 단계 결정).

## 8. 디자인 방향 (→ /design 출발 브리프)

- **정서/톤:** "우리 가족만의 것"이라는 **따뜻함·애착**이 1순위. 차갑고 사무적인 생산성 툴 느낌은 피한다.
- **색:** **파란색 계열**을 메인으로. (따뜻함과 파랑을 함께 — 차가운 코퍼레이트 블루보다 부드럽고 친근한 파랑 권장. 정확한 팔레트는 토큰 단계에서.)
- **다크모드:** **포함**(라이트/다크 둘 다). 토큰 설계 시 semantic 역할층으로 양쪽 대응.
- **★ 애니메이션·모션: 비중 크게.** 추가·완료(체크)·삭제·동기화 반영 등 **상태 변화에 풍부한 전환/마이크로 인터랙션**을 입힌다. 이는 `/design`의 5단계(하이파이)·6단계(인터랙션 검증)에서 본격 구현·검증할 핵심 요소다.
- **우선순위:** 모바일 온리(아이폰 세로).
- **레퍼런스(제안 — 확정 아님):** Things 3(정돈된 체크 모션), Linear(다크 + 부드러운 전환), TickTick/일반 가족·메모 앱의 따뜻한 톤 중 일부. 사용자가 선호 레퍼런스를 더 주면 교체.

## 9. 비기능 요구사항

- **플랫폼:** 모바일 웹(아이폰 Safari). Next.js + Supabase + Vercel.
- **동기화:** 근실시간 또는 진입 시 최신화. 충돌은 LWW.
- **프라이버시:** 가족 PIN으로 접근 차단(공개 URL이므로 필수). PIN 보관·검증의 안전한 구현은 개발 단계 책임(§11 가정).
- **성능:** 데이터 규모가 작아(둘이 쓰는 투두) 부담 낮음. 빠른 진입·즉각적 입력 반응이 중요.
- **오프라인:** 최소한 끊겨도 깨지지 않고, 재연결 시 동기화.

## 10. 범위 외 (Out of Scope)

- 담당자 지정(누가 할 일인지)
- 알림·푸시
- 월간 캘린더(달력 그리드) 뷰 — 날짜는 속성으로만, 달력 화면 없음
- 여러 목록·카테고리(장보기/집안일 분리)
- 반복 일정, 사진·첨부, 댓글
- 실수 삭제 복구(undo)
- 개인 계정·개인 로그인(Gmail 등) — PIN 공통 입장으로 대체
- 데스크톱/태블릿 레이아웃
- 완료 통계·성취 화면(§F7) — 2차 이후

## 11. 미해결 이슈와 가정

**미해결 이슈**
- **PIN 분실 처리** — 둘만 쓰고 기기가 입장을 기억하므로 빈도는 낮지만, 분실 시 재설정 경로가 MVP에 없다. 추후 보완.
- **완료 항목 누적 정리** — 흔적이 무한정 쌓일 때 보관/숨김 정책(디자인·구현 단계).
- **S3 형태** — 추가·편집을 별도 화면으로 둘지, S2 인라인/시트로 흡수할지(디자인 단계).

**가정**
- 두 사용자는 같은 가족 PIN을 안전하게 공유한다(둘 사이 신뢰 전제).
- PIN 방식은 Supabase 개인 인증을 쓰지 않으므로, 접근 제어·PIN 안전 보관(해시 등)은 **개발 단계에서 직접 구현**한다 — 개인 로그인이 주는 자동 보안장치 일부를 쓰지 않는 트레이드오프를 수용.
- 동기화 충돌은 드물며 LWW로 충분하다.
- 일정은 취미 프로젝트로 데드라인 없음(여유롭게).
- 디자인 레퍼런스(§8)는 제안이며, 사용자 선호로 교체될 수 있다.
