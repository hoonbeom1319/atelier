# spotlog — PRD (Product Requirements Document)

> 작성: 시니어 PM · 2026-06-26 · 상태: /design 인계용 초안
> 모드: /forge(무인). 시작 1회 인테이크 + 시장조사 근거(장소축 사진 SNS 경쟁 분석, EXIF/역지오코딩 관행, 실사용자 니즈·프라이버시 조사)로 작성.
> 이 문서는 **개발에 넘길 디자인 산출물의 입력**이다. "최종 앱 코드"가 아니라 화면·동작·데이터 모델의 합의본을 정의한다.
> 한 줄 정의: **인스타가 '시간 축'으로 올리는 곳이라면, spotlog는 '공간(장소) 축'으로 올리는 곳.** 사진을 여러 장 던지면 EXIF로 장소별 자동 그룹핑되어 '장소 카드 피드'가 된다.

---

## 1. 배경과 문제 정의

**배경.** 사람들은 하루에 여러 장소를 다니며 사진을 찍는다. 그러나 기존 사진 SNS(인스타그램)는 콘텐츠를 **시간 순서(피드·스토리)**로 정렬한다 — "어디에 다녀왔는가"가 아니라 "언제 올렸는가"가 축이다. 그 결과 "이번 주말에 내가 다녀온 장소들"을 한눈에 묶어 보거나 남에게 공유하기 어렵고, 장소 정보는 캡션 텍스트로 흩어진다. spotlog는 그 축을 **장소(공간)**로 바꾼다: 콘텐츠의 1차 단위가 '글/사진 한 장'이 아니라 **'다녀온 장소 1곳 + 그곳에서 찍은 사진 묶음'(장소 카드)**이다.

**문제.** 장소 단위로 기록하려면 사용자가 장소를 하나하나 골라 사진을 분류해야 하는데, 이 **수동 입력 마찰**이 결정적 이탈 요인이다. 시장조사상 위치 기반 소셜 앱의 최대 실패 패턴이 바로 수동 입력 피로다 — Foursquare/Swarm의 반복적 수동 체크인 피로와 앱 분리가 이탈을 불렀고(2024년 City Guide 종료), 여행 기록 앱 이탈의 가장 큰 단일 원인도 "매 엔트리를 손으로 정리하는 부담과 완벽주의 압박"으로 보고된다. 즉 **"장소별 정리를 사용자에게 시키면 죽는다."**

**해법과 핵심 가치 제안.** spotlog는 인스타처럼 **사진을 여러 장 한 번에 선택해 올리면**, 각 사진에 박힌 **EXIF(위도·경도·촬영시간)로 장소를 자동 그룹핑**하고, 역지오코딩으로 **근처 장소명을 자동 추출**해 장소 카드를 만들어준다. 사용자는 '정리'가 아니라 '확인·보정'만 한다. 이것이 본 제품의 존재 이유이자 사활이다.

**왜 지금/왜 이 빈칸인가 (근거 있는 결정).** 시장조사 결론: 장소축 사진 SNS는 빈 시장이 아니지만, 기능 조각이 흩어져 있을 뿐 **조합이 비어 있다.** Mapstr는 '지도 위 핀 큐레이션'이지 사진 피드가 아니고, Polarsteps는 GPS 경로 추적의 '시간축' 여행일기이며, Google/Apple Photos는 EXIF로 장소 자동 그룹핑하지만 *개인 보관함*이지 SNS가 아니고, Foursquare/Swarm은 체크인 소셜, Instagram Map은 친구 위치 지도다. **"인스타식 다중 사진 업로드 → EXIF 자동 장소 그룹핑 → 세로 장소 카드 피드(SNS)"라는 조합은 어디에도 통째로 없다.** 개별 기능(EXIF 그룹핑·역지오코딩·핀 편집)은 모두 선행 사례가 있어 기술 난이도가 아니라 **UX 조합과 마찰 제거**가 승부처다.

**처음부터 안고 가는 두 가지 구조적 현실 (설계 전제).**
1. **EXIF GPS는 자주 없거나 깨진다.** 스크린샷·메신저 전송본·SNS 다운로드본은 GPS가 strip돼 있고(인스타·페북·X는 업로드 시 EXIF 제거, 압축 전송은 약 89%가 GPS 제거), 실내 GPS는 약 11.5m(CEP50)·도심 캐니언은 20m+ 튄다. 따라서 **'미분류 사진 수동 편집'은 드문 엣지케이스가 아니라 상시 발생하는 핵심 경로**다 — 1급(first-class)으로 설계한다.
2. **위치 공유는 즉시 프라이버시 역풍을 부른다.** Instagram Map(2025)은 출시 직후 '집 주소 노출' backlash를 맞았다. spotlog는 위치가 곧 콘텐츠 축이라 이 리스크가 더 직접적이다 — **기본 비공개·과거 시점 공유·집/실시간 위치 비노출**을 처음부터 깐다.

## 2. 목표 및 성공 지표 (측정 가능 — 정량·기간·기준)

**제품 목표.** 사용자가 사진 여러 장을 던지면 **거의 손대지 않고도** 장소별로 묶인 카드 피드가 만들어지고, 자동이 틀린 부분은 **최소 탭으로** 보정해 공유할 수 있다.

**북극성 지표 (근거 있는 결정).** 이 제품의 가치는 "자동 그룹핑이 얼마나 안 틀리나 + 틀렸을 때 얼마나 쉽게 고치나"에 직결된다. 따라서 절대 DAU보다 **첫 업로드의 자동화 성공 경험**이 북극성에 가깝다(시장조사 일관 권고).

> ⚠ 아래 표의 **제품 운영 지표(N1·N2·R1·E1·E2)**는 실서비스 출시 후 측정하는 *목표*다. 본 산출물(디자인) 단계에서는 데이터가 없으므로, **G1~G5(검증 게이트 지표)**가 디자인 완성 시점의 실측 합격선이다. 운영 지표 수치는 산업 일반 벤치마크 기반 권고이며 베타 데이터로 보정한다(§11 가정).

| # | 성공 지표 | 측정 기준 (정량) | 측정 시점/기간 |
|---|---|---|---|
| **N1** | 첫 세션 활성화 | 첫 업로드에서 **장소 카드 1개 이상 자동 생성 성공률 60%+** (자동 그룹핑이 핵심 가치이므로 최우선). **집계 모집단 = 업로드 세션에서 자동 생성된 카드 기준**(이후 게시 여부와 무관 — '업로드하면 카드가 자동으로 만들어졌나'. R1·E2의 postedAt 키와 달리 N1은 생성 사건 기준, §5) | 출시 후 코호트 |
| **N2** | 자동화 품질(핵심) | **사용자 수동 수정 없이 게시된 장소 카드 비율** = 자동 그룹핑 정확도의 직접 지표 (1차 목표 70%+). **분모 = 직접 촬영(EXIF GPS 보존) 사진을 1장 이상 포함한 게시 카드**(스크린샷·전송본만으로 만들어진 미분류-출발 카드는 자동화 성공 모집단에서 제외 — GPS strip은 자동화가 아니라 입력의 한계) | 출시 후 주간 |
| **N3** | 소셜 활성화(F5 최소 신호) | 첫 주 신규 사용자 중 **탐색에서 타인 장소 카드 1개 이상 열람 또는 1명 이상 팔로우한 비율 ≥ 25%** — F5(탐색·팔로우) 핵심 루프 *진입*의 최소 성공 신호. 인게이지먼트 최대화(좋아요·체류시간)가 아니라 "탐색·팔로우 루프에 한 번이라도 들어왔나"만 본다(§2 비목표와 구분). *측정 출처: 팔로우 측은 §5 Follow(도메인 모델)로 결정적 산정, '열람' 측은 분석 이벤트 로그 의존(도메인 모델 밖).* 방향성 권고치 — 베타로 보정(§11) | 출시 후 코호트 |
| **R1** | 리텐션 | Day-1 **35~45%**, Day-7 **15~20%** (소셜앱 중앙값 근처; 상위 벤치마크는 PMF 이후). **코호트 키 = `postedAt`(첫 게시 시점, §5)** | 출시 후 코호트 |
| **E1** | 보정 마찰 | **미분류 사진 1건당 장소 지정이 평균 ≤ 3탭**(후보 팝오버 1탭 또는 핀 1회)으로 완료 | 출시 후 / 사용성 측정 |
| **E2** | 기록 축적 | 활성 사용자의 **주당 추가 장소 카드 수 중앙값 ≥ 3장/주**(개인 기록 효용의 대리지표 — 혼자 써도 가치). **측정 키 = `postedAt`(게시 시점), 방문시간(`timeStart/End`)이 아님** — 과거 사진을 올리면 방문≠게시이므로 '주당 추가'는 게시 시점으로 집계(§5). **기준선 = 베타 4주 측정값**, 목표 = 기준선 대비 정체/하락 없이 ≥3장/주 유지. 방향성 권고치 — 베타로 보정(§11) | 출시 후 주간 |
| **G1** | 핵심 흐름 무결성 | 업로드→자동 그룹핑→보정→게시, 탐색→팔로우, 피드 열람 전 경로가 **죽은 컨트롤 0개**로 동작 | 검증 3·6단계 Playwright 1·2층 |
| **G2** | 엣지케이스 처리 | 미분류 트레이·장소명 후보 팝오버·핀 직접 지정·사진 카드 간 이동이 **명세대로 100% 동작** | 검증 3단계 spec |
| **G3** | 접근성 대비 | 다크 모드 기준 텍스트/주요 컨트롤 대비 **WCAG AA(본문 4.5:1, 큰 글자/UI 3:1) 통과** | a11y 자동 게이트 |
| **G4** | 뷰포트 무파손 | **390px ~ 1280px** 모바일 폭 컬럼 셸에서 레이아웃 깨짐·가로 스크롤·요소 겹침 0 | 검증 6단계 |
| **G5** | 상태 영속 | 업로드 중 보정 진행·피드 상태가 재진입 후 유지(목업 시드 기준 일관 동작) | 검증 6단계 |

**비목표(이번에 추구하지 않음).** 절대 DAU 총량 목표, 성능 벤치마크 수치 목표, 실시간 위치 공유 인게이지먼트(의도적 제외 — §10). **풀 소셜 인게이지먼트 최대화 지표**(댓글·좋아요·DM 볼륨·세션 길이 등) — 단, F5 핵심 루프 *진입*의 최소 신호(N3)는 추적한다(인게이지먼트 *최대화*를 추구하지 않는 것이지, 탐색·팔로우 루프가 도는지조차 안 보는 것은 아니다 — 이로써 §4 F5(P0)와 §2 지표의 추적성이 닫힌다).

## 3. 타깃 사용자·시나리오

**주 사용자 (페르소나 2종).**

- **P1 — "기록형 개인"(1차·필수).** 나들이·여행·맛집 탐방을 자주 다니고 사진을 많이 찍는 사람. 인스타에 일일이 올리긴 번거롭고, "이번 여행에서 다녀온 곳들"을 장소별로 깔끔히 모아 두고 싶다. **혼자 써도 가치가 있어야 한다**(Mapstr·Google Photos가 혼자 써도 쓸모 있는 이유) — 초기 리텐션은 소셜보다 개인 기록 효용에서 먼저 선다.
- **P2 — "탐색형 팔로워"(2차).** 남이 다녀온 장소를 구경하고, 마음에 드는 사람을 팔로우해 "이 사람이 어디를 다니나"를 본다. 가벼운 소셜 루프의 수요측.

**기술 친숙도.** 인스타 사용 경험이 있는 일반 모바일 사용자. 별도 온보딩 없이 "사진 올리기"가 인스타와 같은 직관으로 시작돼야 한다.

**사용 맥락.** 모바일이 주(이동 중·집에서 정리). 데스크톱(최대 1280px)에서도 같은 모바일 폭 컬럼 셸로 본다(반응형 분기 아님). 다크 모드 기반.

**대표 시나리오.**
- **S1 다녀온 하루 정리(핵심).** 주말 나들이 후, 갤러리에서 사진 20장을 한 번에 선택해 올린다 → spotlog가 EXIF로 "카페 A(오전)·전망대 B(오후)·식당 C(저녁)" 3개 장소 카드로 자동 묶어 보여준다 → 카페 A의 자동 추출명이 옆 가게로 잘못 떠서 탭 → 후보 팝오버에서 올바른 상호 선택 → 게시.
- **S2 미분류 사진 구조(상시 경로).** 올린 30장 중 8장이 스크린샷·메신저로 받은 사진이라 GPS가 없다 → '미분류' 섹션에 따로 모임 → 사용자가 그 8장을 기존 카드로 이동하거나, 지도를 열어 핀으로 위치를 직접 찍어 새 장소 카드를 만든다.
- **S3 검색에 없는 장소.** 노점·사유지라 후보가 0건 → 장소명 없이 좌표만 둔 채 사용자가 직접 라벨("할머니 분식")을 입력하거나, 그대로(좌표만) 둔다.
- **S4 내 기록 훑기(혼자 가치).** 메인 피드에서 내가 다녀온 장소 카드들을 세로로 스크롤하며 회상한다.
- **S5 남 탐색·팔로우.** 탐색에서 타인의 장소 카드를 보고 마음에 들면 팔로우 → 그 사람의 새 장소 카드가 내 탐색/피드에 들어온다.
- **S6 프라이버시 점검.** 게시 전 "이 장소가 공개됩니다" 고지를 보고, 집 근처 등 노출하기 싫은 카드는 비공개로 두거나 위치 정밀도를 낮춘다.

## 4. 기능 요구사항 (우선순위 + 근거)

> 규율: "다 중요" 금지 — MVP 경계를 직접 긋는다. 형식은 "사용자가 X → 시스템이 Y" + 엣지케이스.
> 우선순위: **MVP(P0)** = 이번에 만든다 / **다음(P1)** = 다음 버전 후보 / **나중·범위 밖** = §10.
> 각 항목은 시장조사로 근거 확보됨 → "가정"이 아니라 **근거 있는 결정**(미검증분만 §11).

### 4-1. MVP (P0) — 이번에 만든다

**F1. 다중 사진 업로드 + EXIF 자동 장소 그룹핑** *(근거: 사용자 MVP ① + "수동 입력 마찰이 location SNS를 죽인다"는 핵심 교훈 → 자동화가 본 앱의 사활)*
- 사용자가 인스타처럼 여러 장(수십~수백 장 상정 — 평균 카메라롤 ~2,795장)을 한 번에 선택한다 → 시스템이 각 사진의 EXIF에서 **위도·경도·촬영시간**을 읽어 **시공간 결합 클러스터링**으로 장소별 묶음(장소 카드)을 만들고, 카드를 **시간 순으로 배열**한다.
- 클러스터링 로직(근거 있는 결정): 시간 단독·공간 단독이 아니라 결합 — ① 촬영시간 갭으로 세션 분리(권고 기본 90~120분 비연속), ② 각 세션 내 좌표를 반경(권고 기본 100~200m)으로 묶어 카드 1개 생성. **정확한 임계값은 실데이터 튜닝 대상**이라 handoff 계약에 '튜너블 파라미터'로 노출한다(§11).
- 엣지케이스: ① EXIF DMS 좌표는 Ref(N/S/E/W) 부호 적용 필수 — 미적용 시 반대 반구. ② 좌표 (0,0)·명백한 이상치는 null 처리해 미분류로. ③ 타임존 모호성(EXIF 2.31 이전 DateTimeOriginal에 TZ 없음; Apple은 OffsetTimeOriginal 별도) — 해외 사진 정렬 어긋남 방지 위해 오프셋/GPS UTC 우선(§11). ④ GPS만 있고 시간 없음(또는 반대) → 가능한 축으로만 묶고 부족분은 미분류 후보로.
- **본 산출물 단계에서는 실제 EXIF 디코딩·클러스터링이 아니라 시드 데이터로 결과 화면·상태변화만 목업**(외부 의존성 없는 독립 HTML 규칙). 실연동은 handoff(§10·§11).

**F2. 역지오코딩 — 근처 장소명 자동 추출 + 후보 팝오버 선택 + 지도 핀 직접 지정** *(근거: 사용자 MVP ⑤; '자동 추출명+오류 시 검색 후보+핀 보정'을 매끄럽게 묶은 소비자 앱은 드물다 → 차별 디테일)*
- (a) **자동 추출.** 시스템이 카드의 좌표를 역지오코딩해 **근처 장소명을 '제안(suggestion)'으로** 카드에 표시한다(항상 '추정'으로 보여주고 1탭 수정 경로 노출 — 정확도 불신 완충).
- (b) **후보 선택.** 사용자가 장소명을 탭해 수정하면 → 시스템이 **좌표 주변 반경의 POI 후보를 거리순 팝오버**로 띄운다(각 후보: 상호명 + 도로명주소 + 거리 예 "32m"). 사용자가 하나를 고르면 카드 장소명이 그 후보로 확정된다.
- (c) **핀 직접 지정.** 후보가 0건이거나 다 틀리면 → 사용자가 **지도를 열어 핀을 옮겨 위치를 직접 찍는다.** 또는 라벨을 직접 입력한다.
- 엣지케이스: ① 후보 0건(노점·사유지·신규 상점) → 장소명 없이 좌표만 보존하고 카드에 좌표/사용자 라벨 표시(사용자 답 '없는 대로 두기'). ② 복합건물·다층(같은 좌표 여러 상호) → 좌표만으론 구분 불가, 후보 목록에서 사용자 선택. ③ 실내·도심 좌표 튐으로 엉뚱한 인접 상호 추출 → (b)/(c)로 보정.
- **장소명(가변)과 좌표(불변)는 분리 저장**(§5) — 역지오코딩이 틀리거나 사용자가 고쳐도 원좌표 보존. 본 단계 목업, 실연동 handoff.

**F3. 미분류 사진 수동 편집(업로드 중 장소 지정·이동)** *(근거: 사용자 MVP ②; EXIF 결손이 상시이므로 fallback이 아니라 핵심 상시 경로 — 시장조사 3개 축이 모두 1급 대우를 강하게 뒷받침)*
- (a) **미분류 트레이.** 좌표 없음/이상치 사진은 자동 그룹핑 결과 화면에서 **눈에 띄는 별도 '미분류' 섹션**으로 모인다(에러 처리가 아니라 업로드 주경로의 일부).
- (b) **카드로 이동.** 사용자가 미분류 사진(또는 경계 사진)을 **기존 장소 카드로 이동**(드래그 또는 '이동' 액션)하거나 새 카드를 만든다.
- (c) **장소 지정.** 미분류 사진에 F2의 (b)/(c)(후보 검색·핀 지정·라벨 입력)로 장소를 부여한다.
- 엣지케이스: ① 자동 그룹핑 경계에서 잘못 묶인 사진을 **다른 카드로 재배치** 가능해야 함(Photo→PlaceCard는 가변 FK, §5). ② 미분류가 0건이면 트레이 섹션 숨김. ③ 한 카드의 사진을 전부 빼면 빈 카드는 자동 제거 또는 사용자 확인.
- 핵심 지표: 미분류 1건당 장소 지정 ≤ 3탭(E1).

**F4. 내 기록 피드 (메인 — 장소 카드 피드)** *(근거: 사용자 MVP ③ + 인테이크 1번 "메인 = 장소 카드 피드"; 혼자 써도 가치 = 초기 리텐션의 핵심)*
- 사용자가 앱에 들어오면 → 시스템이 **세로 스크롤 피드**를 보여준다. **카드 1개 = 다녀온 장소 1곳 + 그 장소의 사진 묶음 + 시간 범위 + (옵션)메모·태그.** 인스타 피드와 가장 친숙한 형태.
- 카드 구성: 장소명(+주소/거리), 사진 묶음(대표 + 추가 N장), 방문 시간 범위, 메모·태그(있으면), 공개 범위 표시(§F7).
- 엣지케이스: ① 사진이 0장인 카드는 존재하지 않음(카드는 사진 묶음에서 생성). ② 기록이 0개면 막다른 길 금지 — 빈 상태에서 "사진을 올려 첫 장소를 기록하세요" 업로드 유도. ③ 한 장소 카드의 사진이 많으면 캐러셀/그리드로 무파손 표시(G4).

**F5. 타인 탐색 + 팔로우** *(근거: 사용자 MVP ④; 가벼운 소셜 루프 — 단, 한 셸 안에서 기록+탐색을 끝낸다(앱/탭 분리로 핵심 흐름 쪼개지 말 것 = Foursquare 실패 교훈))*
- (a) **탐색.** 사용자가 탐색에 들어가면 → 시스템이 타인의 공개 장소 카드를 보여준다(피드형/추후 지도형은 P1).
- (b) **팔로우.** 사용자가 어떤 사용자를 팔로우하면 → 그 사람의 새 공개 장소 카드가 내 탐색/피드 흐름에 반영된다.
- 엣지케이스: ① 비공개 카드는 탐색에 노출되지 않음(§F7·기본 비공개). ② 자기 자신 팔로우 불가. ③ 팔로우 0명/탐색 결과 0건일 때 빈 상태 안내.
- 범위: 댓글·좋아요·DM·푸시는 이번 제외(§10) — 핵심 루프부터 검증.

**F6. 장소 카드 메모·태그(옵션)** *(근거: 사용자 인테이크 "태그·간단한 메모는 인스타처럼 옵션")*
- 사용자가 **장소 카드**에 **짧은 메모와 태그를 선택적으로** 단다 → 카드에 표시된다. **입력은 강제하지 않는다**(빠른 게시가 기본).
- **단위는 장소 카드 레벨**이다(개별 사진 1장 단위의 메모·태그는 이번 범위 밖 — §10·§5와 정합). spotlog의 1차 콘텐츠 단위가 '장소 카드'(인스타의 '한 게시물'에 대응)이므로 메모·태그도 카드에 단다. 사진 1장은 카드 안의 묶음 원소일 뿐이라 별도 캡션을 갖지 않는다.
- 엣지케이스: 빈 메모·태그는 저장하지 않음(카드에 빈 영역 안 만듦).

**F7. 공개 범위 + 위치 프라이버시(최소 안전망)** *(근거: 사용자 미답이었으나 시장조사 3개 축이 강하게 경고 → "기본 비공개"는 근거 있는 결정으로 승격; 단 구체 정밀도 UX는 §11에서 사용자 검수)*
- (a) **공개 전 고지.** 게시 전 시스템이 "이 장소가 공개됩니다" 류 고지를 명확히 보여준다.
- (b) **공개 범위.** 카드별 공개/비공개. **기본값은 보수적으로**(전체공개 디폴트 지양 — Instagram Map backlash 교훈).
- (c) **위치 정밀도 낮추기/숨김(최소).** 집·민감 장소는 카드를 비공개로 두거나 위치를 동네 단위로 흐리는 최소 장치(MVP는 비공개 토글 + 고지를 필수, 정밀도 단계 슬라이더는 §11에서 범위 확정).
- 엣지케이스: ① 실시간/현재 위치는 절대 자동 노출하지 않음 — spotlog는 과거 방문 기록 공유만(§10). ② 비공개 카드는 본인만, 팔로워·탐색 모두 노출 금지.

### 4-2. 다음 (P1) — 다음 버전 후보 (이번 MVP 제외)

근거: 핵심 루프(업로드→자동그룹핑→보정→게시→탐색·팔로우) 검증이 먼저. 아래는 가치는 인정하나 핵심 흐름 없이도 앱이 성립.
- **지도 뷰 탐색** — 내/타인 장소 카드를 지도 위에서 보기(Mapstr식). MVP는 카드 피드로 충분, 지도형은 다음.
- **댓글·좋아요·DM** — 풀 소셜 인터랙션(사용자 가정대로 이번 제외).
- **푸시 알림** — 팔로우/새 카드 알림.
- **여행/날짜 묶음(Trip 계층)** — 장소 카드들을 여행/날짜로 묶는 상위 그룹(Polarsteps trip 계층). MVP는 '날짜 묶음'으로 가볍게 시작 가능하나 필수 아님(§5).
- **데이터 내보내기** — 내 사진/장소 전체 export(Polarsteps 표준; 후행 수익화 반발 완충 장치 — §11 메모).
- **위치 정밀도 다단계 슬라이더** — 동네/구/시 단위 흐리기 세분화.

### 4-3. 나중·범위 밖

§10 참조. 실시간 위치 공유, 백그라운드 GPS 경로 자동 추적(Polarsteps형), 게이미피케이션(배지·mayor), 수익화/구독 — 모두 이번 범위 밖(근거는 §10).

## 5. 데이터 모델 (개념 모델 — 물리 스키마는 범위 밖)

> **물리 스키마(테이블/컬럼 타입/인덱스/저장 포맷·외부 API 응답 캐시 구조의 최종 확정)는 범위 밖**이다 — 개발 단계 결정. 여기서는 개념 엔터티와 관계만 정의한다. 데이터 모델 골격은 업계 사실상 표준인 Polarsteps 3계층(trip→step(location)→media)을 따르며, 본 앱의 **PlaceCard ≈ step(장소 1곳의 기록 묶음)**으로 1:1 대응된다(근거 있는 결정).

### 엔터티

**User (사용자)** — 팔로우 기능 전제상 계정 존재(인테이크 가정 → §11에서 사용자 검수, 단 팔로우가 MVP라 사실상 확정).

| 필드 | 개념 타입 | 의미 | 비고 |
|---|---|---|---|
| `id` | 고유 식별자 | 사용자 키 | 물리 표현 범위 밖 |
| `handle` | 텍스트 | 사용자명/핸들 | 표시·탐색용 |
| `displayName` | 텍스트 | 표시 이름 | |
| `avatar` | 이미지 참조 | 프로필 사진 | 옵션 |

**Photo (사진)** — 자기 좌표·시간을 자기 필드로 보유(미분류→재배치를 쉽게).

| 필드 | 개념 타입 | 의미 | 비고 |
|---|---|---|---|
| `id` | 고유 식별자 | 사진 키 | |
| `imageRef` | 이미지 참조 | 원본 + 썸네일 | 물리 저장 범위 밖 |
| `lat` / `lng` | 십진수(WGS84) | EXIF 좌표 정규화값 | DMS→십진·Ref 부호 적용은 업로드 파이프라인 1회. null 가능(미분류) |
| `capturedAtLocal` | 일시 | DateTimeOriginal(로컬) | null 가능 |
| `tzOffset` | 시간 오프셋 | OffsetTimeOriginal | 없으면 null |
| `capturedAtUtc` | 일시(UTC) | GPS*Stamp 도출 또는 오프셋 환산 | 정렬 기준(§4 F1 TZ 처리) |
| `exifStatus` | enum `geotagged`/`unclassified` | EXIF 좌표 유무 | 미분류 트레이 분기 |
| `source` | enum `original`/`screenshot`/`received` | 출처(가능 시) | 미분류 원인 진단·표시 |
| `placeCardId` | FK → PlaceCard | 소속 장소 카드 | **가변** — 사용자가 다른 카드로 이동 가능. null = 미분류 |

**PlaceCard (장소 카드 — 피드의 단위, ≈ Polarsteps step)**

| 필드 | 개념 타입 | 의미 | 비고 |
|---|---|---|---|
| `id` | 고유 식별자 | 카드 키 | |
| `ownerId` | FK → User | 작성자 | |

| `placeName` | 텍스트 | 확정 장소명 | **가변** — 역지오코딩 결과 캐시/후보 선택/사용자 라벨. null 가능(좌표만) |
| `placeSource` | enum `reverse-geocode`/`poi-search`/`manual-pin`/`manual-label` | 장소명 출처 | 신뢰도 표시·정렬 |

| `address` | 텍스트 | 도로명/지번 주소 | 역지오코딩 캐시, 옵션 |
| `region` | 구조 | region 1/2/3depth(시·도/시군구/동) | 옵션 |

| `createdAt` | 일시 | 카드 인스턴스 최초 생성(자동 그룹핑) 시점 | 제안 카드도 **생성 즉시 PlaceCard 인스턴스**(confirmed=false). 라이프사이클 시작점 |
| `postedAt` | 일시 | **게시**(confirmed=true 전환) 시점 | **R1 코호트·E2 '주당 추가 카드'의 측정 키.** 방문시간(timeStart/End)과 **구분** — 과거 사진을 올리면 방문≠게시이므로 이 두 지표는 이 필드로 집계(§2). N1은 createdAt(생성) 기준. 미게시(제안) 카드는 null |
| `memo` | 텍스트 | 메모(옵션) | F6 |
| `confirmed` | 불리언 | 사용자 확정 플래그 | 자동 제안을 '미확정'으로 두는 패턴(Polarsteps Suggestion). true 전환 시 `postedAt` 채워짐 |
| `visibility` | enum `private`/`public` | 공개 범위 | **기본 private**(F7) |

**Tag (태그)** — 옵션, N:M.

| 필드 | 개념 타입 | 의미 |
|---|---|---|
| `id` | 고유 식별자 | 태그 키 |
| `label` | 텍스트 | 태그명 |

### 관계
- **User 1 : N PlaceCard** — 한 사용자가 0개 이상의 장소 카드를 가진다.
- **PlaceCard 1 : N Photo** — 한 카드는 1장 이상의 사진을 묶는다. `Photo.placeCardId`는 **가변 FK**라 사용자가 사진을 다른 카드로 재배치할 수 있다(F3). `placeCardId = null`인 Photo = 미분류(업로드 세션 동안의 임시 상태).
- **PlaceCard N : M Tag** — 카드 하나에 여러 태그, 태그 하나가 여러 카드에. 메모/태그는 옵션(F6).
- **User N : M User (Follow)** — 팔로우는 사용자 간 다대다(방향성 있는 follower→followee). 자기 자신 제외(F5).
- **PlaceCard 1 : 1 좌표(lat/lng), 1 : 1(가변) placeName** — 좌표는 불변, 장소명은 그 좌표에 대한 가변 라벨(별도 필드). 둘을 **분리 저장**하는 것이 핵심 설계 포인트(역지오코딩 오류·재검색·핀 보정을 흡수; Google Photos 관례와 일관).

### 업로드 세션 & PlaceCard 라이프사이클 (라이프사이클 일원화)
- **제안 카드는 처음부터 PlaceCard 인스턴스다.** 자동 그룹핑이 만든 카드는 생성 즉시 `confirmed=false`·`createdAt` 기록·`postedAt=null` 상태의 PlaceCard다. 사용자가 보정·확정하면 `confirmed=true`로 전환되며 `postedAt`이 채워진다. **PlaceCard 라이프사이클은 한 줄: 제안(confirmed=false) → 게시(confirmed=true).** (이로써 createdAt/confirmed=false가 *왜 PlaceCard 필드인지*가 명확하다 — 제안 카드가 곧 PlaceCard이기 때문.)
- **'영속 아님'인 것은 카드 자체가 아니라 ⓐ 업로드 세션 동안의 임시 귀속**(미분류 트레이의 사진↔카드 잠정 배치 — `Photo.placeCardId`가 확정 전까지 유동)과 **ⓑ 게시 없이 세션을 버릴 때의 처리**다. **미게시 초안 카드(confirmed=false)를 영구 저장소에 드래프트로 남길지 여부는 물리 표현 범위 밖**(개발 단계 결정) — 개념 모델은 "제안 카드 = PlaceCard(confirmed=false)"로 일관된다.
- **N1(자동 생성 성공률) 집계 모집단** = 업로드 세션에서 자동 생성된 카드(createdAt 기준, 이후 게시 여부 무관). R1·E2만 postedAt(게시) 기준 — 지표별 측정 키가 라이프사이클의 어느 지점인지 구분된다.

### Trip(여행/날짜 묶음) — MVP 미포함(개념만)
- Polarsteps의 trip 계층(User → Trip → PlaceCard)은 사실상 업계 관례지만 **MVP 필수 아님**(§4-2). 도입 시 '날짜 묶음'으로 가볍게 시작 가능. 지금은 카드가 시간순으로 평면 배열된다.

## 6. 사용자 흐름

> 모바일 폭 컬럼 셸(최대 1280px). 메인은 장소 카드 피드. 업로드는 인스타식 다중 선택에서 출발.

**메인 흐름 A — 업로드(핵심 루프, S1~S3).**
```
[+ 업로드] → 갤러리에서 사진 여러 장 선택
  → 시스템: EXIF로 시공간 클러스터링 → [자동 그룹핑 결과 화면]
       ├─ 장소 카드 N개(각 카드: 자동 추출 장소명='제안', 사진 묶음, 시간범위)
       └─ [미분류 섹션] (EXIF 없음/이상치 사진들)
  → (보정·상시) 카드 장소명 탭 ─┬─> [POI 후보 팝오버] 거리순 → 선택 → 장소명 확정
                               ├─> 후보 0건/틀림 → [지도 핀 직접 지정] 또는 [라벨 직접 입력]
                               └─> 그대로 두기(좌표만)
  → (보정·상시) 미분류 사진 ─┬─> 기존 카드로 이동(드래그/이동)
                            └─> 장소 지정(후보/핀/라벨) → 새 카드
  → (옵션) 메모·태그 입력
  → [공개 범위 확인 + "이 장소가 공개됩니다" 고지] (기본 비공개)
  → [게시] → 카드 확정(confirmed=true) → 메인 피드에 반영
```

**메인 흐름 B — 내 기록 열람(S4).**
```
[메인 진입] → 장소 카드 피드(세로 스크롤, 시간순)
  → 카드 탭 → 카드 상세(사진 묶음 전체·메모·태그·지도 위치)
  → (기록 0개) → 빈 상태: "사진을 올려 첫 장소를 기록하세요" → 업로드 유도
```

**메인 흐름 C — 탐색·팔로우(S5).**
```
[탐색] → 타인 공개 장소 카드
  → 사용자 프로필 → [팔로우] → 그 사람 새 공개 카드가 내 탐색/피드 흐름에 반영
  (비공개 카드는 탐색에 노출 안 됨)
```

**상태 전이 요약.**
- `Photo.exifStatus`: 업로드 시 `geotagged`/`unclassified` 결정 → 미분류는 사용자 보정으로 카드에 귀속(`placeCardId` 채워짐).
- `PlaceCard.placeName`: `제안(reverse-geocode)` → 사용자 보정 시 `poi-search`/`manual-pin`/`manual-label`로 갱신(좌표는 불변).
- `PlaceCard.confirmed`: `false(제안)` → `true(게시)`. 게시 전환 시 `postedAt` 기록(방문시간과 별개 — 운영지표 측정 키, §2·§5).
- `PlaceCard.visibility`: 기본 `private` ⇄ `public`(사용자 선택).

## 7. 화면 목록·요구사항

> 모바일 폭 컬럼 셸. 모든 인터랙티브 요소는 와이어드(죽은 컨트롤 0 — G1). 셀렉터 안정성 위해 role·aria·텍스트로 식별 가능하게.

**SC1. 메인 피드 (장소 카드 피드 — 홈).**
- 세로 스크롤 피드. 카드 1개 = [장소명/주소] [사진 묶음(대표+캐러셀/그리드)] [방문 시간 범위] [메모·태그(있으면)] [공개 표시].
- **좌표만 카드(placeName=null, F2 엣지케이스 ①)**: 장소명 자리에 사용자 라벨 또는 좌표/동네 단위 표시로 폴백 렌더(빈 장소명으로 깨지지 않음 — §4 F2와 정합).
- 하단/상단 글로벌 내비: 홈·탐색·**[+ 업로드]**·프로필. 한 셸 안에서 기록+탐색 완결(탭 분리로 핵심 흐름 쪼개지 않음).
- 빈 상태: 기록 0개 시 업로드 유도(막다른 길 금지).

**SC2. 업로드 — 사진 선택.**
- 인스타식 다중 선택 그리드. 선택 개수 표시, '다음' 액션. (산출물에선 시드 사진으로 목업.)

**SC3. 업로드 — 자동 그룹핑 결과(핵심 화면).**
- 자동 생성된 장소 카드 리스트(각 카드 장소명은 '추정'으로 명확히 표시 + 1탭 수정 진입점).
- **미분류 섹션을 눈에 띄게 별도 노출**(F3 — 1급 대우). 미분류 0건이면 숨김.
- 각 카드: 장소명 편집 진입, 사진 묶음, 사진 이동(드래그/액션), 시간범위.

**SC4. 장소명 후보 팝오버.**
- 좌표 주변 POI 후보 거리순 목록: [상호명] [도로명주소] [거리(예 "32m")]. 선택 시 확정. 검색 입력으로 재질의. 후보 0건 시 "지도에서 직접 지정" 폴백 노출.

**SC5. 지도 + 핀 직접 지정(목업).**
- 지도(목업 이미지/캔버스) 위 핀을 드래그/탭으로 이동해 좌표 확정. 라벨 직접 입력 가능. (실제 지도 SDK 미사용 — 목업, 실연동 handoff.)

**SC6. 메모·태그 입력 + 공개 범위/고지.**
- 메모(옵션)·태그(옵션) 입력. **"이 장소가 공개됩니다" 고지** + 공개/비공개 토글(기본 비공개). 게시 버튼.

**SC7. 카드 상세.**
- 한 장소 카드 전체: 사진 묶음 전체, 메모·태그, 지도 위치(목업), 방문 시간. 본인 카드는 편집/공개범위 변경 진입.

**SC8. 탐색.**
- 타인 공개 장소 카드 피드. 카드/프로필 진입. 결과 0건 빈 상태.

**SC9. 프로필.**
- 사용자(본인/타인)의 장소 카드 모음, 팔로우 버튼(타인), 팔로워/팔로잉 수. 본인은 비공개 카드도 보임.

**화면 요구(공통).**
- 다크 모드 기준. 390px~1280px 모바일 폭 컬럼 무파손(G4 — 사진 많은 카드도 캐러셀/그리드로 안 깨짐).
- 키보드/포커스 접근, 폼·토글·팝오버에 적절한 role/aria.
- 외부 지도·역지오코딩·EXIF는 전부 목업/시드(독립 HTML) — 흐름·상태변화는 완결.

## 8. 디자인 방향 (톤·레퍼런스·뷰포트·다크 모드)

- **톤.** 사진이 주인공인 콘텐츠 우선 UI. 사진을 돋보이게 하는 차분한 다크 배경, 절제된 크롬(chrome), 카드 중심 위계. 인스타와 친숙한 피드 감각 + 장소(지도/핀/주소) 메타의 명료한 표시.
- **레퍼런스.** 닮고 싶은 특정 서비스는 미지정(인테이크). 비유 기준점: **인스타의 피드 UX 친숙성 + Mapstr의 장소 큐레이션 + Google Photos식 자동 장소 그룹핑**의 조합. 화려한 일러스트/과한 그라데이션 지양, 사진과 지도 메타가 주연.
- **뷰포트.** **모바일 기준 디자인을 최대 1280px까지 그냥 풀어놓는 모바일 폭 컬럼 셸.** 데스크톱이라고 디자인이 달라지지 않는다 — **반응형 분기 아님.** 중앙 정렬된 모바일 폭 컬럼을 넓은 화면에서도 유지(과도하게 넓히지 않음). /design 인테이크(00-flow)에서 390px~1280px를 대상 뷰포트로 명시.
- **색 모드(다크 모드).** **다크 모드 기반**(인테이크). 다크 단일을 기본으로 설계하되 토큰은 표준 2단(primitive `--gray-50..900` 등 중립 램프 → semantic 역할명 `--color-surface`·`--color-text`·`--color-primary`). 브랜드 단어로 토큰명 짓지 않음. 다크 모드에서 WCAG AA 대비 충족(G3). (라이트 모드는 이번 필수 아님 — §11.)
- **언어.** 한국어 UI. 모든 라벨·고지·빈 상태·거리표시("32m")·시간범위 한국어.
- **사진 표시.** 한 카드의 다중 사진은 캐러셀/그리드로 무파손. 미분류·제안 상태는 시각적으로 명확히 구분(보정 유도).

## 9. 비기능 요구사항

- **접근성(a11y).** 다크 모드 텍스트/주요 컨트롤 대비 WCAG **AA**(본문 4.5:1, 큰 글자/UI 3:1). 키보드 전체 조작, 폼·팝오버·토글에 라벨/aria.
- **뷰포트.** 390px~1280px 모바일 폭 컬럼 무파손(가로 스크롤·겹침·잘림 0). 반응형 분기 없이 동일 셸.
- **프라이버시(구조적 요구).** 기본 비공개·공개 전 고지·실시간/현재 위치 비노출(F7·§10). 위치는 좌표(불변)와 장소명(가변) 분리 저장. 위치 정밀도 낮추기 최소 장치. (위치 기반 SNS의 구조적 신뢰 리스크 — 시장조사 3개 축 경고.)
- **자동화 견고성.** EXIF 결손/오차/타임존이 상수임을 전제 — 자동 그룹핑은 best-effort로 제시하고 보정 경로를 1급으로(F2·F3). 자동 결과는 '제안(미확정)'으로 저장.
- **성능(합리적 기대).** 한 번에 수십~수백 장 선택→그룹핑이 체감 지연 없이 처리(평균 카메라롤 규모 상정). 산출물 단계는 목업이라 수치 벤치마크 목표 없음.
- **독립 실행.** atelier 규칙상 산출물은 외부 의존성(외부 라이브러리·지도 SDK·EXIF 라이브러리·역지오코딩 API) 없는 독립 실행 HTML — 상태 변화용 인라인 JS 허용. 외부 연동은 전부 목업, 실연동은 handoff 계약.
- **데이터 소유권(메모).** 내 사진/장소는 후행 수익화 반발(Mapstr 사례)을 줄이기 위해 처음부터 '내보내기 가능' 가정을 명확히(내보내기 기능 자체는 P1 §4-2).

## 10. 범위 외 (Out of Scope)

이번 MVP가 **명시적으로 하지 않는 것** — 착수자가 추측하지 않도록 못박는다.

- **실시간 위치 공유 / 현재 위치 노출** — 의도적 제외. Instagram Map(2025) backlash처럼 리스크 대비 MVP 가치 낮음. spotlog는 **과거 방문 기록 공유만**.
- **백그라운드 GPS 경로 자동 추적(Polarsteps형)** — 배터리·프라이버시 비용 큼. 본 앱은 사진 EXIF 기반 사후 정리 모델이라 상시 추적 불필요.
- **풀 소셜 인터랙션** — 댓글·좋아요·DM·푸시 알림(P1 §4-2). 핵심 루프부터 검증.
- **지도 뷰 탐색** — 카드 피드로 MVP 성립, 지도형은 P1.
- **Trip/여행 묶음 상위 계층** — 카드 시간순 평면 배열로 시작, Trip은 P1.
- **게이미피케이션** — 배지·mayor 등(Foursquare가 잃고 후회했으나 MVP는 핵심 루프 우선 → 후속).
- **수익화·구독·저장 한도** — 이번 범위 밖(데이터 소유권/내보내기 가정만 미리 박아둠 §9).
- **실제 외부 연동(산출물 단계)** — 카카오 coord2address·keyword 검색 API 키 발급, EXIF 디코딩 라이브러리, 클러스터링 임계 실튜닝, 지도 SDK — 전부 **handoff 계약으로 이관**(산출물은 목업/시드). handoff에 넣을 것: API 응답→PlaceCard 필드 매핑표, 클러스터링 파라미터(시간갭·반경) 노출, 미분류/오차 폴백 UX 계약.
- **물리 데이터 스키마/저장 포맷 확정** — 개발 단계 결정(§5).
- **라이트 모드** — 이번은 다크 기반. 라이트는 P1(§11).

## 11. 미해결 이슈와 가정

> 시장조사로 근거 확보된 항목은 §1·§4·§5에서 **근거 있는 결정**으로 본문에 반영했다. 아래는 (a) 사용자가 답하지 않아 가정으로 닫은 것, (b) 여전히 추측이라 본문에서 단정하지 않은 것이다.

**A. 사용자 미답 → 가정으로 닫음 (확인되면 본문 승격).**
- **A1. (가정) 로그인/계정 존재.** 팔로우(F5)가 MVP라 계정 개념은 존재로 가정 — 사실상 확정에 가깝다(팔로우의 전제). 단, 계정 생성·인증 흐름(소셜 로그인 등) 자체는 산출물 핵심이 아니라 목업/약식 처리.
- **A2. (가정) 언어 한국어 단일.** UI·역지오코딩 결과·라벨 전부 한국어.
- **A3. (가정) 외부 연동은 목업, 실연동은 handoff.** 지도·역지오코딩(카카오맵 등)·EXIF 디코딩은 외부 의존성 없는 독립 HTML 규칙상 산출물 단계에서 목업/시드. 실연동은 handoff 계약(§10).
- **A4. (가정) 풀 소셜 기능 제외.** 댓글·좋아요·DM·푸시는 이번 MVP 밖(§4-2·§10).
- **A5. (가정→근거로 보강) 기본 비공개·프라이버시 최소 안전망.** 사용자는 프라이버시를 명시 답하지 않았으나, 시장조사 3개 축이 강하게 경고 → '기본 비공개·공개 고지·실시간 위치 비노출'(F7)을 근거 있는 결정으로 넣었다. **단 "위치 정밀도 낮추기"의 구체 UX 범위(비공개 토글만 vs 동네 단위 흐리기 슬라이더)는 사용자 검수 필요** — MVP는 비공개 토글+고지를 필수로, 정밀도 슬라이더는 P1로 잠정.
- **A6. (가정) 다크 모드 단일.** 인테이크 "다크 모드 기반"을 다크 단일 기본으로 해석. 라이트 모드 동시 제공 여부는 사용자 확인 필요(현재 P1).

**B. 본문에 단정하지 않은 추측 (검증/결정 필요).**
- **B1. (추측·guess) 클러스터링 임계값.** 세션 분리 시간갭(90~120분)·장소 묶음 반경(100~200m)은 시장조사상 confidence "guess"(체류지점 연구는 도심 250~1000m·체류 15분 기준이나 '사진첩 장소 카드' 용도엔 더 타이트해야 함). **정확값은 실데이터 튜닝 대상** → handoff에 튜너블 파라미터로 노출(§10). 본문은 '권고 기본값'으로만 기술.
- **B2. (추측·guess) 자동 분류율 목표.** "직접 촬영 원본 위주 시 자동 그룹핑 성공률 70~90%"는 정확한 산업 실측이 아니라 방향성 권고(N2). 업로드 사진엔 스크린샷·전송본이 섞여 GPS strip 비율이 상당하므로 100% 전제 금지. 베타 데이터로 보정.
- **B3. (추측·guess) 리텐션·활성화·축적·소셜 목표 수치.** N1(60%+)·N3(소셜 활성화 ≥25%)·R1(Day1 35~45%/Day7 15~20%)·E2(주당 ≥3장/주)는 소셜앱 일반 벤치마크 기반 방향성 권고 — 본 도메인 전용 실측 아님. **E2의 ≥3장/주와 기준선, N3의 25%는 베타 4주 측정으로 확정·보정**한다. PMF 이후 상향.
- **B4. (결정 보류) 미분류 사진 이동 인터랙션.** '드래그' vs '이동 액션(메뉴/롱프레스)' 중 무엇을 주 인터랙션으로 할지는 /design에서 확정(요구는 ≤3탭·무파손, E1). 모바일 드래그의 사용성 리스크 고려.
- **B5. (결정 보류) 타임존 처리 깊이.** 해외 사진 정렬을 위해 OffsetTimeOriginal/GPS UTC 우선(§4 F1)을 원칙으로 두되, 실제 디코딩은 handoff. 한국 단일 서비스라도 해외 여행 사진에서 카드 시간순이 어긋날 수 있으므로 명시.
- **B6. (구현 자유) 데이터 영속·저장 메커니즘.** §5의 엔터티는 개념 모델 — 저장 위치/포맷/외부 API 캐시 구조는 대상 프로젝트가 자기 컨벤션으로 결정. atelier는 구현에 묶이지 않는다.
- **B7. (결정 보류) 위치 정밀도 흐리기 알고리즘.** A5의 '동네 단위 흐리기'를 도입한다면 좌표 양자화/표시 단위(region depth) 방식은 미정 — P1 범위 확정 시 결정.
