# team-leave 디자인 인계 — 대상 프로젝트에 구현

/ forge(무인)로 기획→디자인을 끝까지 내린 산출물이다. 아래 디자인 패키지를 **대상 프로젝트의 컨벤션**으로 구현한다. 디자인은 *참고*, 흐름·기능·데이터 모델은 *계약(필수)*이다. atelier는 대상 코드를 직접 짜지 않는다 — 받는 쪽 Claude Code가 자기 프레임워크·아키텍처로 짠다.

## 무엇을 만드나
사내 **휴가 신청·승인 내부툴**(데스크톱 1280px+·라이트·한국어). 직원이 셀프로 신청·잔여 조회, 팀장이 한 큐에서 승인/반려, 모두가 팀 캘린더로 부재를 보고, HR이 멤버·부여 연차를 관리한다. 메신저·엑셀로 흩어진 휴가를 한 곳으로 모은다.

## 먼저 읽을 것 (순서대로)
1. **대상 프로젝트의 CLAUDE.md** — 너의 코드 컨벤션(폴더구조·아키텍처·프레임워크·상태관리·토큰 시스템). **여기 맞춰 짠다.** 아래 atelier 산출물은 이 컨벤션으로 *번역*하는 것이지 복붙이 아니다.
2. **제품 요구사항(왜·무엇의 근거)**: `projects/team-leave/PRD.md` — 배경·목표·성공지표·기능 요구사항(F1~F8, 동작·엣지케이스)·**데이터 모델(§5)**·범위 외(§10)·가정(§11). 흐름과 디자인이 *왜* 이렇게 됐는지 여기서 확인.
3. **흐름·기능 계약**: `projects/team-leave/00-flow.md` — 화면 목록·§플로우(A~D)·§화면별 구성요소·상태·§인터랙션 모델 = **지킬 계약**.
4. **디자인 패키지(atelier 산출물)**:
   - chosen final 화면(최종 비주얼·상태·전환): `projects/team-leave/screens/` (10개 HTML, 브라우저로 바로 열림)
   - 비주얼 문서(스크린샷+흐름맵+화면별 스펙): `projects/team-leave/handoff/index.html`
   - 토큰 매핑(semantic → 너의 토큰): `projects/team-leave/handoff/token-mapping.md`
   - 컴포넌트 인벤토리(variant/state + 쓰이는 화면): `projects/team-leave/handoff/component-inventory.md`
   - 디자인 토큰 원본: `projects/team-leave/foundation/tokens.css` (표준 2단 primitive→semantic)

## 지킬 것 (계약 — 임의 변경 금지)
- **화면 10개**(SC1~SC10: 로그인·홈(role-aware)·신청·승인 큐·캘린더·잔여·내 신청·알림·멤버 관리·설정)와 좌측 내비 + 상단바 셸.
- **흐름·상태 동작**: 신청(pending 가차감)→승인(used 확정)/반려(복원, 사유 필수)/취소(used 복원), 잔액 불변식 `잔여=부여−사용−대기`, 상태마다 이력 기록(감사 추적).
- **데이터 모델(PRD §5)**: User·Team·LeaveType·LeaveRequest·Approval·LeaveBalance·**BalanceAdjustment**(HR 시드 변경 감사)·Notification. 상태 머신·불변식 그대로.
- **RBAC 가시성**: 직원/팀장/HR이 보는 화면·데이터가 다름(승인 큐·멤버 관리는 팀장/HR).
- **a11y**: 상태·종류는 색만이 아니라 색+텍스트/아이콘, WCAG AA 대비, 키보드/포커스. (현재 산출물은 axe serious 0 통과.)
- **토큰은 매핑표대로** 대상 시스템 토큰에 매핑(통째 복붙 금지).

## 만들 순서 (제안)
1. 토큰 매핑(token-mapping.md) → 너의 토큰 시스템에 semantic 반영
2. 공통 컴포넌트(component-inventory.md: 버튼·태그·폼·테이블·카드·모달/드로어) + 셸(사이드 내비·상단바)
3. 화면 조립(SC1~SC10) — 비주얼은 screens/ 참고
4. 흐름·상태 연결(신청→승인/반려/취소 + 잔액 동기화 + 이력) — 00-flow §플로우·PRD §5 계약대로

## 범위 경계 (PRD §10 — 디자인은 그리되 백엔드는 개념)
- 실 인증/SSO·실시간 동기화·알림 실 발송·다단계 승인·누적/이월 자동계산·공휴일 자동제외·다크모드·모바일 1급·다국어 = **이번 범위 밖**. 로그인 등 **진입 화면 UI는 그린다**(백엔드만 개념).
- 잔액 모델은 4필드(total/used/pending/remaining) 표시 — 트랜잭션 원장은 대상 구현 선택(PRD §11 B4).

## 검수
- 흐름·기능이 `00-flow.md`·`PRD.md` 계약대로 동작하는지 확인.
- 산출물은 동작·접근성·트렌드 부합·인계 완결성까지 자동 통과했으나 **"옳은 제품인가·최종 미감"은 무인이라 사람 미검수**다(forge 무인 한계). 가정(PRD §11)을 먼저 검토 후 구현.
