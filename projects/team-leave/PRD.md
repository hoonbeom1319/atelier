# team-leave — PRD (Product Requirements Document)

> 작성: 시니어 PM · 2026-06-29 · 상태: /design 인계용 초안
> 모드: /forge(무인). 시작 1회 인테이크 + 시장조사 근거(휴가관리 SaaS 핵심 기능·승인 워크플로우 관례 / 연차·휴가 데이터 모델·잔여 계산·휴가 종류 표준 / 내부툴 UX·승인 큐·팀 캘린더 패턴·목표 지표 수준)로 작성.
> 이 문서는 **개발에 넘길 디자인 산출물의 입력**이다. "최종 앱 코드"가 아니라 화면·동작·데이터 모델의 합의본을 정의한다.
> 한 줄 정의: **메신저·이메일·엑셀로 흩어진 휴가 신청·승인·잔여 연차를 한 곳으로 모으는 사내 휴가관리 내부툴.** 직원은 셀프로 신청·잔여 조회, 팀장은 한 큐에서 승인/반려, 모두가 팀 캘린더로 부재 현황을 본다.

## 1. 배경과 문제 정의

**배경.** 많은 조직에서 휴가는 "카톡으로 팀장에게 보고 → 이메일 참조 → HR이 엑셀에 수기 기록"의 비공식 경로로 처리된다. 신청·승인·잔여 추적이 서로 다른 채널에 흩어져 있어, (1) 신청이 누락·지연되고, (2) 누가 언제 쉬는지 팀 단위로 한눈에 안 보이며, (3) 잔여 연차가 사람마다 엑셀 셀에 묻혀 있어 직원이 자기 잔여를 물어보려면 HR에 문의해야 한다.

**문제(핵심).** 휴가 신청이 **메신저·이메일·엑셀로 분산**되어 ① 승인 워크플로우가 표준화되지 않고(누가 처리했는지·언제 처리됐는지 추적 불가), ② 잔여 연차가 단일 출처(single source of truth)로 관리되지 않으며(엑셀 수기 → 오류·불일치), ③ 팀 부재 현황 가시성이 없어(같은 날 몇 명이 비는지 모름) 업무 공백·승인 충돌이 발생한다. 시장조사 결론: 이 카테고리의 가치 제안은 일관되게 **"승인 누락 방지 · 추적 일원화 · 관리 부담 감소"**다.

**해법과 핵심 가치 제안.** team-leave는 흩어진 세 가지 흐름(신청·승인·잔여)을 **하나의 내부 웹앱**으로 모은다. 직원은 폼으로 신청하고 자기 잔여를 셀프로 본다. 팀장은 **단일 승인 큐**에서 인라인으로 승인/반려한다(반려 시 사유 필수). 모두가 **팀 캘린더**에서 누가 언제 비는지 본다. HR은 멤버·잔여를 관리한다. 모든 처리는 **상태 이력(누가/언제/무엇을)**으로 남아 추적·감사가 된다.

**왜 이 골격인가 (근거 있는 결정).** 시장조사 3개 축이 사용자가 정한 MVP 5종(신청·승인/반려·잔여조회·팀캘린더·알림)이 **업계 표준 핵심 기능 세트와 정확히 일치**하며 빠진 핵심이 없음을 확인했다(Rippling·Vacation Tracker·cflow 등). 또한 내부툴 IA 표준(듀얼 대시보드 = 직원 self-service / 관리자 승인 큐 + 신청 + 팀 캘린더 + 잔여 + 설정/멤버)이 사용자 default-IN 골격과 일치한다. 즉 본 제품은 빈 시장을 개척하는 것이 아니라 **검증된 표준 골격을 흩어진 현실에 적용**하는 것이며, 승부처는 신기능이 아니라 **마찰 없는 신청·명료한 승인·신뢰할 수 있는 잔여 표시**다.

**처음부터 안고 가는 구조적 현실 (설계 전제).**
1. **잔여 차감 타이밍이 핵심 설계 포인트.** 신청 시점에는 아직 확정 차감이 아니다 — "신청했는데 잔액이 안 깎였다 / 이미 깎였다"는 혼란을 막으려면 잔액을 `부여(total)·사용(used)·대기(pending)·잔여(remaining)` 4개로 분리해, 신청 시 pending에 잡고 승인 시 used로 확정하는 모델이 관례다(시장조사 grounded).
2. **반려 시 사유 필수는 강한 업계 관례.** "Q3 출시와 겹침" 같은 actionable 사유 입력을 필수 필드로 둔다(best practice).
3. **엣지케이스가 표면을 만든다.** 본인 기간 겹침·잔액 초과·승인 후 취소(잔액 복원)·반차 0.5일·기간 유효성은 드문 예외가 아니라 휴가툴의 일상 경로다 — PRD가 빈약해지지 않으려면 본문에서 다뤄야 한다(시장조사 grounded).

## 2. 목표 및 성공 지표 (측정 가능 — 정량·기간·기준)

**제품 목표.** 직원이 휴가를 **1~2분 안에** 신청하고, 팀장이 **한 화면에서** 충돌을 보며 승인/반려하며, 모두가 잔여·팀 부재를 **HR 문의 없이 셀프로** 확인하게 한다. 휴가 데이터의 단일 출처를 메신저·엑셀에서 이 시스템으로 옮긴다.

**북극성 지표 (근거 있는 결정).** 내부툴이므로 성장지표(가입 전환·MAU)가 아니라 **업무 대체율·처리 시간·데이터 정확성**이 가치에 직결된다(시장조사 일관 권고). 북극성에 가장 가까운 것은 **"휴가 신청의 시스템 경유율"**(메신저/엑셀 이탈 제거) — 이 제품이 존재하는 이유의 직접 측정값.

> ⚠ 아래 표의 **운영 지표(O1~O4)**는 실서비스 도입 후 측정하는 *목표*다. 본 산출물(디자인) 단계에서는 데이터가 없으므로, **G1~G5(검증 게이트 지표)**가 디자인 완성 시점의 실측 합격선이다. 운영 지표 수치는 산업 일반 벤치마크·도메인 가치 제안 기반 권고이며, 도입 후 4주 베이스라인으로 보정한다(§11). 내부툴이라 "첫 분기 활성 70~80%"가 흔한 롤아웃 벤치마크지만 HRIS 평균 실사용은 32%(Gartner)에 불과해 야심적 상한이지 default 기대가 아니다 — 그래서 활성률보다 **행동지표**를 KPI로 잡는다.

| # | 성공 지표 | 측정 기준 (정량) | 측정 시점/기간 |
|---|---|---|---|
| **O1** | 신청의 시스템 경유율 (북극성) | 조직 내 발생 휴가 중 **시스템을 통해 신청·기록된 비율 ≥ 90%** (메신저/이메일/엑셀 우회 제거). 분모 = 해당 기간 전체 휴가 사용 건, 분자 = 시스템 내 신청 건 | 도입 후 월간 |
| **O2** | 승인 리드타임 | 신청 → 승인/반려 처리까지 **중앙값 ≤ 1 영업일**. 측정 키 = `requestedAt → decidedAt`(§5) | 도입 후 주간 |

## 3. 타깃 사용자·시나리오

**주 사용자 (역할 3종 — RBAC).**

- **R1 — 직원(신청자, 1차·필수).** 휴가를 신청하고, 자기 신청 상태와 잔여 연차를 셀프로 확인한다. 팀 부재 현황을 보고 일정을 잡는다. **혼자(직원 시점만) 써도 가치가 있어야 한다** — "내 잔여가 얼마고, 내가 낸 신청이 지금 어디까지 갔나"가 즉시 보이는 것이 초기 효용.
- **R2 — 팀장/관리자(승인자, 1차·필수).** 팀원의 신청을 **승인 큐**에서 처리한다(승인/반려, 반려 시 사유 필수). 승인 전에 같은 기간 팀 부재 충돌을 본다. 자기 팀의 휴가 현황을 캘린더로 본다.
- **R3 — HR(관리, 일상 사용은 2차이나 셋업은 선행조건).** 멤버·팀을 관리하고, 직원별 부여 연차(잔여 시드)를 설정한다. 전사 휴가 현황·신청 이력을 본다. (실 인증·HR 권한 백엔드는 개념 — §10.) **빈도상 일상 사용자는 직원·팀장(R1·R2)이지만, HR의 '부여 연차 시드' 설정은 F1(신청 시 잔액 검증)·F3(잔여 조회)이 성립하기 위한 선행조건(prerequisite)이다** — 즉 사용 빈도는 2차이나 데이터 흐름상 load-bearing이므로, 이를 받치는 기능 F8은 사실상 P0다(§4-1 F8). "2차 사용자"와 "P0 기능"은 충돌이 아니다(자주 쓰진 않지만 없으면 핵심 루프가 시드 데이터 없이 빈다).

**기술 친숙도.** 사내 업무 소프트웨어(그룹웨어·슬랙·노션)에 익숙한 일반 오피스 사용자. 별도 교육 없이 폼·테이블·캘린더의 표준 패턴으로 바로 쓸 수 있어야 한다.

**사용 맥락.** **데스크톱 우선 내부 업무 환경**(1280px+). 라이트 모드 기본. 한국어 UI. 모바일 1급 대응은 범위 밖(§10).

**대표 시나리오.**
- **S1 휴가 신청(직원·핵심).** 직원이 [신청]에서 종류(연차)·기간(7/10~7/11)·사유를 입력 → 시스템이 잔여(예: 12일)에서 신청 2일을 **pending으로 잡고** "신청 후 잔여 10일(대기 2일)"을 미리 보여줌 → 제출 → 상태 '대기'.
- **S2 반차 신청(부분일).** 직원이 종류 '반차', 날짜 7/15, 오전/오후 선택 → 시스템이 0.5일로 계산·차감 미리보기 → 제출.
- **S3 승인(팀장·핵심).** 팀장이 [승인 큐] 상단의 '내 액션 필요'에서 직원 신청 행을 봄 → 같은 기간 팀 캘린더 미리보기에서 동료 1명도 겹침을 확인 → [승인] → 직원 잔여가 pending→used로 확정, 상태 '승인'.
- **S4 반려(사유 필수).** 팀장이 [반려] 클릭 → 사유 입력 모달("Q3 출시 주간과 겹침") **필수** → 제출 → 상태 '반려', 직원의 pending 차감 복원, 사유가 신청 상세에 기록.
- **S5 잔여 조회(직원·셀프).** 직원이 [잔여]에서 '부여 15 / 사용 3 / 대기 2 / 잔여 10'을 종류별로 본다. HR에 문의하지 않는다.
- **S6 팀 캘린더(모두).** 월 그리드에서 누가 언제 비는지 색 바로 본다. 같은 날 부재 N명 카운트·승인 대기 건이 보인다.
- **S7 승인 후 취소(엣지).** 직원이 승인된 미래 휴가를 취소 요청 → 처리 시 used가 잔여로 복원, 상태 '취소', 이력에 잔존.
- **S8 멤버/잔여 관리(HR).** HR이 [멤버 관리]에서 신입의 부여 연차를 11일로 설정, 팀 배정.
- **S9 빈/첫 실행.** 신청 0건(직원 첫 사용)·승인 큐 0건(처리할 것 없음)·잔여 0일 상태에서 막다른 길 없이 안내.

## 4. 기능 요구사항 (우선순위 + 근거)

> 규율: "다 중요" 금지 — MVP 경계를 직접 긋는다. 형식은 "사용자가 X → 시스템이 Y" + 엣지케이스.
> 우선순위: **MVP(P0)** = 이번에 만든다 / **다음(P1)** = 다음 버전 후보 / **나중·범위 밖** = §10.
> 각 항목은 시장조사로 근거 확보됨 → "가정"이 아니라 **근거 있는 결정**(미검증분만 §11).

### 4-0. 제품 아키타입 + 표준 화면 골격

**아키타입 = 내부 SaaS 웹앱(admin/대시보드 계열).** 이 유형이 *당연히 갖는* table-stakes 화면을 §7에 **default-IN으로 깐다**: 로그인(진입 UI), 홈/대시보드(role-aware), 핵심 툴(신청 폼·승인 큐·팀 캘린더·잔여 조회), 설정/프로필, 멤버/팀 관리(관리자), 빈/첫 실행 상태. 빼는 화면은 §10에 **의도적 제외**로 명시한다(조용한 누락 금지). "백엔드(실 인증)가 범위 밖"인 것과 "그 화면 UI를 안 그린다"는 다르다 — **로그인·진입 UI는 그린다**(백엔드만 개념).

### 4-1. MVP (P0) — 이번에 만든다

**F1. 휴가 신청 (종류·기간·사유 + 잔액 미리보기)** *(근거: 사용자 MVP ①; '신청→매니저 라우팅→잔액 체크'가 워크플로 essentials — 흩어진 신청을 한 곳으로 모으는 문제의 직접 해결, P0 핵심 가치 루프)*
- 직원이 [신청]에서 **휴가 종류·기간(시작일/종료일)·사유**를 입력한다 → 시스템이 **영업일 기준 신청 일수를 계산**하고, 잔여에서 그 일수를 **pending(대기 차감)으로 잡아** "신청 후 예상 잔여"를 실시간으로 보여준다 → 제출 시 상태 '대기(pending)'로 승인 큐에 들어간다.
- 휴가 종류(근거 있는 결정 — 한국 맥락 표준): **연차(잔여 차감) · 반차(0.5일 차감, 오전/오후 선택) · 병가 · 경조사 · 공가 · 무급** 중 MVP는 최소 **연차·반차·병가·경조사**. 종류는 enum이 아니라 **설정 가능한 마스터 데이터(LeaveType)**로 두어 HR 관리 화면과 연결하고, "잔여에서 차감하는가" 플래그를 갖는다(연차 계열 차감 / 공가·경조사 등 무차감 구분).
- **신청 필드 코어 = 종류·기간·사유**(사용자 명시). 사유는 반차/연차에선 선택~권장, 병가·경조사는 증빙 메모 대체.
- 엣지케이스(근거 있는 결정 — 시장조사 grounded, MVP에서 실제 검증·경고로 구현):
  - ① **본인 기간 겹침** — 같은 직원의 기존 대기/승인 신청과 날짜 중복 시 **차단·경고**.
  - ② **잔액 초과** — 신청 일수 > 잔여(가용) 시 **차단**(음수 잔액 방지; 한국 관례). 무급 전환 안내는 P1.
  - ③ **반차 0.5일** — 시작=종료일이지만 0.5일 차감, 오전/오후 선택.
  - ④ **기간 유효성** — 종료일 < 시작일(역순)·0일·과거일자 소급 신청은 검증(과거일 신청 허용 여부는 §11 가정 → MVP는 당일 포함·미래 기본, 과거는 경고).
  - ⑤ **영업일 계산** — 주말 제외(한국 관례는 영업일 차감 일반적). 공휴일 자동 제외는 P1(공휴일 마스터 필요 — §11).
- **본 산출물 단계에서는 실제 잔액 트랜잭션·영업일 엔진이 아니라 시드 데이터로 계산 결과·상태변화만 목업**(외부 의존성 없는 독립 HTML). 실연동은 handoff(§10·§11).

**F2. 승인/반려 워크플로우 (승인 큐 + 반려 사유 필수)** *(근거: 사용자 MVP ②; ①신청+②승인이 핵심 가치 루프 — P0)*
- (a) **승인 큐.** 팀장이 [승인 큐]에 들어가면 → 시스템이 **'내 액션 필요(대기)' 항목을 상단 고정**한 필터 가능 테이블을 보여준다. 각 행: 신청자·휴가종류·기간·신청일·일수·사유·상태 태그(대기/승인/반려).
- (b) **인라인 승인.** 팀장이 행에서 [승인] → 상태 '승인(approved)', 해당 신청의 잔액이 **pending → used로 확정 차감**, 캘린더에 즉시 반영(개념적 실시간 — 정적 시연으로 충족).
- (c) **반려(사유 필수).** 팀장이 [반려] → **사유 입력 모달 필수**(강한 업계 관례) → 상태 '반려(rejected)', 해당 신청의 pending 차감 **복원**, 사유가 신청 상세·이력에 기록.
- (d) **충돌 사전 표면화.** 승인 전, 해당 기간의 **팀 캘린더 미리보기**(우측/하단)로 같은 기간 동료 부재·겹침을 보여준다(자동 차단 X — 가시화만).
- 엣지케이스: ① 대기 0건이면 빈 큐 안내(막다른 길 금지). ② 승인자 본인 신청의 **자기 승인 방지**(개념 — 다른 승인자/HR 라우팅은 P1 위임). ③ 처리됨 탭(승인/반려/취소)으로 이력 조회.
- 범위: **MVP는 단일 승인자(팀장→완료) 1단계.** 다단계 승인·위임(backup approver)·SLA 자동 에스컬레이션은 §10/P1 — 단 데이터 모델은 승인 단계를 별도 엔티티(Approval)로 두어 확장 가능하게(근거 있는 결정).

**F3. 잔여 연차 조회 (셀프서비스 · total/used/pending/remaining)** *(근거: 사용자 MVP ③; self-service의 핵심 — 매니저/HR 부담 감소, P0)*
- 직원이 [잔여]에 들어가면 → 시스템이 **종류별 잔액을 4개 값으로 분해**해 보여준다: **부여(total) · 사용(used) · 대기(pending) · 잔여(remaining = total − used − pending)**. 대기분을 분리 표시해 "신청했는데 아직 안 깎임" 혼란을 막는다(근거 있는 결정).
- 추가 표시(관례): 사용률(%), (개념) 소멸 예정일·D-day 배지, 부여 기준(입사일/회계연도) 배지.
- 엣지케이스: ① 잔여 0일 — 빈/소진 상태 명확히. ② 대기>0이면 "대기 중 N일 포함" 안내. ③ 종류별 분해(연차/반차는 같은 연차 풀에서 차감, 병가/공가는 별도 또는 무차감 — F1 LeaveType 플래그와 정합).
- **MVP에서 누적(accrual)·이월(carryover) 자동 계산 엔진은 범위 밖**(법적으로 복잡 — 입사일 vs 회계연도·비례배분·소멸). HR이 직원별 **부여일수·기준일을 수동 입력/시드**하는 모델. 화면은 표시 규칙(반차=0.5 차감 등)만 구현(근거 있는 결정 — 사용자 '개념적 백엔드' 범위 위 권고).

**F4. 팀 휴가 현황·캘린더** *(근거: 사용자 MVP ④; 매니저가 승인 전 충돌을 판단하는 도구 + 팀 가시성, P0~P1 → P0)*
- 모두가 [팀 캘린더]에 들어가면 → 시스템이 **월 단위 그리드**로 누가 언제 비는지 보여준다(행=멤버 또는 셀=날짜에 **휴가 종류별 색 바**). **팀 필터**·**같은 날 부재 N명 카운트**·**승인 대기 건 수** 위젯.
- 색 구분은 종류별(연차/반차/병가/경조사) — **색만이 아니라 라벨 병기**(a11y, G3).
- 엣지케이스: ① 같은 기간 다수 부재 시 **겹침 가시화**(자동 차단 아님 — 팀 단위는 first-come·역할 커버리지 정책은 사람 판단). ② 부재 0인 달 — 빈 그리드 정상 표시. ③ 종류 색 범례 표시.
- 범위: 월 뷰가 기본. **연 평면 뷰(flat year)**·팀 계층 트리는 P1(있으면 좋음). 실시간 반영은 개념(승인 시 캘린더 즉시 표시되는 정적 시연으로 충족).

**F5. 알림 (개념 — 인앱 상태)** *(근거: 사용자 MVP ⑤ + "알림은 개념으로 한정"; 상태 전이 가시성, P1 성격이나 골격상 MVP에 인앱만)*
- 사용자가 앱에 있으면 → 시스템이 **인앱 알림 목록 + 읽음/안읽음 뱃지**로 상태 전이를 표면화한다: 신청→승인자에게 "승인 요청", 승인/반려→신청자에게 "처리됨".
- **실제 발송(이메일·푸시·슬랙)은 없음** — UI 상태(목록·뱃지·토스트)로만 표현(사용자 의도와 일치, 근거 있는 결정).
- 엣지케이스: ① 알림 0건 빈 상태. ② 읽음 처리 후 뱃지 카운트 갱신.

**F6. 신청 상세 / 상태 이력 (감사 추적)** *(근거: 시장조사 grounded — '누가/무엇을/누가 승인/언제/어떤 종류로'를 수기 재구성 없이 추적; 취소·반려 건도 잔존시켜 이력 보존)*
- 사용자가 신청 행을 열면 → 시스템이 **신청 상세 + 상태 이력**(신청→승인/반려→취소)을 타임스탬프·처리자와 함께 보여준다.
- 엣지케이스: ① 취소·반려된 신청도 **시스템에 잔존**(삭제 아님 — 이력 보존). ② 본인/승인자/HR 권한별 표시 범위.

**F7. 인증/진입 UI (로그인 — 개념 멀티유저·역할)** *(근거: 표준 골격 default-IN; 디자인 산출물이므로 로그인 UI는 그린다 — 백엔드만 개념)*
- 사용자가 진입하면 → 로그인 화면(개념) → **역할(직원/팀장/HR) 선택 또는 시드 계정**으로 role-aware 홈에 진입한다. **실 인증은 범위 밖**(§10) — UI와 역할 전환 시연만.

**F8. 멤버/팀 관리 + 부여 연차 시드 (HR)** *(근거: 시장조사 grounded — '잔액은 직원이 셀프로 실시간 확인'·'HR이 부여일수·기준일을 관리'가 LMS 관례; **우선순위 = 사실상 P0** — F1의 잔액 초과 검증과 F3의 잔여 조회·G5 잔액 동기화 전체가 'HR이 부여한 시드 잔액'을 전제로 하므로, 이 기능이 없으면 핵심 루프가 시드 데이터 없이 빈다. SC9를 받치는 §4 기능으로서 §4→§7 추적을 닫는다.)*
- (a) **직원 목록·조회.** HR이 [멤버/팀 관리]에 들어가면 → 시스템이 직원 목록(이름·팀·역할·입사일·부여 기준·현재 잔여)을 테이블로 보여준다. 권한 없는 역할(직원/팀장)에는 **비노출**(RBAC, G3·§9).
- (b) **부여 연차 시드 설정(가장 load-bearing 액션).** HR이 한 직원의 종류별 **부여일수(LeaveBalance.total)와 부여 기준(입사일/회계연도)을 입력/수정**한다 → 시스템이 그 직원의 `remaining = total − used − pending`을 **즉시 재계산**해 F3 잔여 조회·F1 신청 검증·SC2 대시보드 위젯에 반영한다(불변식 G5).
- (c) **팀 배정.** HR이 직원의 소속 팀(`teamId`)·승인자(`managerId`)를 설정한다 → 이후 그 직원의 신청이 해당 팀장의 승인 큐(F2)로 라우팅된다.
- (d) **전사 현황·이력.** HR이 전사 휴가 현황·신청 이력(F6 감사 추적)을 팀 무관하게 조회한다.
- 엣지케이스(MVP에서 검증·경고로 구현):
  - ① **음수·하향 시드** — `total`을 0 미만으로 설정 차단. 이미 `used + pending`이 발생한 직원의 `total`을 그보다 낮게 내리면(잔여 음수 유발) **차단·경고**(기존 사용분과의 정합).
  - ② **부여 0(미설정) 직원** — 시드가 0/미배정인 직원은 F1 신청 시 '잔액 초과'로 막힘 → 멤버 화면에서 **'부여 미설정' 플래그**로 표면화(조용한 0 방지).
  - ③ **직원 0명(첫 실행)** — 멤버 0건 빈 상태에서 '직원 추가/시드 부여' 안내(막다른 길 금지, S9·G2).
  - ④ **시드 변경 이력** — 부여 변경은 처리자·시각과 함께 이력으로 잔존(F6 감사 — 수기 재구성 방지). 변경이 기존 승인 건 잔액을 소급 훼손하지 않음.
  - ⑤ **자기 시드 편집** — HR이 자신의 부여를 편집할 때도 동일 검증(자기 승인 방지와 별개, 시드는 허용하되 이력 기록).
- 범위: **계정 생성/비활성·실 권한 부여는 개념**(멀티유저·역할 시연 — 실 인증 백엔드는 §10). MVP는 시드된 직원 목록 위에서 **부여 연차 편집·팀 배정·현황 조회**가 핵심. 휴가 종류 마스터(LeaveType) 편집은 P1(MVP는 기본 세트 고정 — §4-2·B8).

### 4-2. 다음 (P1) — 다음 버전 후보 (이번 MVP 제외)

근거: 핵심 루프(신청→승인/반려→잔여반영→캘린더) 검증이 먼저. 아래는 가치는 인정하나 핵심 흐름 없이도 앱이 성립.
- **다단계 승인 + 위임(backup approver)** — 팀장→HR 2단계, 승인자 부재 시 대리. 데이터 모델은 이미 확장 가능(Approval 엔티티).
- **누적·이월·소멸 자동 계산 엔진** — 입사일/회계연도 비례배분, use-it-or-lose-it. 법적 복잡도 큼.
- **공휴일 마스터 + 자동 제외** — 영업일 계산에 공휴일 반영.
- **연 평면 뷰 + 팀 계층 트리** — 캘린더 고급 뷰.
- **실 알림 발송** — 이메일·슬랙·푸시(현재 인앱 개념만).
- **반반차(0.25일/2시간)** — 회사 자율 운영 단위.
- **인수인계 메모·대체자 지정** — 장기 휴가 시(현재 사유 필드로 일부 대체).
- **무급 전환 안내** — 잔액 초과 시 차단 대신 무급 옵션.
- **CSV/엑셀 내보내기 · 리포트** — HR 정산용.
- **휴가 종류 마스터(LeaveType) 편집 UI** — HR이 종류·차감 규칙·색을 추가/수정. MVP는 기본 세트(연차·반차·병가·경조사) 고정, F8은 부여 시드·배정까지만(B8).

### 관계
- **Team 1 : N User** — 한 팀에 여러 직원. 직원은 한 팀 소속.
- **User 1 : N LeaveRequest** — 한 직원이 0개 이상의 신청을 가진다(`requesterId`).
- **User(manager) 1 : N User(employee)** — `managerId`로 승인자→팀원(자기 참조 1:N). 승인 큐 라우팅의 근거.
- **LeaveType 1 : N LeaveRequest** — 한 종류가 여러 신청에 쓰임.
- **LeaveRequest 1 : N Approval** — 한 신청에 여러 처리 이력(MVP는 보통 1건, 취소 시 2건). 감사 추적(F6).
- **User 1 : N LeaveBalance** — 직원 × 종류 × 연도 단위로 잔액 레코드. (User × LeaveType × year 복합으로 사실상 1:1 per 조합.)
- **LeaveType 1 : N LeaveBalance** — 종류별 잔액.
- **User 1 : N Notification** — 수신자별 알림.
- **User 1 : N BalanceAdjustment** — 직원별 부여/시드 조정 이력(`targetUserId`). 처리자도 User(`actorId`, HR). 한 LeaveBalance 조합(user×type×year)이 0개 이상의 조정 이력을 가진다.
- **(불변식) BalanceAdjustment → LeaveBalance / User** — 부여 시드 변경은 BalanceAdjustment로 기록되며, `field=total`이면 대상 LeaveBalance(user×type×year)의 `total`을, `field=balanceBasis`이면 User의 `balanceBasis`(per-user)를 갱신한다(원장이 현재상태를 설명). 변경은 `used+pending` 미침범·기존 승인 건 소급 무훼손(F8 엣지①·④).
- **(불변식) LeaveRequest.status ⇄ LeaveBalance 동기화** — 신청 제출=pending+, 승인=pending−/used+, 반려=pending−, 승인후취소=used−. 상태 전이가 잔액을 일관되게 움직인다(G5).

### 상태 머신 (LeaveRequest.status)
```
[신청] → pending ─┬─ [승인] → approved   (pending→used 확정)
                  ├─ [반려] → rejected   (pending 복원, rejectReason 필수)
                  └─ [신청자 취소] → cancelled (pending 복원)
approved ─ [승인 후 취소] → cancelled    (used→remaining 복원, 미래 일자 한정)
```
- 반려/취소된 신청은 삭제하지 않고 이력으로 잔존(F6·감사). `cancel_requested`(취소 승인 흐름)는 P1 — MVP는 미래 일자 본인 취소 즉시 처리.

### 부여 연차 산정 (MVP 표시 규칙만)
- 근로기준법 참고 시드값(현실값): 1년 이상·80% 출근 시 **연 15일**, 1년 미만은 월 1일(최대 11), 3년 이상 2년마다 +1(최대 25). **데모 직원에 11~25일 범위 시드** 부여.
- **자동 발생/소멸/이월 엔진은 범위 밖**(P1). MVP는 `total`을 HR이 시드/입력하고, 화면은 '부여 N / 사용 K / 대기 P / 잔여 R'과 기준(입사일/회계연도) 배지만 표시.

- **직원 뷰(R1)**: 내 잔여(부여/사용/대기/잔여) 위젯(←F3) · 내 신청 상태 요약(←F1·F6) · 팀 부재 미리보기(←F4) · [+ 신청] CTA(→SC3).
- **팀장 뷰(R2)**: 승인 대기 큐 카운트(←F2, →SC4) · 팀 캘린더 요약(←F4) · 같은 날 부재 N명 · 본인 잔여(←F3).
- **HR 뷰(R3)**: 전사 휴가 현황·승인 대기 총계(←F8(d)·F4) · '부여 미설정' 직원 플래그 수(←F8 엣지②, →SC9) · 전사 캘린더 요약. §3에서 HR은 팀이 아닌 **전사** 1급 역할이므로 팀장 뷰의 '내 팀' 스코프 대신 전사 스코프 위젯을 둔다(역할 차이 명시 — 추적 F7→SC2가 HR에서도 닫힘).
- 빈 상태: 신청 0건·대기 0건·부여 미설정 0건이어도 막다른 길 없이 다음 행동 유도.

**SC3. 휴가 신청 폼.**
- 종류 선택(연차/반차/병가/경조사) · 기간(데이트 피커, 반차 시 오전/오후 토글) · 사유.
- **잔액 미리보기**(신청 일수·대기·예상 잔여) 실시간. 검증 경고(겹침·초과·기간 유효)·반차 0.5일 표시.

**SC4. 승인 큐 (관리자).**
- 필터 가능 테이블: '대기'/'처리됨' 탭. 행 = 신청자·종류·기간·일수·신청일·사유·상태 태그.
- 행별 [승인]/[반려](반려는 사유 모달 필수). 선택 시 **해당 기간 팀 캘린더 미리보기**로 충돌 표시.
- 빈 큐 상태 안내.

**SC5. 팀 캘린더 / 현황.**
- 월 그리드. 종류별 색 바 + **라벨 병기**(a11y). 팀 필터. 같은 날 부재 N명 카운트·대기 건 위젯. 색 범례.
- 셀/바 클릭 → 신청 상세(권한 범위). 부재 0 빈 그리드 정상.

**SC6. 잔여 연차 조회.**
- 종류별 카드: 부여/사용/대기/잔여 + 사용률(%) + (개념)소멸 D-day·기준 배지. 잔여 0 소진 상태 명확.

**SC7. 내 신청 목록 + 신청 상세/이력.**
- 내 신청 테이블(상태 태그·기간·종류). 행 → **상세 + 상태 이력**(신청→승인/반려→취소, 타임스탬프·처리자). 미래 승인 건 [취소] 액션. (감사 추적 F6.)

**SC8. 알림(개념).**
- 인앱 알림 목록 + 안읽음 뱃지. 항목 클릭 → 관련 신청. 읽음 처리. 빈 상태.

**SC9. 멤버 / 팀 관리 (HR/관리자).** *(← F8 — §4 기능 backing 있음)*
- 직원 목록(이름·팀·역할·입사일·부여 기준·현재 잔여). **부여 연차(잔여 시드) 설정·수정**(검증: 음수·기존 사용분 정합)·**팀/승인자 배정**·전사 현황/이력 조회. 부여 변경 이력. '부여 미설정' 플래그. 권한 없는 역할(직원/팀장)엔 비노출(RBAC). 직원 0명 빈 상태.

**SC10. 설정 / 프로필 (셸 화면).**
- 본인 프로필 조회(이름·팀·역할·입사일·부여 기준 배지) + 로그아웃. **표준 골격상 default-IN인 셸(shell) 화면** — 전용 핵심 기능을 두지 않고 F7(역할/세션)·F8(프로필 데이터)에서 읽어 표시만 한다. 편집 가능한 환경설정(알림 on/off 등)은 개념 라벨로만(범위 밖 — §10). 즉 SC10은 쓰기 기능을 만들지 않는 **읽기·세션 종료 셸**임을 명시한다(조용한 누락이 아니라 의도적 셸).

**화면 요구(공통).**
- 라이트 모드 기준. 1280px+ 데스크톱 폭 테이블·캘린더 그리드 무파손(G4 — 가로 스크롤·겹침 0).
- 상태(대기/승인/반려/취소)는 **색+텍스트/아이콘**으로 구분(색만 의존 금지, G3). 키보드/포커스 접근, 폼·모달·테이블·토글에 적절한 role/aria.
- 실 인증·실시간·알림 발송·잔액 엔진은 전부 목업/시드(독립 HTML) — 흐름·상태변화는 완결.
- **빈/첫 실행 상태를 표준 골격에 포함**(신청 0·큐 0·잔여 0·알림 0·부재 0) — 표면 빈약·막다른 길 방지.

## 8. 디자인 방향 (톤·레퍼런스·뷰포트·다크 모드)

- **톤.** **미니멀·전문적인 내부툴.** 정보 밀도와 스캔성의 균형 — 테이블·상태 태그·카운트 뱃지 중심. 화려한 일러스트·과한 그라데이션 지양. 업무에 집중되는 차분한 크롬(chrome).
- **레퍼런스.** **Linear / Notion류 차분·정돈**(인테이크). admin/대시보드 UX 베스트프랙티스: RBAC 역할별 뷰, 검색 가능한 audit trail, 명료한 상태 태그. 정보가 주연, 장식은 최소.
- **뷰포트.** **데스크톱 1280px+ 우선.** 좌측 사이드 내비 + 다열 본문(테이블·캘린더·폼). 모바일 1급 아님(§10) — 데스크톱 내부 업무 환경 기준. /design 인테이크(00-flow)에서 1280px+를 대상 뷰포트로 명시.
- **색 모드.** **라이트 모드 기본**(인테이크). 다크 모드는 범위 밖(§10·§11). 토큰은 표준 2단: primitive(중립 `--gray-50..900` 램프 + 액센트/상태색) → semantic 역할명(`--color-surface`·`--color-text`·`--color-primary`·`--color-success`·`--color-warning`·`--color-danger`). 브랜드 단어로 토큰명 짓지 않음. 상태색(대기/승인/반려)은 시맨틱 토큰으로. 라이트 모드 WCAG AA 충족(G3).
- **언어.** 한국어 UI. 모든 라벨·고지·상태 태그·빈 상태·날짜·일수("2일", "0.5일") 한국어.
- **상태 표현.** 휴가 종류·신청 상태를 **색+텍스트/아이콘** 일관 체계로(캘린더 색 바, 큐 상태 태그). 색맹 안전(범례·라벨 병기).

## 9. 비기능 요구사항

- **접근성(a11y).** 라이트 모드 텍스트/주요 컨트롤·상태 태그 대비 WCAG **AA**(본문 4.5:1, 큰 글자/UI 3:1). **상태·종류를 색만으로 구분 금지**(색+텍스트/아이콘/범례). 키보드 전체 조작, 폼·모달·테이블·캘린더에 라벨/aria(데이트 피커·정렬·필터 포함).
- **뷰포트.** 1280px+ 데스크톱 무파손(가로 스크롤·겹침·잘림 0). 테이블·캘린더 그리드가 넓은 폭에서 정렬 유지.
- **데이터 정합성(구조적 요구).** 잔액 불변식(`remaining = total − used − pending`)이 모든 상태 전이에서 유지. 승인=확정 차감, 반려/취소=복원. 본인 겹침·잔액 초과는 신청 단계에서 차단(음수 잔액 방지).
- **감사 추적(구조적 요구).** 모든 처리(신청·승인·반려·취소)에 처리자·타임스탬프 기록, 취소·반려 건도 잔존(F6). '누가/무엇을/누가/언제/어떤 종류로'를 수기 재구성 없이 추적.
- **역할 기반 가시성(RBAC).** 직원/팀장/HR이 보는 화면·데이터가 다름(승인 큐·멤버 관리는 관리자/HR만). 개념 멀티유저로 시연.
- **성능(합리적 기대).** 테이블·캘린더가 수십~수백 건 신청에서 체감 지연 없이 렌더(내부툴 규모 상정). 산출물 단계는 목업이라 수치 벤치마크 목표 없음.
- **독립 실행.** atelier 규칙상 산출물은 외부 의존성(외부 라이브러리·캘린더/데이트 피커 SDK·차트 라이브러리) 없는 독립 실행 HTML — 상태 변화용 인라인 JS 허용. 외부 연동·실 인증·실시간은 전부 목업, 실연동은 handoff 계약.

## 10. 범위 외 (Out of Scope)

이번 MVP가 **명시적으로 하지 않는 것** — 착수자가 추측하지 않도록 못박는다.

- **실 인증 / SSO / 권한 백엔드** — 의도적 제외. 로그인·역할 **UI는 그리되**(SC1) 백엔드는 개념(멀티유저·역할 시연만). 실 연동은 handoff.
- **실시간 동기화 백엔드** — 승인 시 캘린더 즉시 반영은 **정적 시연으로 충족**(개념). 실 실시간(WebSocket 등) 범위 밖.
- **알림 실 발송** — 이메일·슬랙·푸시. MVP는 **인앱 상태(목록·뱃지)만**(F5). 실 발송은 P1.
- **다단계 승인 + 위임 + SLA 에스컬레이션** — MVP는 단일 승인자 1단계. 데이터 모델만 확장 대비(Approval). 다단계는 P1.
- **누적·이월·소멸 자동 계산 엔진** — 입사일/회계연도 비례배분·use-it-or-lose-it. MVP는 HR 시드 부여 + 표시 규칙만. 자동 엔진 P1.
- **공휴일 자동 제외** — MVP는 주말만 영업일 제외. 공휴일 마스터 P1.
- **반반차(0.25일/2시간)** — 회사 자율 단위. MVP는 일·반차(0.5)까지.
- **결제 / 구독 / 저장 한도** — 내부툴이라 해당 없음.
- **모바일 1급 대응 / 반응형 분기** — 데스크톱 1280px+ 우선. 모바일 최적화 범위 밖.
- **다국어** — 한국어 단일.
- **급여·근태·HRIS 외부 연동** — 범위 밖(handoff에 매핑 계약으로 이관 가능).
- **CSV/엑셀 내보내기·리포트** — P1.
- **다크 모드** — 라이트 기본. 다크 P1(§11).
- **실제 외부 연동(산출물 단계)** — 데이트 피커/캘린더 SDK, 잔액 트랜잭션 DB, 영업일/공휴일 엔진 — 전부 **handoff 계약으로 이관**(산출물은 목업/시드). handoff에 넣을 것: 상태 머신↔잔액 동기화 계약, LeaveType 마스터 스키마, 영업일 계산 규칙, RBAC 가시성 매핑.
- **물리 데이터 스키마/저장 포맷 확정** — 개발 단계 결정(§5).

## 11. 미해결 이슈와 가정

> 시장조사로 근거 확보된 항목은 §1·§4·§5에서 **근거 있는 결정**으로 본문에 반영했다(반려 사유 필수·반차 0.5일·pending/remaining 분리·엣지케이스 5종·신청 이력 — 전부 grounded). 아래는 (a) 사용자가 답하지 않아 가정으로 닫은 것, (b) 여전히 추측이라 본문에서 단정하지 않은 것이다.

**A. 사용자 미답 → 가정으로 닫음 (확인되면 본문 승격).**
- **A1. (가정) 로그인/계정·역할 존재.** RBAC(직원/팀장/HR)가 핵심이라 계정·역할 개념은 존재로 가정. 인증 흐름 자체는 산출물 핵심이 아니라 목업/역할 전환 시연(F7·§10).
- **A2. (가정) 언어 한국어 단일.** UI·라벨·날짜 전부 한국어.
- **A3. (가정) 외부 연동은 목업, 실연동은 handoff.** 데이트 피커·캘린더·잔액 DB·영업일 엔진은 독립 HTML 규칙상 산출물 단계 목업/시드. 실연동 handoff(§10).
- **A4. (가정) 단일 승인자 1단계.** 사용자 '단순 승인/반려' 답에 따라 팀장→완료 1단계. 다단계·위임 P1(데이터 모델은 확장 대비).
- **A5. (가정→근거로 보강) 반려 사유 필수·잔액 4분할.** 사용자가 세부 미답이었으나 시장조사 grounded → '반려 사유 필수'·'total/used/pending/remaining 분리'·'본인 겹침/잔액 초과 차단'을 근거 있는 결정으로 본문에 넣음.
- **A6. (가정) 라이트 모드 단일.** 인테이크 '라이트 기본·다크 범위 밖'을 라이트 단일로 해석. 다크 동시 제공 여부 확인 필요(현재 P1).
- **A7. (가정) 데스크톱 1280px+ 단일 셸.** 모바일 1급 아님. 반응형 분기 없이 데스크톱 셸(§8·§10).

**B. 본문에 단정하지 않은 추측 (검증/결정 필요).**
- **B1. (추측·guess) 운영 목표 수치.** O1(경유율 ≥90%)·O2(리드타임 ≤1영업일)·O3(HR 문의 ≥50%↓)·O4(신청 ≤2분)는 내부툴 가치 제안 기반 방향성 권고 — 본 도메인 전용 실측 아님. **도입 후 4주 베이스라인으로 확정·보정.** 활성률 70~80%는 야심적 상한이라 KPI에서 제외하고 행동지표로 대체(시장조사: HRIS 평균 실사용 32%).
- **B2. (결정 보류) 영업일/공휴일 계산 깊이.** MVP는 주말 제외만. 공휴일 자동 제외는 공휴일 마스터 필요 → P1. 한국 공휴일·대체공휴일 처리는 handoff 계약 시 결정.
- **B3. (결정 보류) 과거일자 소급 신청 허용.** MVP는 미래·당일 기본, 과거일 신청은 경고. 소급 허용 정책(병가 사후 등록 등)은 조직 규정 의존 — /plan 후속 또는 handoff에서 확정.
- **B4. (결정 보류) 잔액 모델 — 4필드 vs 트랜잭션 원장.** MVP는 total/used/pending/remaining 4필드로 단순화. 감사·되돌리기에 우월한 트랜잭션 원장(ledger)은 물리 구현·P1 선택(§5). 대상 프로젝트가 자기 컨벤션으로 결정.
- **B5. (결정 보류) 승인 후 취소 흐름.** MVP는 미래 일자 본인 취소 즉시 처리(잔액 복원). '취소 승인'(cancel_requested, 승인자 재확인) 단계는 P1.
- **B6. (구현 자유) 데이터 영속·저장 메커니즘.** §5 엔터티는 개념 모델 — 저장 위치/포맷/제약은 대상 프로젝트가 자기 컨벤션으로 결정. atelier는 구현에 묶이지 않는다.
- **B7. (결정 보류) 캘린더 표현 — 행=멤버 vs 셀=날짜.** 월 그리드에서 멤버 행 타임라인 방식과 날짜 셀 내 부재 바 방식 중 무엇을 주로 할지 /design에서 확정(요구는 무파손·색+라벨·N명 카운트).
- **B8. (추측·guess) 휴가 종류 기본 세트.** MVP 연차·반차·병가·경조사는 한국 표준 기반 권고 — 조직별로 공가·무급·특별휴가 추가 가능(LeaveType 마스터라 설정으로 흡수).
