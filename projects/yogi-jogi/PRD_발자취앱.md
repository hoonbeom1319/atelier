# PRD — (가칭) 발자취: 다녀온 곳을 코스로 남기는 앱

> 버전 0.1 (MVP 기획 확정본) · 작성 단계: 개발 착수 직전
> 이 문서는 기획 합의 내용을 개발자가 추가 질문 없이 착수할 수 있는 수준으로 정리한 것이다.
> 물리 스키마(컬럼 타입·인덱스·정규화), 정교한 디자인 시안, 해외 대응은 의도적으로 범위 밖에 둔다.

---

## 1. 배경과 문제 정의

여행·데이트·가족 나들이처럼 "어딘가 다녀오는" 사람들은 지금 인스타그램이나 휴대폰 카메라 롤에 사진을 쌓는다. 그러나 이것은 단순한 사진 더미라, **내가 공간을 어떻게 누볐는지** — 어디를 어떤 순서로 돌았고, 지금까지 어디어디 가봤는지 — 가 한눈에 남지 않는다.

이 앱은 런닝 앱이 "달린 경로"를 지도에 남기듯, 한 번의 나들이를 **"내 코스"(지도 위 동선 + 그 위에 얹은 사진·태그·짧은 글)** 로 남겨준다. 코스가 쌓이면 **"내 발자취 지도"** 가 된다.

**핵심 가치 구조 (이 순서가 흔들리면 안 됨):**
- **1층(심장) — 기록:** 혼자 써도 가치가 있다. 보는 사람이 0명이어도 내 발자취가 쌓인다. → 출시 직후 사용자가 적어도 앱이 죽지 않는 이유.
- **2층 — 연결:** 그 코스를 친구·팔로워에게 보여주고 "같이 하고 있다"는 느낌을 얻는다. 1층 위에 얹힌다.

**경쟁/차별점:** "코스를 추천·큐레이션해주는" 영역은 데이트팝(600만 다운로드), 트리플 등 강한 선수가 이미 점유하고 있다. 이 앱은 그 시장에 들어가지 않는다. 차별점은 **"진짜 사람이 실제로 다녀온 진짜 코스"가 기록의 부산물로 쌓인다**는 점이며, 인스타가 해주지 못하는 "동선·발자취"를 채운다. 인스타를 대체하지 않고 보완한다.

## 2. 목표와 성공 지표

**프로젝트 성격:** 메인은 "나와 내 주변이 잘 쓰는 것", 반응이 좋으면 "진짜 서비스로" 전환.

지표는 **다운로드 수가 아니라 "코스가 계속 쌓이고 사람이 돌아오는가"** 로 본다.

| 구분 | 지표 | 판정 |
|---|---|---|
| 핵심 (살아있나) | 출시 4주차에, 리마인드 없이 친구 2명이 **각자 2주에 코스 1개씩** 꾸준히 올린다 | 둘 다 → 성공 / 한 명이라도 꾸준 → 가능성 있음 / 둘 다 1~2주 내 이탈 → 제품을 고쳐야 함(사람을 더 모을 게 아니라) |
| 졸업 (2번 전환 신호) | 내가 직접 안 깔아준 사람 1명이 들어와 코스를 올린다 | 발생 시 사업 모드 전환 검토. 큰 버전 = 그런 사람이 한 달 넘게 꾸준히 증가 |

이 두 숫자의 목적은 **감정이 아니라 계기판으로 판단하기 위함**이다. 잘되는데 불안해서 접거나, 안 되는데 미련으로 붙잡는 것을 막는다.

## 3. 타깃 사용자와 핵심 사용 장면

**핵심 사용자:** 30대, 가족 단위로 다니며, 이미 인스타그램을 사용하는 사람. (개발자 본인과 그 친구 2명이 1차 사용자.)

함의:
- 가족 단위 → 올라오는 코스는 화려한 핫플보다 **주말 외출·아이 동반 나들이 등 일상 반경**일 가능성이 높다.
- 이미 인스타 사용 → 인스타로 **내보내기**가 적이 아니라 친구가 될 수 있다(Later).

**핵심 사용 장면 (확정):**
> 30대 사용자가 가족과 주말 나들이/여행을 다녀온 **그날 저녁, 집에서**, 폰에 쌓인 사진 수십~수백 장을 앱에 던진다. 앱이 EXIF로 장소·시간을 읽어 그날의 동선과 사진을 자동으로 코스로 엮어준다. 사용자는 미분류된 몇 장만 손보고, 짧은 글·태그를 얹어 "그날의 코스"를 완성한다. 그리고 그걸 친구·가족에게 보여준다.

## 4. 기능 요구사항 (우선순위 + 근거)

기능 채택 기준: **① 심장(1층)을 거스르지 않는가 ② 1인 개발에 과하지 않은가 ③ 성공 지표("코스를 올리는가")에 기여하는가.** 셋 중 하나라도 어기면 MVP에서 뺀다.

### 4.1 MVP — 1층: 기록 루프 (협상 불가)

| 기능 | 동작 (사용자가 X 하면 시스템은 Y) | 해결하는 문제 |
|---|---|---|
| 사진 무더기 업로드 | 사용자가 갤러리에서 사진 수십~수백 장을 한 번에 선택하면, 시스템은 전체를 업로드 대상으로 받는다 | 한 장씩 올리면 아무도 안 한다 (90장 검증 완료) |
| EXIF 자동 조립 | 시스템은 각 사진의 EXIF(위·경도·촬영시각)를 읽어 시간·위치로 묶고, 방문지(Stop)를 시간순으로 정렬해 코스 초안을 생성한다 | "어디를 어떤 순서로 갔나"를 손입력하면 이탈. 인스타가 안 해주는 핵심 가치 |
| 역지오코딩(장소명 부여) | 시스템은 Stop의 좌표를 역지오코딩하여 "전주 한옥마을" 같은 장소명을 붙인다 | 사람은 좌표가 아니라 이름으로 장소를 기억함. **자동 조립 magic의 핵심 부품** |
| 미분류 사진 보정 ★최우선 | (1) 위치 있으나 잘못 묶인 사진: 사용자가 사진을 **탭 선택 → 시트로 올라온 Stop 목록에서 탭**하면 해당 Stop으로 이동 (드래그 ❌). 가능하면 **근처 Stop 후보**를 먼저 추천해 한 번 탭으로 끝낸다. (2) 위치 없는 사진: **지역명 검색 → (실패 시) 핀 직접 이동**. (3) Stop **합치기/쪼개기** | **이 화면이 막히면 친구가 창을 닫고 성공 지표가 0이 된다.** 최대 이탈 위험 구간 |
| 코스 동선 지도 뷰 | 사용자가 코스를 열면, 시스템은 동선을 지도 위에 런닝 앱처럼 그려 보여준다 | "그날 어떻게 돌았나"를 한눈에 |
| 발자취 누적 지도 | 시스템은 사용자의 모든 코스의 모든 Stop을 한 지도에 표시한다 | "지금까지 어디어디 가봤나" = 1단계 핵심 문제. **재방문 엔진** |
| 짧은 글·태그 | 사용자가 **코스 단위로** 짧은 글과 태그를 입력(선택)한다. 미입력이어도 코스는 완성된다 | 올리는 마찰을 0에 가깝게 유지 (Stop별 입력은 의도적으로 배제) |

**엣지 케이스:**
- EXIF가 아예 없는 사진(카톡·인스타로 받은 것, 스크린샷): 위치정보가 제거돼 있을 수 있음 → 전부 "미분류"로 유입 → 보정 화면에서 처리.
- 한 장소에서 다수 사진: Stop 1개 + Photo 다수로 묶임. 자동 묶음이 과·소 분할되면 합치기/쪼개기로 보정.
- 미분류 = 데이터상 "Stop이 비어 있는 Photo"로 자연 표현됨(별도 플래그 불필요).

### 4.2 MVP — 2층: 얇은 연결

| 기능 | 동작 | 근거 |
|---|---|---|
| "다 같이" 화면 | 시스템은 사용자가 팔로우한 사람들의 코스를 시간순 목록으로 보여준다 | "다른 사람도 올리고 있다"는 느낌의 90%를 줌. 풀 소셜의 1/10 비용 |
| 일방 팔로우 | 사용자가 상대를 "팔로우"하면 즉시 구독된다 (요청·수락·차단 없음) | "내가 정한 사람 걸 본다" 욕구 충족. 양방향 친구 시스템 불필요(맞팔=일방 팔로우 2개) |
| 인앱 배지 | 팔로우한 사람이 새 코스를 올렸고 사용자가 아직 안 봤으면, 시스템은 "다 같이"에 안 읽은 개수/빨간 점을 표시한다 | "확인 안 하면 표시되는" 욕구 충족. 그 화면 구현 시 거의 공짜 |
| 코스 링크 공유 | 사용자가 코스 공유를 누르면, 시스템은 웹 링크를 생성한다. 앱 없는 사람도 웹으로 열람 | 받은 친구의 친구 = **졸업 신호의 씨앗** |
| 코스별 공개 설정 | 코스는 **기본 비공개**. 사용자가 명시적으로 공개로 켤 때만 공유·노출된다 | 위치정보=프라이버시 폭탄. 안전 기본값 |

### 4.3 Later (졸업 후)
좋아요·댓글, 푸시 알림 / 가족 공유 앨범·아이 태그 / 집·민감 장소 자동 가리기 / 인스타 내보내기 / 풀 소셜 피드.
※ 양방향 친구 맺기는 **영구 불필요**(일방 팔로우로 대체).

### 4.4 Out of Scope (이번 앱이 거부한 방향)
- 남의 코스로 내 여행 **계획·추천·탐색** (= B방향. 데이트팝·트리플 영역, 정면승부 회피)
- 여러 사람이 한 코스 **공동 편집** (= 해석2. 닭-달걀 함정 재유입)
- **GPS 실시간 추적** (= 다녀온 뒤 업로드 방식을 택함)
- **공유 마스터 장소 엔터티** (탐색의 입구, 범위 팽창 유발)
- **다중 지도 동시 운영**

## 4-1. 개념 데이터 모델

핵심 엔터티 5개. 물리 스키마(타입·인덱스·정규화)는 개발 착수 시 확정 — 범위 밖.

**엔터티 및 주요 속성**
- **User** — 닉네임, 프로필 이미지, "다 같이 화면 마지막 조회 시각"(배지 계산용)
- **Course** — 제목, 나들이 날짜, 공개여부(기본 비공개), 공유 링크 식별자, **짧은 글, 태그**
- **Stop** — 장소명(역지오코딩 결과), 대표 위·경도, 도착시각, 코스 내 순번
- **Photo** — 이미지, EXIF 위·경도, 촬영시각
- **Follow** — User→User, 방향 있음(일방)

**관계**
- User `1:N` Course (코스는 한 사람 소유)
- Course `1:N` Stop (순서 있는 방문지)
- Course `1:N` Photo (직속) / Stop `0~1 : N` Photo (선택적) → **Stop 빈 Photo = 미분류**
- User `N:M` User (일방 팔로우, 자기참조)

**텍스트 ERD**
```
User ──1:N── Course ──1:N── Stop ──1:N── Photo
  │             │ (글·태그)                ▲
  │             └──────────1:N─────────────┘   (Photo는 Course에 직속, Stop엔 선택적)
  │
  └──N:M(자기참조, Follow)── User            (일방 팔로우)
```

**이 모델의 핵심 설계 결정 3가지**
1. 미분류 사진은 별도 장치 없이 "Stop 미지정 Photo"로 표현된다. 보정 = 그 Photo에 Stop을 채우는 동작.
2. 발자취 누적 지도·인앱 배지는 **새 엔터티가 아니라 기존 데이터의 다른 view**다.
3. **공유 마스터 장소를 일부러 만들지 않는다.** 각 Stop이 자기 장소명·좌표를 보유한다. → 지도 교체 시 기존 코스가 안전하고, B방향(탐색)으로 새지 않는다. (단, 장소명은 역지오코딩 출처에 의존 — 6장 참조)

## 5. 사용자 흐름과 주요 화면

**핵심 흐름 (이게 막히면 앱이 죽음):**
`빈 상태 → 사진 선택 → 자동 조립(로딩) → 검토·보정 → 마무리 → 코스 상세`

1. **빈 상태(코스 0개):** 한 줄 설명("다녀온 곳 사진을 올리면 지도에 코스가 그려져요") + 완성 예시(샘플 코스) 1개 + 큼직한 "첫 코스 만들기" 버튼. **긴 튜토리얼 금지** — 목적은 가르치기가 아니라 첫 업로드로 미는 것.
2. **사진 선택:** 갤러리에서 무더기 선택.
3. **자동 조립 로딩:** EXIF 읽어 Stop·동선·장소명 초안 생성.
4. **검토·보정 ★:** (4.1 미분류 보정 참조) 탭-투-지정, 근처 Stop 추천, 지역검색·핀, 합치기/쪼개기.
5. **마무리:** 제목·짧은 글·태그(선택), 공개여부(기본 비공개).
6. **코스 상세:** 동선이 그려진 완성 코스. 발자취 누적 지도에도 합산됨.

**전체 화면 목록**

*핵심 흐름 화면:* 빈 상태 / 사진 선택 / 자동 조립 로딩 / 검토·보정(★최우선) / 마무리 / 코스 상세

*곁가지 화면:* 홈·내 코스 목록 / 발자취 누적 지도 / 다 같이 / 사용자 프로필(공개 코스+팔로우) / 코스 웹뷰(앱 없는 사람용) / 온보딩·로그인·설정(최소)

*권장 구조:* 하단 탭 3개(내 발자취 / 다 같이 / 프로필) + 가운데 "새 코스" 버튼. (디자인 단계에서 확정)

## 6. 기술적·일정상 제약

- **만드는 사람:** 본인 (FE/BE 가능), 클로드코드와 협업. 평일 하루 약 6시간. → **MVP 도달 가능, 무리한 일정 아님.**
- **인프라:** FE Vercel / **BE·DB·스토리지·인증 Supabase 확정** (EC2 직접 구축은 Out — 핵심에서 멀고 시간 소모). 단 사진 다량 저장 시 스토리지 한도/비용은 사용자 증가 후의 행복한 고민.
- **로그인:** Supabase 기본 제공 **카카오 / 지메일(구글) / 이메일** 3종.
- **MVP 플랫폼:** **모바일 웹**으로 시작. 추후 웹뷰로 감싸 앱 출시 검토(현 결정에 영향 없음).
- **최대 리스크:** **검토·보정 화면** 한 곳에 기술 난이도·시간이 집중. → "제일 무서운 것부터" 원칙으로 **EXIF 자동 조립 + 검토·보정 화면을 1순위로 구현**(동의됨).

**지도 결정 (최종):**
- 카카오맵으로 시작(한국 장소·검색·역지오코딩 우위. MVP 무대가 한국).
- **지도 표시 + 장소 검색 + 역지오코딩(좌표→장소명)** 세 기능을 **하나의 추상화 레이어로 감싼다.** 앱 코드는 `지도.마커찍기()`, `지도.장소검색()`, `지도.좌표를장소명으로()` 형태의 껍데기만 호출.
- 역지오코딩은 **자동 조립의 핵심 부품**이므로 지도 의존성은 단순 표시가 아니라 앱 심장에 닿아 있음.
- 향후 해외 대응은 "표시 지도 통째 교체"가 아니라 **좌표 위치 기반으로 역지오코딩·검색 출처를 분기**(한국 좌표→카카오, 해외 좌표→구글)하는 방식으로 접근.
- 기존 저장 코스는 장소명이 문자열로 박혀 있어 지도 교체에도 안전.

## 7. 디자인 방향 (디자인 단계 출발점)

- **톤:** 따뜻하고 가벼운 기록 도구. 자랑/경쟁보다 "내 발자취가 쌓이는" 정서. 가족 30대 타깃에 맞는 부드러움.
- **레퍼런스(참고용 2~3):** 런닝 앱의 경로 시각화(예: Strava류의 동선 표현), 일상 기록 앱의 타임라인 정서(예: 데이원류), 사진 중심 카드 레이아웃.
- **우선순위:** **모바일 우선**(핵심 동작이 폰에서의 사진 업로드·지도 조작). 데스크톱은 후순위.
- **다크모드:** MVP 범위 밖(Later). 라이트 기준으로 시작.
- **핵심 화면 디자인 난이도:** 검토·보정 화면이 UX의 승부처. 탭-투-지정·근처 추천을 모바일에서 손가락 동선이 짧게 설계할 것.

## 8. 비기능 요구사항

- **프라이버시 (중요):** 사진 EXIF에서 위·경도를 추출한다는 것은 집·아이 동선 등 민감 좌표가 노출될 수 있음을 뜻한다. **모든 코스 기본 비공개**, 공유는 코스별 명시적 활성화 시에만. 집·민감 장소 자동 가리기는 Later.
- **성능:** 사진 수십~수백 장 업로드·EXIF 파싱이 모바일 웹에서 멈춤 없이 동작해야 함. 로딩 중 진행 상태 표시.
- **데이터 보존:** 코스·사진은 사용자 자산. 유실 방지(클라우드 저장).
- **확장성:** 초기 소수 사용자 기준. 대규모 트래픽 최적화는 졸업 후.

## 9. 미해결 이슈 (Open Issues)

1. **해외 좌표 역지오코딩 전략:** 카카오 역지오코딩을 해외 좌표에 쓰면 결과가 부실하거나 빈 값일 수 있어, 해외 사진은 이름이 안 붙고 좌표만 남을 수 있음. 해외 지원 시 "좌표 기반 출처 분기"로 해결 예정. **MVP 범위 밖.**
2. **지도 교체 시 데이터 일관성:** 구글로 교체 후 새로 생성되는 코스의 장소명 형식이 기존(카카오)과 달라질 수 있음(예: "전주 한옥마을" vs 영문 주소). 해외 도입 시 정규화 정책 필요.
3. **미분류 비율의 실제 크기:** EXIF 없는 사진(카톡 수신분 등) 비중이 높으면 자동 조립 가치가 하락. 실사용 데이터로 측정 필요.

## 10. 가정 (Assumptions)

1. **[착수 전 반드시 검증 — 최우선]** 실제 폰(iOS 사파리 + 안드로이드 크롬 **둘 다**)에서 갤러리 사진을 모바일 웹으로 업로드할 때 **EXIF 위·경도가 보존되어 넘어온다.** 일부 모바일 브라우저/OS는 프라이버시 명목으로 업로드 시 위치정보를 제거할 수 있음. 이것이 깨지면 모든 사진이 미분류가 되어 앱 전제가 무너지므로, **검토·보정 화면 착수보다 먼저 이 검증을 수행한다.** (90장 테스트가 어느 환경이었는지 재확인 필요.)
2. 해외 여행 사진은 EXIF가 비교적 잘 보존된다(일반적 경향, 미검증) — 실제 해외 사진 표본으로 확인 필요.
3. 1차 사용자(본인+친구 2명)가 30대·가족·인스타 사용자라는 특성이 초기 코스 유형(일상 반경)을 대표한다.
