# 요기조기 (Yogi-Jogi) — PRD

> **장소 중심 사진 SNS.** 인스타그램이 *시간* 축으로 흐른다면, 요기조기는 *장소(POI)* 축으로 쌓인다. 사진을 다중 선택하면 EXIF(촬영 위치·시각) + 역지오코딩으로 장소별 자동 분류하고, 지도 중심 홈에서 내·팔로잉의 사진 핀과 동선 라인으로 회상한다.
>
> - **아키타입:** 모바일 소비자 사진 SNS (인스타급 table-stakes) + 지도 도구(네이버지도/구글맵류) 하이브리드
> - **뷰포트:** 모바일 390px 기본, 라이트 우선 · 한국어 UI
> - **이 문서의 산출 대상:** 개발 인계용 *디자인 산출물*(화면·플로우·토큰). 백엔드 실인증·실시간 GPS는 범위 밖이되, 진입·온보딩 UI는 그린다.
> - **근거:** 본문의 "[근거]"는 동봉 시장조사(Mapstr·Polarsteps·Swarm·Instagram Map·Foursquare Places·Google Photos·카카오 Local API 등)로 확보된 결정이다. "[가정]"은 미검증으로 §11에 모은다.

---

## 1. 배경과 문제 정의

### 1.1 배경
사람들은 여행·카페투어·산책에서 대량의 사진을 찍지만, 그 사진은 시간순 카메라 롤이나 인스타 피드에 흩어진다. "그때 그 카페가 어디였더라", "이 동네에서 내가 뭘 봤더라"를 **장소로 되짚는** 도구는 부재하다. Google Maps Timeline + Photos가 "방문 장소 핀 + 이동 경로로 여행을 다시 사는" 경험을 제공해 이 니즈가 실증되지만 [근거: Google Maps Timeline], 그것은 개인 회상 도구일 뿐 **소셜(팔로우·발견)** 이 약하다.

### 1.2 문제 정의
1. **시간 축 SNS의 한계** — 인스타는 게시 시각으로 흐르므로, "이 장소에 누가 무엇을 남겼나"를 모으지 못한다. 위치는 게시물당 1개의 부가 태그일 뿐이다.
2. **업로드 마찰** — 위치를 일일이 수동 태깅하는 부담. 사진에는 이미 EXIF에 촬영 위치·시각이 들어 있는데도 활용되지 않는다.
3. **장소 회상·발견의 부재** — 내가 가본 곳을 지도에서 한눈에 보고, 친구가 좋아한 장소를 발견하는 경험이 기존 SNS엔 없다.

### 1.3 해결 가설 (차별점)
경쟁 카테고리에는 4개 아키타입이 있다 [근거]: Mapstr(수동 장소 북마크 SNS), Polarsteps(상시 GPS 동선 추적), Swarm(수동 체크인 라이프로그), Instagram Map(기존 SNS에 지도 레이어). **어느 하나도 "사진 다중선택 → EXIF로 POI 자동 군집 업로드 + 장소핀 소셜 피드 + 사후 동선 라인"을 한 앱에 묶지 않는다.** 요기조기의 빈칸이 바로 이 조합이다:
- **업로드 마찰 제로** — EXIF 자동 POI 분류(구글/애플 포토 Places 방식).
- **장소(POI) 단위 소셜 피드** — Mapstr식 팔로우 + 공개/비공개.
- **사후 동선 라인** — Polarsteps식 경로를, *상시 추적이 아니라* 업로드된 사진의 촬영시각순으로 잇는 폴리라인으로 [근거: Polarsteps는 상시 GPS라 배터리·프라이버시 부담 → 요기조기는 EXIF 기반 사후 동선으로 회피].

### 1.4 핵심 리스크 (설계가 정면으로 다뤄야 할 것)
- **GPS 없는 사진이 흔하다** — 스크린샷·SNS 다운로드본(인스타/페북/왓츠앱은 업로드 시 EXIF 제거)·실내·위치권한 OFF [근거]. 따라서 **수동 분류는 엣지케이스가 아니라 MVP 1급(first-class) 흐름**이다.
- **위치 공유 프라이버시 백래시** — Instagram Map은 2025년 출시 직후 백래시 [근거: NBC/CBC]. 카카오맵 즐겨찾기가 사생활(모텔 기록)을 의도치 않게 노출해 기본값을 비공개로 바꾼 사례 [근거: 한국일보]. → 공개범위 제어·옵트인이 설계 1급 요소.
- **좌표→POI 매핑은 본질적으로 모호** — 도심 빌딩숲·실내 몰은 GPS가 수십~수백m 튀고 한 좌표에 여러 매장 → "후보 POI 리스트 + 사용자 확정"이 필수 [근거].

## 2. 목표 및 성공 지표

> 이 제품은 신규 카테고리라 신뢰할 절대 벤치마크(D1/D7 리텐션·MAU)가 외부에 없다 [근거: 시장조사 confidence=guess]. 따라서 **절대 수치는 못 박지 않고**, 차별 가설을 직접 검증하는 **선행 지표(leading metric)** 를 정의하고 수치는 출시 후 실측으로 보정한다(§11 가정). 아래는 디자인이 끌어올려야 할 설계 목표다.

### 2.1 북극성 지표 (North Star)
**"첫 업로드 완료율"** — 신규 가입자가 가입 후 첫 세션(D0) 안에 사진을 1건 이상 게시 완료한 비율. 차별점(업로드 마찰 제로)이 직접 드러나는 지표 [근거: 신규 소셜앱은 가입자의 첫 의미있는 행동 완료가 리텐션을 2~3배 가른다].

### 2.2 성공 지표 (측정 가능 — 정량·기간·기준)

| # | 지표 | 정의 | 목표(설계 가설) | 측정 기간 | 근거/비고 |
|---|------|------|-----------------|-----------|-----------|
| M1 | 첫 업로드 완료율 | D0 가입자 중 첫 게시 완료 비율 | ≥ 40% [가정] | 출시 후 D0 코호트 | 북극성. 자동분류가 마찰을 낮춘다는 가설 검증 |

### 2.3 정성 목표 (디자인이 달성해야 할 경험)
- "MVP라도 인스타와 최소 동등한 경험" — table-stakes 화면(피드·프로필·검색·알림·팔로우)이 빈약하지 않다 [근거: 사용자 명시 요구].
- 소셜 로그인 우선 배치(전환 이메일 대비 2~3배 [근거]) · 권한 프라이밍(opt-in 20~40%↑ [근거]).

## 3. 타깃 사용자·시나리오

### 3.1 1차 타깃
- **카페투어·맛집·전시 등 "장소 소비"가 취미인 20~30대** (한국어 UI). 사진을 많이 찍고, 가본 곳을 기록·과시·회상하려는 욕구가 큼.
- **여행 기록자** — 다녀온 여행을 동선으로 되짚고 친구와 나누고 싶은 사용자 [근거: Google Maps Timeline 회상 니즈].

### 3.2 2차 타깃
- 친구의 "검증된 장소"를 발견·팔로우하려는 캐주얼 탐색자(직접 기록은 적음, 소비 위주).

### 3.3 페르소나
- **지수(27, 카페투어러)** — 주말마다 성수·연남 카페 3~4곳을 돈다. 사진 수십 장을 찍지만 인스타엔 한두 장만 올리고 나머지는 묻힌다. "내가 간 카페들을 지도에 핀으로 모으고, 친구들에게 추천하고 싶다."
- **민호(31, 여행자)** — 제주 3박4일 사진 200장. 동선으로 되짚어 보고 싶지만 Google Photos는 소셜이 없고 인스타는 시간순이라 장소가 흩어진다.

### 3.4 핵심 시나리오
1. **(자동분류 happy path)** 지수가 카페투어 후 30장을 다중선택 → 앱이 EXIF로 "블루보틀 성수 / 대림창고 / 서울숲" 3개 POI 그룹으로 자동 분류 → 지수가 확인하고 그룹별 메모 한 줄 → 공개로 게시 → 지도 홈에 핀 3개 + 동선 라인.
2. **(수동분류 폴백)** 민호가 인스타에서 받아둔 사진(EXIF 0)을 올림 → 전량 "위치 없음" 묶음 → 지도 핀 드래그/장소 검색으로 일괄 지정.
3. **(발견·소셜)** 지수가 검색에서 친구를 찾아 팔로우 → 지도 홈에 친구의 핀이 뜸 → "블루보틀 성수" 핀 탭 → 바텀시트로 그 장소의 친구 사진·메모를 봄.

## 4. 기능 요구사항

> 규율: **모두 IN 금지.** P0=차별 코어 + 인스타 table-stakes(빈약 금지), P1=다음, P2=나중. 각 기능에 정당화. "사용자가 X → 시스템 Y" + 엣지케이스.

### 4.0 우선순위 한눈에
- **P0 (MVP) —** 인증·온보딩 UI / 지도 중심 홈(핀·클러스터·동선) / 다단계 업로드(EXIF 자동분류 + 수동 보정) / 피드 / 검색(장소·사람·태그) / 프로필(내 지도·그리드) / 친구 프로필 / 팔로우 / 공개·비공개 / 알림 / 빈 상태.
- **P1 (다음) —** 좋아요·간단 댓글 / 장소 노출 정밀도 토글(정확좌표 vs POI만 vs 동네) / 비공개 계정 팔로우 승인 / 게시물 상세 풍부화.
- **P2 (나중) —** 게이미피케이션·배지 / 협업·공유 지도 / 실시간 위치공유 / AI 추천·여행책 / POI 운영상태 판정. (전부 [근거]로 후순위 — 차별 아님·프라이버시 부담·스코프.)

---

### P0 — MVP

#### F1. 인증 화면 (UI만, 백엔드 범위 밖) — P0
table-stakes. Supabase 제공 **카카오 / 구글(지메일) / 이메일** 로그인 [근거: 사용자 답]. 소셜 버튼을 이메일 폼 위에 [근거: 소셜 전환 2~3배].
- 사용자가 카카오 버튼 탭 → 시스템: (디자인에선) 더미 인증 후 온보딩/홈 진입. *실 OAuth 연동은 범위 밖, 화면은 그린다.*
- 엣지: 이메일 형식 오류·빈 입력 → 인라인 검증 메시지. 신규/기존 분기(회원가입 ↔ 로그인 토글).

#### F2. 온보딩 + 권한 프라이밍 — P0
[근거: 위치 SNS 표준 + 백래시 교훈]. 권한은 OS 팝업 전 **가치 설명 프라이밍 화면**을 선행.
- 가치 소개 1~3장(지도 중심이 무엇인지) → **위치 권한 프라이밍**(지도 홈 진입/첫 업로드 직전) → **사진 권한 프라이밍**(업로드 탭 누를 때) → (선택) 팔로우 시드 → 첫 업로드 유도 CTA.
- 사용자가 "허용 안 함" → 시스템: 기능별 폴백(위치 거부 시 수동 입력 폴백, 사진 거부 시 빈 갤러리 안내 + 재요청 프라이밍).
- 엣지: iOS 제한적 사진 접근(일부만 허용) → 선택된 사진만 노출 안내.

#### F3. 지도 중심 홈 — P0 (차별 코어)
[근거: 사용자 답 ①, 지도앱 패턴]. 전체 지도에 내·팔로잉의 사진이 **POI 핀**으로, 동선 **라인**으로.
- 사용자가 지도 진입 → 시스템: 뷰포트 내 POI 핀 렌더. 한 POI에 사진 여럿이면 **카운트 배지**. 줌아웃으로 핀 밀집 → **클러스터 마커(숫자 버블)**, 탭하면 영역 확대 [근거: 마커 클러스터링 관례].
- 사용자가 핀 탭 → 시스템: **바텀시트 장소 상세**(Google Maps식) — POI명·대표 사진 그리드·방문한 친구·메모. 지도는 뒤에서 인터랙티브 유지.
- 동선 라인 = 같은 사용자의 visit을 **촬영시각순**으로 이은 폴리라인.
- FAB(+) 업로드 진입 / 우상단 아바타(프로필) · 검색 · 알림 진입(하단 3탭이 흡수 못 하는 진입점, §7 참조).
- 엣지: 핀 0개(첫 실행) → 빈 지도 + "첫 사진 올리기" CTA. 동선상 큰 점프(다른 도시) → 라인 끊김 처리.

#### F4. 다단계 업로드 (EXIF 자동분류 + 수동 보정) — P0 (차별 코어, MVP 난이도 핵심)
[근거: 인스타 4~5스텝 + 요기조기 분기]. **인스타는 다중사진→1위치(다대일)** 지만, **요기조기는 다중사진→다중 POI 그룹(다대다)** 으로 갈라진다 — 이 분기 화면이 핵심 차별이자 가장 두텁게 그려야 할 표면.
- 단계(2차 개선 — 별도 수동지정 단계 제거): **(1) 갤러리 다중선택**(최대 N장, 순번 배지) → **(2) 자동분류 + 미분류 in-place 배정**(POI 그룹 카드 + "추정/확정" 배지 + **"미분류" 그룹에서 사진을 *이 화면에서* 선택→장소 배정**: 카카오맵 검색 결과 택1 또는 기존 그룹 합치기, 없는 장소[할머니네집]만 직접 핀 찍기) → **(3) 그룹별 태그·메모 + 공개/비공개** → **(4) 게시**. [근거: 사용자 2차 — "미분류는 다음 단계가 아니라 현재 단계에서 선택해 장소로 넣기"]
- 사용자가 그룹의 잘못 묶인 사진 선택 → 시스템: 다른 POI로 **이동/병합/분리**.
- 사용자가 한 좌표에 후보 POI 여럿 → 시스템: **후보 선택 시트**(가장 가까운 1개 기본 선택, 변경 가능).
- 사용자가 DB에 없는 신규 장소 → 시스템: **직접 장소명 입력** 폴백.
- 엣지(반드시 별도 상태/화면): ① EXIF GPS 0 → "위치 없음" 버킷, ② 좌표 있으나 POI 모호 → 후보 picker, ③ 역지오코딩 오답 → "여기 맞나요?" 한 탭 확인, ④ 시각만/위치만 있음 → 한 신호로만 그룹핑·동선 정렬 보정, ⑤ 같은 POI 반복 방문(다른 날) → 시간 갭으로 별도 visit, 같은 place에 누적, ⑥ 수동 지정 후 저장 실패 → 낙관적 UI 주의·재시도.

#### F5. 피드 — P0 (table-stakes)
[근거: 인스타급 경험 요구]. 미디어 우선 카드. 단, 장소 중심 정체성 반영 — 카드에 **POI명·지도 미니맵/핀 컨텍스트**를 함께.
- 사용자가 피드 스크롤 → 시스템: 팔로잉의 게시물을 시간 또는 장소 묶음으로 노출(MVP 기본 시간순 [가정], 장소 묶음 보기는 P1 검토).
- 엣지: 빈 피드(팔로잉 0) → 추천 친구·인기 장소 CTA.

#### F6. 검색 — P0 (table-stakes + 정체성)
[근거: 장소 중심이므로 사람+장소+태그 3축]. 상단 세그먼트 **[장소 | 사람 | 태그]**, 기본 **장소**(앱 정체성).
- 장소 탭: **카카오맵 검색 결과**(장소명·카테고리·도로명주소·거리 — 카카오 Local API) [근거: 사용자 2차]. 사람 탭: 계정 검색→팔로우. 태그 탭: 태그로 사진 모아보기.
- 사용자가 빈 검색 진입 → 시스템: 인기/근처 장소 그리드(빈약 방지).

#### F7. 프로필 (내 지도 / 그리드) — P0 (table-stakes)
[근거: 사용자 답 ③]. 헤더(아바타·이름·소개) + 카운트 3종(게시물·팔로워·팔로잉) + **[그리드 | 지도] 토글**.
- 그리드: 인스타식 3열. 지도뷰: 내 방문 POI 핀 + 동선.
- 내 프로필=편집/공유, 공개/비공개 게시 구분 표시.

#### F8. 친구(타인) 프로필 — P0 (table-stakes)
- 팔로우 상태(팔로잉/팔로우/요청됨) 표시. 비공개 계정이면 잠금 상태(승인 전 콘텐츠 숨김).
- 사용자가 팔로우 탭 → 시스템: 공개 계정은 즉시 팔로잉, 비공개 계정은 **요청됨**(P1: 승인 흐름).

#### F9. 팔로우 / 공개·비공개 — P0
[근거: directed follow graph + 인스타 private]. 비대칭 팔로우. 게시 시 공개범위 선택.
- 엣지: 비공개 계정 게시물은 승인된 팔로워만 열람.

#### F10. 알림 (액티비티) — P0 (table-stakes)
[근거: 사용자 답 ③]. 좋아요·팔로우·(P1 댓글)을 시간순 리스트, 동종 그룹핑, 미열람 레드닷.
- **장소 맥락 추가(차별)**: "A님이 *블루보틀 성수*에 사진을 올렸어요".

#### F11. 빈/첫실행 상태 — P0 (table-stakes, 표면 빈약 방지)
[근거: 빈 상태가 첫 세션 리텐션 핵심]. 지도 홈·피드·프로필·검색·업로드 각각에 빈 상태 + 다음 행동 CTA.

## 5. 데이터 모델 (개념)

> **물리 스키마(컬럼 타입·인덱스·마이그레이션·실 DB 선택)는 범위 밖** — 개발 단계 결정. 여기선 엔터티·관계·핵심 필드만 [근거: Foursquare Places 스키마가 업계 관례].

### 5.1 엔터티와 핵심 필드
- **User** — id, handle, display_name, avatar, bio, is_private(공개/비공개 계정), created_at.
- **Place (POI)** — id, name(예: "블루보틀 성수"), category(Food&Drink·Arts 등), lat, lng, address, **external_place_id**(카카오/구글 place id 캐싱 — SDK 비종속용 추상 외부키 [근거: 사용자 답 ⑥]), parent_place_id?(몰 안 매장=subvenue [근거: Foursquare parent/subvenue]).
- **Photo** — id, owner_id(→User), file_ref, exif_lat?, exif_lng?, taken_at?(EXIF DateTimeOriginal), **location_source** ∈ {exif, estimated, manual, none}, geocode_confidence?, place_id?(→Place, **nullable** — 미분류 1급 [근거]), confirmed_by ∈ {auto, user}.
- **Post** — id, author_id(→User), place_id(→Place), caption?, tags[], **visibility** ∈ {public, followers, private}, created_at. (1번 업로드가 여러 Post로 갈라질 수 있음 — POI 그룹당 Post.)
- **PostPhoto** — post_id(→Post), photo_id(→Photo), order. (게시물-사진 묶음.)
- **Visit** — id, user_id(→User), place_id(→Place), taken_at. (동선 파생용 — user의 visit을 taken_at 정렬 → 좌표 잇기 = 폴리라인.)
- **Follow** — follower_id(→User), following_id(→User), status ∈ {pending, accepted}, created_at. (복합 PK로 중복 방지.)
- **Tag** — id, name. / **Notification** — id, recipient_id, type(like·follow·comment), actor_id, target_ref, place_ref?, is_read, created_at.
- **(P1) Like** — user_id, post_id. **(P1) Comment** — id, post_id, author_id, body, created_at.

### 5.2 관계 (cardinality)
- User **1:N** Photo / Post / Visit (소유).
- Place **1:N** Photo · Post · Visit (한 POI에 여러 사진/게시물/방문). Photo **N:1** Place(미분류면 0/nullable).
- Place **1:N** Place (parent_place_id — 몰↔매장, 자기참조).
- Post **N:M** Photo (PostPhoto 조인). Post **N:M** Tag (조인).
- User **N:M** User (Follow — directed, 비대칭. 자기 자신 제외).
- User **1:N** Notification.
- **핵심 결정:** Photo는 좌표를 그대로 들지 않고 Place에 FK로 묶는다(N:1) — "좌표 1개 = 확정 POI 1개"로 모델링하지 않고, location_source + confirmed_by + 후보 리스트로 모호성을 1급으로 다룬다 [근거: 역지오코딩 오답·다중후보].

## 6. 사용자 흐름

### 6.1 신규 진입 (콜드 스타트)
로그인/회원가입(F1) → 온보딩 가치소개(F2) → 위치 권한 프라이밍 → (선택) 팔로우 시드 → 지도 홈(빈 상태, "첫 사진 올리기" CTA).

### 6.2 업로드 (차별 코어 흐름)
FAB(+) → 사진 권한 프라이밍(최초 1회) → 갤러리 다중선택 → **자동분류 + 미분류 in-place 배정**(POI 그룹 카드, 미분류 사진을 이 화면에서 선택 → 카카오맵 검색 결과 또는 직접 핀으로 장소 배정) → 그룹별 태그·메모 + 공개/비공개 → 게시 → 홈(지도)에 핀·동선 갱신.

### 6.3 발견·소셜
검색(장소/사람/태그) → 사람 팔로우 → 지도 홈/피드에 팔로잉 핀·게시물 → 핀 탭 → 바텀시트 장소 상세 → 친구 프로필 → 팔로우.

### 6.4 회상
프로필 → [지도] 토글 → 내 방문 POI 핀 + 동선 라인 → 핀 탭 → 그 장소의 내 사진.

### 6.5 알림
알림 탭 → 좋아요/팔로우/장소 액티비티 → 해당 게시물/프로필로 딥링크.

- **지도 SDK = 카카오 가닥** [근거: 사용자 2차] — 장소 검색은 **카카오맵 검색(카카오 Local API) 결과**(장소명·카테고리·도로명주소·거리)를 사용. 핀·동선은 모노크롬, 핀만(작은 사진 마커 없음).
- **뷰포트** — **모바일 390px 기본** [근거: 인스타식 모바일 앱, 사용자 가정 ⑤]. 단일 컬럼, 하단 3탭 + 상단 앱바.
- **다크모드** — **MVP는 라이트 우선** [근거: 사용자 가정 ⑤]. 다크모드는 P1(토큰을 semantic 역할명으로 설계해 후속 다크 대응이 쉽게 — 색을 컴포넌트에 직접 박지 않음). MVP 산출물은 라이트만 그린다.
- **지도 SDK 비종속** [근거: 사용자 답 ⑥, 카카오/구글 미확정] — 핀·클러스터 버블·정보 카드·동선 폴리라인을 **일반화된 지도 표현**으로 그린다. 특정 SDK 마커 스타일에 묶지 않는다.
- **컬러** — 핀/카테고리 구분용 컬러 시스템(Food·Cafe·Culture 등 카테고리별 색 [가정: 카테고리 팔레트]). 토큰은 primitive(컬러 램프·중립 그레이 램프) → semantic(--color-primary·--color-surface·--color-text·--color-pin-* 등) 2단.
- **언어** — 한국어 UI [근거: 사용자 가정].

## 9. 비기능 요구사항

> 디자인 산출물 관점의 비기능 — 실 구현 성능 수치는 개발 단계(범위 밖). 디자인이 보장할 것:
- **접근성** — 텍스트·배경 대비 WCAG AA(자동 a11y 게이트 통과). 핀·카드 위 텍스트 가독(지도 위 오버레이 대비 주의). 터치 타깃 ≥ 44px.
- **성능(설계 함의)** — 줌아웃 시 핀 수백~수천 → **클러스터링 전제로 화면 설계**(개별 핀 나열 금지). 동선·핀은 뷰포트 단위 로드를 가정.
- **프라이버시·보안(1급)** [근거: 백래시 교훈] — 위치 공유 기본 OFF·옵트인, 게시 시 공개범위 명시, 집/반복 장소 노출 줄이는 옵션(P1 정밀도 토글), 동선 공개 여부 별도 제어. EXIF가 집주소·동선을 드러낼 수 있음을 온보딩에서 고지.
- **신뢰성** — 수동 위치 지정 저장 실패 사례 보고됨 [근거] → 낙관적 UI + 재시도/오류 노출.
- **오프라인/저신호** — 지도 로딩 실패·권한 거부 시 빈 상태·폴백 화면(설계 포함).
- **국제화** — MVP 한국어 단일. 토큰·레이아웃은 가변 텍스트 길이 견디게(P1 다국어 여지).

## 10. 범위 외 (Out of Scope)

> 조용한 누락이 아니라 **의도적 제외** — 빼는 이유 명시.
- **백엔드 실인증·실 OAuth** — Supabase 카카오/구글/이메일은 *진입 화면 UI만* 그린다. 실 토큰·세션은 개발 범위 [근거: 사용자 답].
- **실 지도 SDK 연동(카카오/구글)** — 미확정. 디자인은 SDK 비종속 표현 [근거: 사용자 답 ⑥].
- **물리 DB 스키마·실 EXIF 추출/역지오코딩 구현** — 개념 모델·플로우만. 추출 라이브러리·API 키·서버는 개발 단계.
- **상시 백그라운드 GPS 추적(Polarsteps식)** — 배터리·프라이버시 부담. 동선은 *사후*(EXIF 시각순)로 대체 [근거].
- **스토리·릴스/동영상·DM·해시태그 챌린지·쇼핑** — 인스타엔 있으나 "최소 핵심" 요구에 따라 제외 [근거: 사용자 명시].
- **게이미피케이션·배지(Swarm식)·협업/공유 지도·실시간 위치공유·AI 추천/여행책·POI 운영상태 ML 판정** — P2 후순위. 차별 아님·스코프·프라이버시 [근거].
- **다크모드** — P1(라이트 우선). MVP 산출물 제외.
- **위치 자동 추정(estimate missing)** — 오분류 리스크↓ 위해 MVP는 "수동 지정 우선" [근거: Google Photos estimate보다 보수적].

## 11. 미해결 이슈와 가정

검증된 결정과 미검증 가정을 분리해 둔다 — 본문이 추측을 사실처럼 단정하지 않도록, 근거 없는 항목은 전부 여기로 모았다.

### 11.1 근거 있는 결정 (시장조사로 확보 — 더는 가정 아님)
- POI/장소명 단위 데이터 모델(Foursquare Places 스키마) · directed follow graph · 공개/비공개 모델 · 수동분류 1급 흐름 · 권한 프라이밍 · 클러스터링·바텀시트 패턴 · 사후 동선 라인 · 검색 3축 — 전부 [근거: 동봉 시장조사 grounded].

### 11.2 가정 (미검증 — 본문에서 단정하지 않음, 검수·실측 보완 대상)
- **A1. 목표 수치(M1·M3·M4·M5·M7)** — 신규 카테고리라 외부 벤치마크 없음 [근거: confidence=guess]. 출시 후 코호트로 보정. 절대 수치 못 박지 않음.
- **A2. 자동분류 정확도 ~70~85%** — 학술 일반치(여행 분할 85%, 보조신호 91%) 기반 추정. 우리 도메인(한국 도심·카페) 실측 필요. → 검수·수정 UX를 1급으로 둬 정확도 의존을 낮춤.
- **A3. 피드 기본 정렬 = 시간순** — 장소 묶음 피드(정체성 강화)와 시간순 중 무엇이 1차인지 사용자 검수 필요. MVP는 시간순 가정.
- **A4. 다중선택 최대 장수(N)** — 인스타 20장 관례 차용 가정. 확정 필요.
- **A5. 카테고리별 핀 컬러 팔레트** — 지도앱 톤 전제이나 구체 카테고리 분류·색 매핑 미확정.
- **A6. 뷰포트 390px·라이트·한국어 단일** — 사용자 가정. 데스크톱·다크·다국어는 후속.
- **A7. 실시간 동선 vs 사후 동선** — 사용자 미확정 영역. 사후 동선을 기본 권고로 채택(프라이버시·배터리 근거). 사용자 검수 대상.
- **A8. 공개범위 기본값(공개 vs 팔로워 vs 비공개)** — 백래시 근거상 보수적(비공개/팔로워) 권고하나, "소셜 성장" 관점과 충돌 → 사용자 결정 필요.

### 11.3 열린 질문 (개발/기획 착수 전 정해야)
- Q1. 지도 SDK 최종 선택(카카오 vs 구글) — 비용·국내 POI 커버리지·약관.
- Q2. POI 데이터 출처 — SDK place id 캐싱만으로 충분한가, 자체 POI 보강 필요한가.
- Q3. 좋아요/댓글을 MVP에 넣을지(현재 P1) — "인스타 동등 경험" 요구와의 균형 사용자 재확인.
- Q4. 동선 라인 공개 제어의 세분도(전체/구간/비공개).
