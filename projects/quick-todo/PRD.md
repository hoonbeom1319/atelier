# quick-todo — PRD (Product Requirements Document)

> 작성: 시니어 PM · 2026-06-26 · 상태: /design 인계용 초안
> 모드: /forge(무인). 시작 1회 인테이크 + 시장조사 근거(TodoMVC 표준 명세, MVP 경계 휴리스틱)로 작성.
> 이 문서는 **개발에 넘길 디자인 산출물의 입력**이다. "최종 앱 코드"가 아니라 화면·동작·데이터 모델의 합의본을 정의한다.

---

## 1. 배경과 문제 정의

**배경.** quick-todo는 개인이 머릿속의 할 일을 빠르게 적고, 끝낸 것을 표시하고, 지운다 — 그뿐인 단일 사용자·로컬 전용 할 일 관리 앱이다. 동시에 이 작업장(atelier)의 **디자인/검증 하네스를 돌리기 위한 소형 테스트 대상**이다. 따라서 두 가지 압력이 동시에 작동한다: ① 사용자에게 실제로 쓸모 있는 최소 제품(MVP)일 것 ② 검증 게이트(흐름·기능·a11y·반응형·영속)를 흔들림 없이 통과할 만큼 동작이 명확할 것.

**문제.** 가벼운 메모/할 일을 적을 때 사람들은 흔히 (a) 기능이 과한 앱(마감일·우선순위·태그·반복·캘린더 연동…)에 압도되거나, (b) 그 반대로 종이/메모장처럼 "완료 여부"와 "남은 일만 보기"가 안 되는 도구를 쓴다. quick-todo가 푸는 문제는 좁고 명확하다: **추가 → 완료 토글 → 삭제 → 상태별로 걸러보기**라는 가장 기본적인 할 일 사이클을, 군더더기 없이 빠르고 정확하게 돌게 하는 것.

**왜 지금/왜 이 범위인가.** 시장조사 결과, 사용자가 인테이크에서 명시한 4개 핵심 동작(추가·완료 토글·삭제·필터)은 프레임워크 비교의 사실상 업계 표준인 **TodoMVC 명세의 핵심 행위와 1:1로 일치**한다(add / toggle / delete / filter All·Active·Completed). 즉 이 범위는 "임의로 좁힌 것"이 아니라 **표준이 정의한 최소 코어**다. 성공적 MVP는 기능 3~5개로 출시하고 '있으면 좋음'은 다음으로 미루는 것이 원칙이며(추가 기능은 "더하는 복잡도보다 10배 가치 있을 때만"), 이 앱은 하네스 테스트용 소형 앱이므로 그 원칙을 엄격히 적용한다.

## 2. 목표 및 성공 지표 (측정 가능 — 정량·기간·기준)

**제품 목표.** 단일 사용자가 추가 학습 없이 할 일을 적고/끝내고/지우고/걸러보는 사이클을 즉시 수행한다.

**지표 설정 원칙 (근거 있는 결정).** 이 앱은 로컬 전용·단일 사용자·계정/서버/분석 없음이다. 따라서 **행동형 성장 KPI(DAU·리텐션·전환율 등)는 측정 대상 자체가 없어 부적합** — 설정하면 측정 불가능한 허수 지표가 된다(시장조사 권고와 일치). 대신 **완성도/검증 게이트 통과를 정량 성공 지표로 삼는다.** 아래 지표는 모두 디자인 산출물 완성 시점(검증 단계 3·6)에 측정한다.

| # | 성공 지표 | 측정 기준 (정량) | 측정 시점/기간 |
|---|---|---|---|
| G1 | 핵심 흐름 무결성 | 4개 핵심 동작(추가→완료 토글→삭제→필터 전환)이 **죽은 컨트롤 0개**로 동작 | 검증 3·6단계 Playwright 1·2층 |
| G2 | 엣지케이스 처리 | 빈/공백 입력 차단·빈 상태 안내·긴 텍스트 처리·필터↔카운터 동기화가 **명세대로 100% 동작** | 검증 3단계 spec |
| G3 | 접근성 대비 | 라이트·다크 **양 모드 모두** 텍스트/주요 컨트롤 대비비 **WCAG AA(본문 4.5:1, 큰 글자 3:1) 통과** | a11y 자동 게이트 |
| G4 | 반응형 무파손 | **390px와 1280px 양 뷰포트**에서 레이아웃 깨짐·가로 스크롤·요소 겹침 0 | 검증 6단계 |
| G5 | 상태 영속 | 새로고침/재진입 후 **할 일 목록과 선택된 필터가 그대로 유지** | 검증 6단계 |

**비목표(이번에 추구하지 않음).** 사용자 수·체류 시간·재방문 같은 행동 지표. 성능 벤치마크 수치 목표(소형 앱·로컬이라 사실상 무관 — §9에서 합리적 기대치만 기술).

## 3. 타깃 사용자·시나리오

**주 사용자(단 하나의 페르소나).** "혼자 쓰는 사람." 자기 기기에서 개인 할 일을 관리하는 단일 사용자. 협업·공유·다중 계정 없음. 기술 친숙도는 보통 — 별도 온보딩 없이 입력창과 목록만 보고 바로 쓸 수 있어야 한다.

**사용 맥락.** 데스크톱(1280px, 업무 중 빠른 캡처)과 모바일(390px, 이동 중 적기) 모두. 정보밀도·전문적 톤(Linear/Things 류)을 선호하는 사용자상 — 촘촘하지만 차분한 목록을 빠르게 훑는다.

**대표 시나리오.**
- **S1 빠른 캡처.** 떠오른 할 일을 입력창에 적고 Enter로 즉시 추가한다. 입력창은 진입 시 포커스되어 바로 타이핑 가능.
- **S2 완료 처리.** 끝낸 일의 체크박스를 눌러 완료로 바꾼다. 완료 항목은 시각적으로 구분(취소선/흐림)되고 "남은 할 일 N개" 카운터가 즉시 줄어든다.
- **S3 정리.** 더 이상 필요 없는 항목을 삭제(즉시 삭제)한다.
- **S4 걸러 보기.** "미완료"만 눌러 아직 할 일만 본다. "완료"로 끝낸 것만 확인한다. "전체"로 돌아온다. 새로고침해도 보고 있던 필터가 유지된다.
- **S5 다크 모드 야간 사용.** 어두운 환경에서 다크 모드로 전환해 눈부심 없이 같은 작업을 한다.

## 4. 기능 요구사항 (우선순위 + 근거)

> 규율: "다 중요" 금지. MVP 경계를 직접 긋는다. 형식은 "사용자가 X → 시스템이 Y" + 엣지케이스.
> 우선순위: **MVP(P0)** = 이번에 만든다 / **다음(P1)** = 다음 버전 후보 / **나중·범위 밖** = §10.

### 4-1. MVP (P0) — 이번에 만든다

각 항목은 시장조사로 근거 확보됨 → "가정"이 아니라 **근거 있는 결정**.

**F1. 할 일 추가** *(근거: 사용자 명시 + TodoMVC 코어 `add`)*
- 사용자가 입력창에 텍스트를 적고 Enter(또는 "추가" 버튼)를 누른다 → 시스템이 새 할 일을 목록 끝에 추가하고, 입력창을 비우고, 포커스를 유지한다. "남은 할 일" 카운터 +1.
- 엣지케이스: ① 입력을 `trim()` 후 빈 문자열이면 **추가하지 않음**(공백만 저장 차단 — DB 오염 방지의 전형). ② 중복 제목은 **허용**(차단 로직 없음 — TodoMVC 관례, 단순성 유지). ③ 아주 긴 텍스트는 줄바꿈 또는 말줄임으로 레이아웃을 깨지 않게 처리(§7).

**F2. 완료 토글** *(근거: 사용자 명시 + TodoMVC 코어 `toggle`)*
- 사용자가 항목의 체크박스(또는 항목 라벨)를 누른다 → 시스템이 그 항목의 `completed`를 반전하고, 완료 시 시각적 완료 상태(취소선/흐림)를 적용하며, "남은 할 일" 카운터를 갱신한다.
- 엣지케이스: 현재 필터가 "미완료"인데 항목을 완료로 바꾸면 → **즉시 목록에서 사라진다**(필터 조건 재평가). 반대도 동일. 카운터·목록이 같은 렌더에서 동기화.

**F3. 삭제** *(근거: 사용자 명시 + TodoMVC 코어 `delete`)*
- 사용자가 항목의 삭제 컨트롤을 누른다 → 시스템이 해당 항목을 목록에서 **즉시 제거**(확인 다이얼로그 없음)하고 카운터를 갱신한다.
- 결정 근거: 삭제는 자주 일어나는 저위험 행동 → NN/g 권고상 확인창은 과한 마찰. TodoMVC 표준도 즉시 삭제. (실행취소 토스트는 §4-2 다음 버전 후보.)
- 엣지케이스: 마지막 항목을 삭제하면 빈 상태 안내가 나타난다(F6).

**F4. 상태별 필터** *(근거: 사용자 명시 + TodoMVC 코어 `filter`)*
- 사용자가 **전체 / 미완료 / 완료** 중 하나를 누른다 → 시스템이 그 조건에 맞는 항목만 보여주고, 선택된 탭을 강조한다.
- 모델링(근거 있는 결정): 필터는 **항목 데이터가 아니라 별도의 표시 상태(view state)** 다 — `'all' | 'active' | 'completed'` 단일 enum. 렌더 시 파생 계산(전체=todos / 미완료=!completed / 완료=completed).
- 엣지케이스: ① 선택된 필터는 **새로고침/재진입 후에도 유지**(§9 영속). ② 항목은 있으나 현재 필터 매칭이 0개일 때 → 필터별 맥락 빈 상태(예: "완료된 할 일이 없습니다"). ③ 토글/삭제 후 항상 현재 필터 기준으로 재렌더.

**F5. 남은 할 일 카운터** *(근거 있는 결정 — 경계 항목을 MVP로 끌어올림)*
- 시스템이 미완료(active) 항목 개수를 "남은 할 일 N개" 형태로 항상 표시한다. 항목 추가/완료 토글/삭제 시 즉시 갱신.
- MVP 포함 근거: 구현이 저렴하고(파생 값), 정보밀도·전문적 톤(Linear/Things)에 자연스러우며, 사용자가 "남은 일"을 한눈에 보는 핵심 정보다. 한국어는 단복수 변화가 없어 영문 명세의 복수형 복잡성이 제거됨.

**F6. 빈 상태(empty state)** *(근거: NN/g·UX 권고 — "막다른 길 금지")*
- 항목이 0개일 때 → 시스템이 무미건조한 'no data'가 아니라 **헤드라인 + 짧은 안내 + 입력 유도**를 보여준다(입력창은 항상 노출). 정보밀도 톤이므로 절제된 안내.
- 두 종류 구분: (a) 전체가 0개 — "첫 할 일을 추가해 보세요" 류. (b) 항목은 있으나 현재 필터 매칭 0개 — 필터별 맥락 메시지. 라이트/다크 모두에서 빈 상태 텍스트 대비 확보.

### 4-2. 다음 (P1) — 다음 버전 후보 (이번 MVP 제외)

근거: TodoMVC 표준에는 포함되나 핵심 흐름 없이도 앱이 성립 → 소형 MVP에선 제외.
- **인라인 편집** (라벨 더블클릭 → 수정, Enter/blur 저장, Esc 취소; 편집 결과가 빈 문자열이면 해당 항목 삭제). CRUD의 U지만 소형 스펙에선 선택.
- **전체 완료 토글(toggle-all)** — 모든 항목 일괄 완료/해제.
- **완료 모두 지우기(clear completed)** — 완료 항목 일괄 삭제(완료 0개일 때 컨트롤 숨김).
- **삭제 실행취소 토스트(undo)** — 즉시 삭제의 안전망. 확인창보다 모던하나 추가 상태관리 비용 → 다음 버전.

### 4-3. 나중·범위 밖

§10 참조. 마감일·우선순위·정렬·검색·태그·서브태스크·반복·알림·캘린더·드래그 정렬·통계 등 — 전형적 기능 과잉 후보. 사용자가 인테이크에서 "부풀리지 말 것"으로 명시했고 시장조사 권고와 일치.

## 5. 데이터 모델 (개념 모델 — 물리 스키마는 범위 밖)

> **물리 스키마(테이블/컬럼 타입/인덱스/저장 포맷의 최종 확정)는 범위 밖**이다 — 개발 단계 결정. 여기서는 개념 엔터티와 관계만 정의한다.

### 엔터티

**Todo (할 일 항목)** — 시장조사 표준(TodoMVC) 최소 3필드 채택(근거 있는 결정).

| 필드 | 개념 타입 | 의미 | 비고 |
|---|---|---|---|
| `id` | 고유 식별자 | 항목 고유 키 | 충돌 없는 값(UUID 또는 timestamp). 물리 표현은 범위 밖 |
| `title` | 텍스트(유니코드) | 할 일 내용 | 한국어 등 자유 텍스트. `trim` 후 비어 있으면 항목 미생성 |
| `completed` | 불리언 | 완료 여부 | 기본 `false`. F2가 토글 |

- `createdAt`/`order` 같은 메타 필드는 **MVP에서 불필요**(정렬을 도입하지 않으므로 삽입 순서 = 배열/목록 순서로 충분). 정렬 기능을 나중에 넣을 때만 추가 검토.

**ViewState (표시 상태)** — 항목이 아니라 앱의 화면 상태.

| 필드 | 개념 타입 | 의미 |
|---|---|---|
| `filter` | enum `all`/`active`/`completed` | 현재 선택된 필터 |
| `theme` | enum `light`/`dark` | 현재 색 모드 |

### 관계
- **사용자 1 : N Todo** — 단일 사용자가 0개 이상의 할 일을 가진다. 단, 이 앱은 단일 사용자·로컬 전용이라 "사용자" 엔터티는 **암묵적(구현하지 않음)** — 모든 Todo는 그 한 명에게 귀속된다.
- **Todo ↔ filter는 관계가 아니다.** 필터는 Todo의 `completed` 값에 대한 **파생 뷰**일 뿐 항목에 외래키로 연결되지 않는다(시장조사 근거: 필터는 데이터 필드가 아닌 뷰 상태).
- N:M·1:1 관계 없음. 태그/카테고리 같은 다대다 구조는 §10 범위 밖.

### 영속(개념 수준)
- 할 일 목록과 ViewState는 **로컬에 영속**되어 새로고침/재진입 후 복원된다(개념 요구 — 저장 위치/포맷의 물리 확정은 개발 단계). 시장조사 관례상 로컬 키-값 저장(예: localStorage에 JSON 배열)이 자연스러우나, **이는 권고일 뿐 강제 아님** — atelier는 특정 구현에 묶이지 않는다.

## 6. 사용자 흐름

> 단일 화면(목록 + 입력 + 필터)에서 모든 흐름이 일어난다. 라우팅 없이 상태로 관리.

**메인 흐름 (S1~S4 통합).**
```
[진입] 입력창 자동 포커스
  → 텍스트 입력 + Enter ──(trim 후 비었으면 무시)──> [목록에 추가, 카운터 +1, 입력창 비움]
  → 항목 체크박스 클릭 ──> [완료 토글, 완료 스타일, 카운터 갱신, 현재 필터 기준 재렌더]
  → 항목 삭제 클릭 ──> [즉시 제거, 카운터 갱신] ──(목록 0개면)──> [빈 상태 안내]
  → 필터 탭(전체/미완료/완료) 클릭 ──> [해당 조건만 표시, 탭 강조] ──(매칭 0개면)──> [필터별 빈 상태]
[새로고침/재진입] ──> 저장된 목록 + 마지막 필터 + 마지막 테마 복원
```

**테마 흐름 (S5).**
```
[테마 토글 클릭] → 라이트 ↔ 다크 전환 → 선택 영속 → 재진입 시 복원
```

**상태 전이 요약.**
- Todo.completed: `false ⇄ true` (F2 토글만이 변경).
- ViewState.filter: `all → active → completed → all …` (탭 클릭으로 임의 전환).
- 목록은 항상 `(전체 todos) ∩ (현재 filter 조건)` 으로 렌더되고, 카운터는 항상 `count(completed === false)`.

## 7. 화면 목록·요구사항

> 단일 화면 앱. 아래는 그 한 화면의 영역(region)별 요구다.

**SC1. 메인 화면 (유일 화면)** — 다음 영역으로 구성:

1. **헤더 영역** — 앱 제목, 테마 토글(라이트/다크). 정보밀도 톤에 맞게 컴팩트.
2. **입력 영역** — 항상 노출되는 단일 라인 입력창 + "추가" 액션. 진입 시 자동 포커스. placeholder는 한국어("할 일을 입력하세요" 류).
3. **필터 바** — 전체 / 미완료 / 완료 3개 탭(세그먼트). 선택 탭 강조(`selected`/`aria-current`). "남은 할 일 N개" 카운터 동거.
4. **할 일 목록 영역** — 각 행: [체크박스] [할 일 텍스트] [삭제 컨트롤]. 완료 항목은 취소선/흐림. 촘촘한 리스트(데이터밀도). 긴 텍스트는 줄바꿈 또는 말줄임 — **가로 스크롤/레이아웃 깨짐 금지**(G4).
5. **빈 상태 영역** — 목록이 비었거나 필터 매칭 0개일 때 4를 대체하는 안내(F6). 입력 영역(2)은 빈 상태에서도 유지.

**화면 요구(공통).**
- 모든 인터랙티브 요소는 와이어드(죽은 컨트롤 0 — G1). 범위 밖 요소를 굳이 두지 않는다.
- 키보드 접근: 입력 Enter 추가, 체크박스/삭제/필터 탭/테마 토글 모두 포커스·키 조작 가능.
- 셀렉터 안정성을 위해 role·aria·텍스트 기반으로 식별 가능하게(검증 spec이 포터블 셀렉터로 잡음).
- 반응형: 390px에서는 세로로 촘촘히, 1280px에서는 중앙 정렬된 적정 폭 컨테이너(과도하게 넓히지 않음). 양쪽 모두 무파손(G4).

## 8. 디자인 방향 (톤·레퍼런스·뷰포트·다크 모드)

- **톤.** 전문적·정보밀도. 데이터 중심·촘촘한 레이아웃. 장식 최소, 명료한 위계.
- **레퍼런스.** Linear / Things 류의 차분한 모던 UI — 절제된 색, 또렷한 타이포, 부드러운 상태 전이. 화려한 일러스트/그라데이션 지양.
- **뷰포트(반응형).** **모바일 390px ~ 데스크톱 1280px 둘 다** 1급 타깃. 모바일이 기본이 아니라 둘 다 동등하게 검증(G4). /design 인테이크(00-flow)에서 대상 뷰포트로 이 둘을 명시.
- **색 모드(다크 모드).** **라이트 + 다크 둘 다 필수.** 테마는 의미 토큰(semantic token)으로 분기 — 두 모드 모두 WCAG AA 대비 충족(G3). 토큰은 표준 2단(primitive → semantic 역할명), 브랜드 단어로 토큰명 짓지 않음.
- **언어.** 한국어 UI. 모든 라벨·빈 상태·카운터·placeholder 한국어. 카운터는 "남은 할 일 N개"(한국어 단복수 불변 활용 — 복수형 분기 불필요).
- **빈 상태.** 정보밀도 톤이므로 절제된 안내(과한 일러스트 금지). 라이트/다크 양쪽 대비 확보.

## 9. 비기능 요구사항

- **접근성(a11y).** 라이트·다크 양 모드 텍스트/주요 컨트롤 대비 WCAG **AA**(본문 4.5:1, 큰 글자/UI 3:1). 키보드 전체 조작 가능. 폼 요소·체크박스에 적절한 라벨/aria.
- **반응형.** 390px·1280px 양 뷰포트 무파손(가로 스크롤·겹침·잘림 0). 그 사이 폭에서도 합리적 흐름.
- **상태 영속.** 할 일 목록·선택 필터·테마가 로컬에 저장되어 새로고침/재진입 후 복원(G5). 저장 실패/손상 시 빈 목록으로 안전하게 폴백(앱이 죽지 않음).
- **성능(합리적 기대).** 소형·로컬 앱 — 수치 벤치마크 목표는 두지 않는다. 다만 일상적 항목 수(수십~수백 개)에서 추가/토글/삭제/필터 전환이 체감 지연 없이 즉시 반영되어야 한다.
- **독립 실행.** atelier 규칙상 산출물은 외부 의존성(외부 라이브러리) 없는 독립 실행 HTML — 상태 변화용 인라인 JS 허용.
- **에러/방어.** 빈/공백 입력 차단(F1). 긴 텍스트로 레이아웃이 깨지지 않음(F1/SC1-4).

## 10. 범위 외 (Out of Scope)

이번 MVP가 **명시적으로 하지 않는 것** — 착수자가 추측하지 않도록 못박는다.

- **계정·인증·동기화·협업·공유** — 단일 사용자·로컬 전용(사용자가 닫음).
- **외부 연동** — 알림·캘린더·결제·푸시 등 일체 없음.
- **기능 과잉 후보(전형)** — 마감일, 우선순위, 정렬, 검색, 태그/라벨, 서브태스크, 반복 일정, 드래그 재정렬, 통계/대시보드. (시장조사: 모두 "더하는 복잡도보다 10배 가치" 기준 미달, 사용자가 부풀리지 말 것으로 지시.)
- **P1 다음 버전 후보(이번 제외)** — 인라인 편집, 전체 완료 토글, 완료 모두 지우기, 삭제 실행취소 토스트(§4-2). 가치는 인정하나 핵심 흐름엔 불필요.
- **물리 데이터 스키마/저장 포맷 확정** — 개발 단계 결정(§5).
- **다국어** — 한국어 단일.
- **행동형 분석/KPI 수집** — 측정 인프라 없음(§2).

## 11. 미해결 이슈와 가정

> 시장조사로 근거 확보된 항목은 §1·§4·§5에서 **근거 있는 결정**으로 본문에 반영했다. 아래는 (a) 사용자가 답하지 않아 가정으로 닫은 것, (b) 여전히 추측이라 본문에서 단정하지 않은 것이다.

**A. 사용자 미답 → 가정으로 닫음 (확인되면 본문 승격).**
- **A1.** 단일 사용자·로컬 전용이며 로그인/계정/동기화/협업/공유는 전부 범위 밖이다. *(인테이크에서 사용자가 가정으로 닫음.)*
- **A2.** 외부 연동(알림·캘린더·결제)은 없다. *(동상.)*
- **A3.** 하네스 테스트용 소형 앱이므로 MVP는 작게 유지하고, '있으면 좋음'(마감일·우선순위·정렬·검색·태그·서브태스크 등)은 다음/나중/범위 밖. *(동상.)*

**B. 본문에 단정하지 않은 추측 (검증/결정 필요).**
- **B1. (추측) 성공 지표.** §2의 행동형 KPI 부적합 판단은 시장조사상 confidence "guess" — 다만 이 앱이 로컬·단일사용자라는 사실에서 직접 따라오므로 검증 게이트 지표로 대체했다. 사용자가 행동 지표를 원하면 재논의.
- **B2. (결정 보류) 삭제 안전망.** 즉시 삭제로 확정(F3)하되, 실수 방지 필요가 드러나면 확인창이 아니라 **실행취소 토스트**를 P1으로 도입(§4-2). 어느 쪽도 MVP엔 넣지 않음.
- **B3. (결정 보류) 인라인 편집.** TodoMVC 표준엔 있으나 소형 스펙상 P1로 미룸. 사용자가 "오타 수정이 핵심"이라 보면 MVP로 끌어올릴 수 있음(현재는 보류).
- **B4. (구현 자유) 영속 메커니즘.** §5의 로컬 키-값(localStorage JSON) 저장은 관례 기반 권고일 뿐 강제 아님 — 대상 프로젝트가 자기 컨벤션으로 결정. atelier는 구현에 묶이지 않는다.
- **B5. (디자인 단계 결정) 긴 텍스트 처리.** 줄바꿈 vs 말줄임 중 무엇을 쓸지는 정보밀도 톤·뷰포트에 맞춰 /design에서 확정(요구는 "레이아웃 무파손").
