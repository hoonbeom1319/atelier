# collab-kanban — PRD (Product Requirements Document)

> 작성: 시니어 PM · 2026-06-26 · 상태: /design 인계용 초안
> 모드: /forge(무인). 시작 1회 인테이크 + 시장조사 근거(Trello·Linear·Jira·Notion·Asana 경쟁 분석, 칸반 데이터 모델 관례, 실사용 통점 조사)로 작성.
> 이 문서는 **개발에 넘길 디자인 산출물의 입력**이다. "최종 앱 코드"가 아니라 화면·동작·데이터 모델의 합의본을 정의한다.

---

## 1. 배경과 문제 정의

**배경.** collab-kanban은 소규모 팀(스타트업/사이드 프로젝트, 약 3~10명)이 **하나의 칸반 보드** 위에서 카드를 컬럼 간 드래그&드롭으로 옮기며 일을 관리하는 **데스크톱 우선(1280px+) 협업 도구**다. 비주얼 톤은 Linear류 미니멀·차분(절제된 그레이 램프 + 포인트 컬러 1개, 다크모드 옵션). 이 작업장(atelier)의 산출물은 "디자인 산출물"이므로, 실시간 동기화·계정/인증 같은 백엔드는 범위 밖이고 **멀티유저는 개념적으로만**(아바타·담당자·활동 로그로) 표현한다.

**문제.** 소규모 팀이 칸반 도구를 쓸 때의 실사용 불만은 **기능의 가짓수가 아니라 "협업 디테일의 부재"** 에 몰려 있다(시장조사 근거). 구체적으로 ① 카드에 정보가 과밀해 보드가 시각적으로 무너진다(WIP 미관리, 속성 과노출), ② 댓글이 plain-text라 @멘션·맥락 공유가 안 된다, ③ 모든 변경에 알림이 와서 알림 피로가 생긴다, ④ "누가 무엇을 하는지"가 한눈에 안 보인다. collab-kanban이 푸는 문제는 좁고 명확하다: **단일 보드에서 카드를 옮기고(드래그&드롭) · 사람에게 배정하고(담당자/필터) · 카드 안에서 대화·이력을 남기고(댓글/활동) · 라벨·마감으로 거르는** 핵심 협업 사이클을, 군더더기 없이 차분하게 돌게 하는 것.

**왜 이 범위인가 (근거 있는 결정).** 사용자가 인테이크에서 정한 4개 협업 기능(드래그&드롭 이동 / 담당자·아바타·멤버필터 / 댓글·활동로그 / 라벨·마감일·필터)은 시장조사 결과 **경쟁 도구(Trello·Linear·Jira·Asana)의 '카드 칸반' 핵심과 1:1로 일치**하며, 단일 보드 MVP 경계로 타당함이 확인됐다(명시적 충돌 없음). 포지션은 **'Trello의 진입 단순함 + Linear의 절제된 비주얼·속도감'의 교집합** — 소규모 팀이 온보딩 없이 당일 사용 가능한 차분한 보드. 따라서 디자인 공은 (1) 카드 표면 정보밀도 절제, (2) 드래그&드롭의 체감 품질, (3) 멤버 협업 가시성, 이 셋에 들어간다.

## 2. 목표 및 성공 지표 (측정 가능 — 정량·기간·기준)

**제품 목표.** 소규모 팀이 추가 학습 없이 카드를 옮기고·배정하고·대화하고·거르는 협업 사이클을, 차분한 단일 보드에서 즉시 수행한다.

**지표 설정 원칙 (근거 있는 결정).** 이 산출물은 디자인 단계이며 실시간 백엔드·계정·분석 인프라가 범위 밖이다. 따라서 **행동형 성장 KPI(DAU·리텐션·전환율 등)는 측정 대상 자체가 없어 부적합** — 설정하면 측정 불가능한 허수 지표가 된다(시장조사 권고와 일치, confidence "guess"). 대신 **완성도/검증 게이트 통과를 정량 성공 지표로 삼는다.** 아래 지표는 디자인 산출물 완성 시점(검증 단계 3·6, a11y 게이트)에 측정한다.

| # | 성공 지표 | 측정 기준 (정량) | 측정 시점/기간 |
|---|---|---|---|
| G1 | 핵심 협업 흐름 무결성 | 4개 핵심 동작(드래그&드롭 이동 / 담당자 배정 / 댓글 작성 / 라벨·필터)이 **죽은 컨트롤 0개**로 동작 | 검증 3·6단계 Playwright 1·2층 |
| G2 | 드래그&드롭 상태 완결성 | 드래그 중·드롭존 하이라이트·빈 컬럼 드롭·드롭 위치 삽입의 **모든 상태(state)가 명세대로 렌더**(정적 카드만 그리는 stateInert 0) | 검증 3·6단계 spec |
| G3 | 엣지/빈 상태 처리 | 7개 필수 상태(미배정·overdue·빈 컬럼·필터 0건·읽기전용·아바타 +N·카드 과밀 말줄임)가 **명시적 variant로 100% 존재** | 검증 3단계 spec |
| G4 | 접근성 대비 | 라이트·다크 **양 모드 모두** 텍스트/주요 컨트롤·라벨칩 대비비 **WCAG AA(본문 4.5:1, 큰 글자/UI 3:1) 통과** (촘촘한 그레이 램프의 대비 위험 검증 필수) | a11y 자동 게이트 |
| G5 | 데스크톱 레이아웃 무파손 | **1280px(주) 및 1440px**에서 4~6컬럼 보드가 가로 스크롤 없이 표시되고, 컬럼 초과 시 가로 스크롤·요소 겹침/잘림 0. 카드 상세 패널/모달 무파손 | 검증 6단계 |
| G6 | 멀티유저 가시성(개념) | 아바타 스택·담당자 필터·활동 로그가 "누가 무엇을"을 **한 화면에서 표현**(개념적 멀티유저 충족) | 검증 6단계 |

**비목표(이번에 추구하지 않음).** 사용자 수·체류 시간·재방문 같은 행동 지표. 실시간 동기화 지연·서버 성능 같은 백엔드 수치 목표(범위 밖). 보드당 카드 수 등 정량 운영 벤치마크(공개 근거 부재 — §11 가정).

## 3. 타깃 사용자·시나리오

**주 사용자(페르소나).** "소규모 팀의 실무자." 스타트업/사이드 프로젝트의 3~10명 팀에서 자기 일과 동료 일을 같은 보드에서 본다. 기술 친숙도는 보통 — 별도 온보딩 없이 보드를 보고 바로 카드를 옮기고 배정할 수 있어야 한다(소규모 팀 도구의 핵심 성공 기준: "신규 멤버가 교육 없이 당일 사용 가능").

**역할(개념적 멀티유저).** 실제 인증은 범위 밖이나 디자인은 두 역할을 시각적으로 구분한다(시장조사 근거):
- **Editor/Owner** — 카드 이동·생성·편집·배정·댓글 가능(기본 상태).
- **Viewer(읽기 전용)** — 보드를 보지만 카드 드래그 불가. "읽기 전용" 상태를 디자인에 반영(드래그 핸들 비활성, 안내).

**사용 맥락.** 데스크톱(1280px+) 업무 환경. 데일리 스탠드업·스프린트 계획에서 "내 카드만"·"특정 담당자 카드만" 보기가 대표 용도. 다크모드는 야간/장시간 작업 시 눈부심 완화 옵션.

**대표 시나리오.**
- **S1 카드 이동.** 진행된 일의 카드를 "In Progress" 컬럼으로 드래그한다. 드래그 중 카드가 들리고(lift/shadow), 유효 컬럼이 하이라이트되며, 놓은 위치에 정확히 삽입된다.
- **S2 담당자 배정 + 필터.** 카드에 담당자를 배정해 아바타가 카드 표면에 뜬다. 상단 아바타 스택에서 특정 멤버를 토글해 그 사람 카드만 본다("내 카드만" 빠른 필터 포함).
- **S3 카드 안 협업.** 카드를 클릭해 상세 패널을 연다. 설명·담당자·라벨·마감일을 보고, 하단 타임라인에 @멘션 댓글을 남기며, 변경 활동 로그(이동·배정·라벨 변경)를 확인한다.
- **S4 라벨·마감·필터.** 카드에 라벨과 마감일을 단다. 마감 지난 카드(overdue)는 경고 색으로 강조된다. 라벨·담당자·마감(오늘/이번주/지남)으로 보드를 거른다. 필터 결과 0건이면 빈 상태가 뜬다.
- **S5 카드 생성.** 빈 컬럼 또는 컬럼 하단의 "+ 추가"로 새 카드를 만든다.
- **S6 다크 모드.** 어두운 환경에서 다크 모드로 전환해 같은 작업을 차분하게 이어간다.
- **S7 읽기 전용.** Viewer는 같은 보드를 보지만 카드를 드래그할 수 없고(핸들 비활성), 그 상태가 시각적으로 안내된다.

## 4. 기능 요구사항 (우선순위 + 근거)

> 규율: "다 중요" 금지. MVP 경계를 직접 긋는다. 형식은 "사용자가 X → 시스템이 Y" + 엣지케이스.
> 우선순위: **MVP(P0)** = 이번에 만든다 / **다음(P1)** = 다음 버전 후보 / **나중·범위 밖** = §10.
> 각 P0 항목은 시장조사로 근거 확보됨 → "가정"이 아니라 **근거 있는 결정**.

### 4-1. MVP (P0) — 이번에 만든다

**F1. 카드 드래그&드롭 컬럼 간 이동** *(근거: 사용자 명시 + 경쟁 도구 카드 칸반 핵심; 실사용 체감 품질 1순위)*
- 사용자가 카드를 잡아 다른 컬럼(또는 같은 컬럼 내 다른 위치)으로 드래그&드롭한다 → 시스템이 카드를 **놓은 위치에 정확히 삽입**하고, 출발/도착 컬럼의 카드 수 카운터를 갱신한다.
- 필수 상태(state) — 정적 카드만 그리면 stateInert(G2):
  - ① **드래그 가능 신호**: hover 시 grab 커서 + lift/shadow(또는 drag handle).
  - ② **드래그 중**: 카드가 들린 시각(반투명/그림자), 원위치에 플레이스홀더/삽입선.
  - ③ **드롭존 하이라이트**: 유효 컬럼의 border/배경 하이라이트, 삽입 지점 미리보기 라인.
  - ④ **빈 컬럼 드롭**: 카드 없는 컬럼의 드롭존/"여기에 놓기" 힌트.
- 엣지케이스: ① **빈 컬럼**에 드롭(빈 상태 + 드롭존). ② **드롭 위치** = 항상 맨 아래가 아니라 마우스로 놓은 자리에 삽입(현대적 기대치). 키보드 이동 시 상단 기본. ③ **Viewer(읽기 전용)**는 드래그 불가 — 핸들 비활성·커서 변화 없음(F8과 연동). ④ 카드 정렬은 컬럼 내 수동 순서를 보존(데이터 모델 §5의 position 필드 — 드래그 위치 유지).

**F2. 담당자 배정 · 아바타 표시** *(근거: 사용자 명시 + 멤버 협업 가시성이 소규모 팀 차별 포인트)*
- 사용자가 카드(또는 카드 상세 패널)에서 멤버를 담당자로 배정한다 → 시스템이 **카드 표면에 담당자 아바타**를 표시하고, 미배정 시 placeholder를 보인다.
- 모델링(근거 있는 결정): MVP는 **카드당 담당자 1명(+미배정)** 으로 단순화하되, 데이터 모델은 다중 담당자 확장을 막지 않는다(경쟁 도구는 다중 허용 — §5). 아바타는 이름/이니셜 폴백.
- 엣지케이스: ① **미배정 카드** — 담당자 자리 placeholder("미배정"). ② **아바타 다수 겹침** — "+N" 처리(다중 담당자 확장 대비). ③ 배정 변경은 활동 로그에 기록(F4).

**F3. 멤버별 필터 + 라벨·마감 필터** *(근거: 사용자 명시 + Linear 실사용 — 스탠드업/스프린트에서 담당자별 보기)*
- 사용자가 상단 필터(아바타 스택 토글 / 라벨 / 마감)에서 조건을 선택한다 → 시스템이 **그 조건에 맞는 카드만** 컬럼별로 보여주고, 활성 필터를 강조한다.
- MVP 필터 셋(근거 있는 결정): ① **담당자별**(아바타 토글·멀티선택, "내 카드만" 빠른 필터 포함) ② **라벨별** ③ **마감일**(오늘/이번주/지남) ④ (선택)미배정만.
- 모델링: 필터는 카드 데이터가 아니라 **표시 상태(view state)** 다 — 카드의 assignee/label/dueDate 필드에 술어를 거는 파생 뷰(데이터 모델 추가 없음).
- 엣지케이스: ① **필터 결과 0건** — 카드가 모두 사라진 컬럼/보드의 빈 상태(맥락 메시지). ② 필터 적용 중에도 컬럼 헤더 카운트는 "필터된 개수"를 반영. ③ 필터 해제로 전체 복귀.

**F4. 카드 상세 패널 — 댓글 스레드 + 변경 활동 로그** *(근거: 사용자 명시 + 모든 경쟁 도구 관례 — 댓글/활동은 카드 표면이 아니라 상세 패널 하단 타임라인에 산다)*
- 사용자가 카드를 클릭한다 → 시스템이 **카드 상세 패널/모달**을 열어 제목·설명·담당자·라벨·마감일을 보이고, **하단에 시간순 타임라인**(댓글 + 활동 로그)을 표시한다.
- **댓글**: 작성자 아바타 + 이름 + 상대시간 + 본문. **@멘션 칩**을 전제로 디자인(plain-text 한계가 핵심 통점). 멘션된 사람만 알림(알림 피로 방지 — 개념적 표현).
- **활동 로그**: 시스템이 자동 기록하는 변경 이벤트(이동·담당자 변경·라벨 변경·마감 변경·생성)를 actor·timestamp와 함께 **사람이 읽는 한 줄 문장**("홍길동 님이 In Progress로 이동함")으로.
- 배치(근거 있는 결정): 소규모 팀·미니멀 지향이므로 **통합 타임라인 + '댓글/활동' 필터 토글**(또는 탭 분리)로 정보밀도를 낮춘다. 두 패턴 모두 관례 — 디자인 검수에서 확정(§11).
- **멘션 모델링(근거 있는 결정).** @멘션은 별도 관계 테이블 없이 **Comment.body 안의 멘션 텍스트를 파싱해 Member로 해소**한다(§5 노트). 따라서 "멘션된 사람만 알림"은 body 파싱으로 도출되는 파생 동작이며, 알림 자체는 개념적 표현(실제 전달 UI는 P1 — §4-2 알림 센터).
- 엣지케이스: ① **긴 댓글 스레드** — 접기/더보기. ② **카드 표면 댓글 표시는 "댓글 수(total)"** 로 단순화한다(미니멀 톤 부합) — 카드 표면엔 전체 댓글 개수만 작은 아이콘+숫자로. "미해결(unresolved) 댓글 구분"은 Comment에 해결 상태 필드가 없으므로 **MVP 제외, P1로 미룸**(§4-2). ③ 사람 댓글과 시스템 활동을 톤/아이콘으로 시각 구분.

**F5. 카드 생성/편집 진입점** *(근거: 시장조사 권고 — 빠뜨리기 쉬운 필수 진입점)*
- 사용자가 **빈 컬럼의 "+ 추가" 또는 컬럼 하단 "+ 카드 추가"** 를 누른다 → 시스템이 인라인 입력 또는 상세 패널로 새 카드를 만들고, 그 컬럼 끝(또는 지정 위치)에 추가한다.
- 편집: 카드 상세 패널에서 제목·설명·담당자·라벨·마감일을 수정한다.
- 엣지케이스: ① **빈 제목** trim 후 비면 생성 안 함. ② 긴 제목은 카드 표면에서 2줄 말줄임(§7).

**F6. 라벨 · 마감일 표시** *(근거: 사용자 명시 + 카드 표면 정보밀도 절제 원칙)*
- 사용자가 카드에 라벨(색+이름)과 마감일을 단다 → 시스템이 **카드 표면에 라벨 점/칩과 마감일**을 절제해 표시한다.
- 표면 노출 규칙(근거 있는 결정): 카드 표면엔 **제목 · 담당자 아바타 · 라벨 1~2개(점/칩) · 마감일 · 댓글 수(total)**까지만. 설명·전체 댓글·활동은 상세 패널(F4). 속성 과밀 시 생략/말줄임(Linear도 공간 부족 시 일부 미표시).
- 엣지케이스: ① **마감 지남(overdue)** — 경고 색(붉은 톤) 강조, 임박(D-카운트다운)도 관례. ② 라벨은 채도 낮은 색칩 + 텍스트(미니멀 톤과 충돌 회피). ③ 라벨 색은 **제품 데이터**(라벨 엔티티의 색)이므로 파운데이션 토큰이 아니라 별도 라벨 색 팔레트로(§8).

**F7. 컬럼 헤더 — 제목 + 카드 수 카운터** *(근거: 경쟁 도구 공통 관례; 정보 가시성)*
- 시스템이 각 컬럼 헤더에 컬럼명과 **현재 카드 수**를 표시한다(필터 적용 시 필터된 개수). 컬럼 내 카드가 많으면 세로 스크롤.
- 엣지케이스: ① **카드 과밀** 컬럼 — 세로 스크롤 + 헤더 카운터. ② 컬럼이 화면폭 초과 시 보드 가로 스크롤(G5).

**F8. 읽기 전용(Viewer) 상태** *(근거: 시장조사 권고 — 개념적 멀티유저라도 권한 상태 반영)*
- Viewer 역할일 때 → 시스템이 카드 드래그를 막고(핸들 비활성, grab 커서 없음), 생성/편집/배정 컨트롤을 숨기거나 비활성화하며, "읽기 전용" 상태를 안내한다. 댓글 가능 여부는 디자인 검수에서 확정(§11).
- MVP 포함 근거: 멀티유저를 개념적으로만 표현하더라도 "읽기 전용"은 디자인 상태로 반드시 그려야 협업 가시성이 완결된다(시장조사 권고). 단일 변형(variant)으로 표현 — 실제 권한 로직은 범위 밖.

**F9. 테마(라이트/다크) 토글** *(근거: 사용자 명시 "다크모드 옵션"(A1) + Linear류 톤의 핵심 산출. G4·S6·SC1·§8을 닫는 기능 앵커)*
- 사용자가 **SC1 헤더의 테마 토글**을 누른다 → 시스템이 라이트 ↔ 다크 테마를 **의미 토큰(semantic token) 분기**로 전환한다(라이트의 단순 반전이 아니라 별도 페어 스케일 — §8). 같은 작업을 끊김 없이 이어간다.
- 책임 명세(근거 있는 결정): 다크는 **별도 페어 스케일 + 4단 elevation**(§8)으로 산출하고, **양 모드 모두 WCAG AA 대비(G4)** 를 충족해야 한다 — 테마 토글이 '명세된 기능'이므로 양모드 a11y·토글 와이어링의 책임이 디자인 재량이 아닌 요구로 닫힌다.
- 엣지케이스: ① **시스템 선호(prefers-color-scheme) 따름 여부**·② **세션/새로고침 후 선택 유지 여부**는 영속이 개념 수준이라 디자인 검수에서 확정(§11 C4). MVP는 토글 동작·양모드 렌더·양모드 a11y만 보증. ③ 라벨 색(제품 데이터)은 양 모드에서 대비 충족하도록 검수(§8·G4).

### 4-2. 다음 (P1) — 다음 버전 후보 (이번 MVP 제외)

근거: 경쟁 도구에 흔하나 단일 보드 핵심 협업 흐름 없이도 앱이 성립 → MVP에서 제외(미니멀 톤과도 부합).

- **멀티 보드 / 워크스페이스 전환** — 사용자가 명시적으로 "다음"으로 미룸. 단, 데이터 모델에 boardId를 남겨 확장 대비(§5).
- **WIP 한도** — 칸반 정석이나 사용자 MVP 범위에 없음. 컬럼 헤더 경고 색으로 다음 도입 가능.
- **다중 담당자** — MVP는 1명, 모델은 다중 확장 대비(§5).
- **삭제 실행취소 토스트 / 카드 아카이브(Done 정리)** — 안전망·정리 흐름.
- **카드 우선순위(priority) 마커** — Linear엔 있으나 정보밀도 절제 위해 제외.
- **알림 센터** — @멘션 알림의 실제 전달(개념적 표현은 MVP, 전달 UI는 다음).
- **댓글 해결(resolve) 상태 / 미해결 댓글 구분** — Comment에 `resolved` 불리언(또는 thread/parentId)을 더해 카드 표면에 "미해결 댓글 수"를 구분 노출. MVP는 "댓글 수(total)"만(F4) — 모델 단순화·미니멀 톤 부합.

### 4-3. 나중·범위 밖

§10 참조. 체크리스트/서브태스크, 스윔레인, 추정치(estimate), 타임라인/간트, AI·자동화, 첨부 파일 관리 등 — 정보밀도·미니멀 포지션과 충돌하거나 단일 보드 MVP 경계 밖. 실시간 동기화·계정/인증 백엔드는 디자인 산출물 범위 밖(개념적 멀티유저로만 표현).

## 5. 데이터 모델 (개념 모델 — 물리 스키마는 범위 밖)

> **물리 스키마(테이블/컬럼 타입/인덱스/저장 포맷의 최종 확정)는 범위 밖**이다 — 개발 단계 결정. 여기서는 개념 엔터티와 관계만 정의한다. 모델은 Trello·Linear의 공개 데이터 모델 관례를 근거로 한다.

### 엔터티

**Board (보드)** — MVP는 1개.

| 필드 | 개념 타입 | 의미 | 비고 |
|---|---|---|---|
| `id` | 식별자 | 보드 키 | MVP 단일 보드라도 카드/컬럼에 boardId를 두면 멀티보드 확장이 안 깨짐(다음 대비) |
| `name` | 텍스트 | 보드 이름 | |

**Column (컬럼 / status)** — Board 1:N, **순서 있음**.

| 필드 | 개념 타입 | 의미 |
|---|---|---|
| `id` | 식별자 | 컬럼 키 |
| `name` | 텍스트 | 컬럼명(예: Backlog/To Do/In Progress/Review/Done) |
| `order` | 정렬 키 | 보드 내 컬럼 순서 |

**Card (카드)** — Column 1:N, **컬럼 내 순서 있음**. 컬럼=status가 1차 그룹핑 축.

| 필드 | 개념 타입 | 의미 | 비고 |
|---|---|---|---|
| `id` | 식별자 | 카드 키 | |
| `title` | 텍스트 | 카드 제목 | trim 후 비면 생성 안 함 |
| `description` | 텍스트 | 상세 설명 | 상세 패널에만 표시(카드 표면 미노출) |
| `columnId` | 참조 | 속한 컬럼 | 이동 시 변경 |
| `position` | 정렬 키 | **컬럼 내 수동 정렬 순서** | 드래그 위치 보존. 관례상 정수 시퀀스가 아니라 **분수 인덱싱(fractional pos)** — 인접 두 카드 pos의 중간값을 부여해 다른 카드 재번호 없이 재정렬(멀티유저·낙관적 업데이트 유리). 물리 표현은 범위 밖 |
| `dueDate` | 날짜시간(선택) | 마감일 | ISO8601 관례. 현재시각 대비 지남(overdue) 상태 파생 |
| `dueComplete` | 불리언 | 마감 완료 토글 | 마감일과 짝(Trello 관례) — overdue 판정 시 완료된 마감은 제외 |
| `createdAt` | 날짜시간 | 생성 시각 | |

**Member (멤버)** — 보드 멤버 풀.

| 필드 | 개념 타입 | 의미 |
|---|---|---|
| `id` | 식별자 | 멤버 키 |
| `name` | 텍스트 | 이름 |
| `initials` | 텍스트 | 아바타 이니셜 폴백 |
| `avatarUrl` | 텍스트(선택) | 아바타 이미지 |
| `role` | enum `owner`/`editor`/`viewer` | 권한(개념적 — F8 읽기전용 상태용) |

**Label (라벨)** — 보드 레벨 라벨 풀.

| 필드 | 개념 타입 | 의미 |
|---|---|---|
| `id` | 식별자 | 라벨 키 |
| `name` | 텍스트 | 라벨 이름(이름 없는 색만도 허용) |
| `color` | 색 | 라벨 색(제품 데이터 — 파운데이션 토큰 아님) |

**Comment (댓글)** — Card 1:N, 시간순.

| 필드 | 개념 타입 | 의미 |
|---|---|---|
| `id` | 식별자 | 댓글 키 |
| `cardId` | 참조 | 속한 카드 |
| `authorId` | 참조(Member) | 작성자 |
| `body` | 텍스트 | 본문. **@멘션은 body 내 텍스트로만 보유** — 별도 멘션 참조 관계를 두지 않고, 알림 대상은 body에서 Member를 파싱해 해소(아래 노트) |
| `createdAt` | 날짜시간 | 작성 시각 |

> **멘션 모델링 노트(근거 있는 결정).** Comment에 `mentions: Member[]` 같은 별도 참조를 두지 않는다 — @멘션은 `body` 텍스트를 파싱해 Member로 해소하는 파생 동작이다(F4 "멘션된 사람만 알림"의 데이터 근거). 별도 관계가 필요한 강한 멘션(예: 멘션 인덱스/알림 영속)은 P1 알림 센터에서 도입 가능. **댓글 해결 상태(`resolved`)는 MVP 모델에 없음** — 카드 표면은 "댓글 수(total)"만 쓰고(F4·F6), "미해결" 구분은 P1(§4-2)에서 `resolved` 필드 추가 시 성립.

**Activity (활동 로그)** — Card 1:N, **append-only 시간순**(감사 로그 패턴).

| 필드 | 개념 타입 | 의미 |
|---|---|---|
| `id` | 식별자 | 활동 키 |
| `cardId` | 참조 | 대상 카드 |
| `actorId` | 참조(Member) | 행위자 |
| `type` | enum | 이벤트 타입(`moved`/`assigned`/`labeled`/`dueChanged`/`created`…) |
| `payload` | 객체 | 변경 전후 값 |
| `createdAt` | 날짜시간 | 발생 시각 |

### 관계
- **Board 1 : N Column** (순서 있음).
- **Column 1 : N Card** (컬럼 내 순서 있음 — position).
- **Card N : M Member**(담당자) — 관례는 다대다(다중 담당자). **MVP는 카드당 1명(+미배정)으로 단순화**하되 모델은 다대다 확장을 막지 않음.
- **Card N : M Label** — 카드는 라벨 ID 배열, 보드 라벨 풀을 참조.
- **Card 1 : N Comment**, **Card 1 : N Activity**.
- **Board 1 : N Member**(보드 멤버 풀) — 멤버가 카드에 배정됨.
- **필터는 관계가 아니다.** 담당자/라벨/마감 필터는 카드 필드에 술어를 거는 **파생 뷰 상태**일 뿐 외래키 연결이 아니다.

### 댓글/활동 모델링 노트 (근거 있는 결정)
- 댓글과 활동은 Trello처럼 **하나의 시간순 스트림의 두 종류**로 보거나, 디자인상 **'댓글/활동' 탭/필터로 분리**해도 둘 다 관례적으로 OK. 표시는 시간 역순, 날짜별 그룹핑, 사람이 읽는 문장.

### 영속(개념 수준)
- 보드·컬럼·카드·멤버·라벨·댓글·활동은 개념적으로 영속되나, **저장 위치/포맷·실시간 동기화는 범위 밖**(개발 단계 + 백엔드 몫). 디자인 산출물은 목업 데이터로 상태를 표현한다.

## 6. 사용자 흐름

> 단일 보드 화면(컬럼 + 카드)을 중심으로, 카드 상세 패널/모달이 위에 뜬다. 라우팅 최소.

**메인 보드 흐름 (S1~S5 통합).**
```
[진입] 단일 보드: 컬럼 헤더(이름+카드수) + 카드 + 상단 필터바(아바타 스택/라벨/마감)
  → 카드 드래그 ──(드래그 중: lift/플레이스홀더, 드롭존 하이라이트)──> [놓은 위치 삽입, 양 컬럼 카운터 갱신, 활동 로그 'moved' 기록]
  → 카드 클릭 ──> [카드 상세 패널 열림: 제목·설명·담당자·라벨·마감 + 하단 타임라인(댓글/활동)]
       → @멘션 댓글 작성 ──> [타임라인에 추가, 멘션된 사람만 알림(개념)]
       → 담당자/라벨/마감 변경 ──> [카드 표면 반영 + 활동 로그 기록]
  → "+ 카드 추가"(빈 컬럼/컬럼 하단) ──(제목 trim 비면 무시)──> [컬럼에 카드 추가, 'created' 기록]
  → 필터(담당자/라벨/마감 토글) ──> [조건 매칭 카드만, 활성 필터 강조, 컬럼 카운트=필터된 수] ──(0건)──> [필터별 빈 상태]
```

**테마 흐름 (S6 / F9).**
```
[SC1 헤더 테마 토글] → 라이트 ↔ 다크 전환(의미 토큰 분기, 별도 페어 스케일·4단 elevation) → 양 모드 모두 WCAG AA(G4) → 같은 작업 지속
```

**읽기 전용 흐름 (S7).**
```
[Viewer 진입] → 드래그 핸들 비활성·생성/편집 컨트롤 숨김/비활성 → "읽기 전용" 안내 → 보기만 가능
```

**상태 전이 요약.**
- Card.columnId/position: 드래그&드롭(F1)으로만 변경 → 활동 로그 'moved'.
- 필터(view state): 담당자/라벨/마감 술어 토글 — 보드는 항상 `(전체 카드) ∩ (활성 필터 조건)`으로 렌더, 컬럼 카운트는 필터 반영.
- 카드 상세 패널: 카드 클릭으로 열림/닫힘(보드 위 오버레이) — 변경은 카드 표면·활동 로그에 즉시 반영.
- 테마(F9): SC1 헤더 토글로 라이트 ↔ 다크 — 의미 토큰만 분기, 콘텐츠/보드 상태는 불변. 양 모드 a11y(G4) 유지.
- 댓글 해결 상태: MVP에 **없음** — 카드 표면 댓글 수는 total이며, "미해결" 상태를 만들/해소하는 전이는 정의하지 않는다(P1로 미룸 — dangling 없음).

## 7. 화면 목록·요구사항

> 데스크톱 우선 단일 보드. 보드 위에 카드 상세가 오버레이로 뜨는 구조. 각 화면/영역의 요구.

**SC1. 보드 화면 (메인)** — 영역:
1. **헤더/툴바** — 보드 이름, 상단 필터바(아바타 스택 토글 / 라벨 필터 / 마감 필터 / "내 카드만"), 테마 토글. 컴팩트·차분.
2. **컬럼 영역** — 가로 배치된 컬럼(3~6개). 각 컬럼: 헤더(이름 + 카드 수) + 카드 리스트(세로 스크롤) + 하단 "+ 카드 추가". 컬럼이 화면폭 초과 시 보드 가로 스크롤(G5).
3. **카드(컴팩트, 보드용)** — 표면: 제목(2줄 말줄임) · 담당자 아바타 · 라벨 1~2개(점/칩) · 마감일(overdue 강조) · 댓글 수(total). 그 이상은 노출 금지(정보밀도 절제). "미해결" 댓글 구분은 P1(§4-2).
4. **드래그&드롭 상태** — 드래그 가능 신호(hover lift) / 드래그 중(들림·플레이스홀더) / 드롭존 하이라이트 / 빈 컬럼 드롭존. 명시적 variant(G2).

**SC2. 카드 상세 패널/모달** — 영역:
1. **본문** — 제목 · 설명 · 담당자(배정/변경) · 라벨(추가/제거) · 마감일.
2. **타임라인** — 통합(댓글 + 활동) + '댓글/활동' 필터 토글(또는 탭). 댓글 입력(@멘션 칩 전제). 활동은 사람이 읽는 한 줄 문장.
3. 닫기·오버레이. 보드 위에 떠도 G5 무파손.

**SC3. 빈/경계 상태 (필수 variant — G3).**
- **빈 컬럼** — 드롭존 + "여기에 놓기"/"+ 카드 추가" 힌트.
- **필터 결과 0건** — 컬럼/보드 맥락 빈 상태("조건에 맞는 카드가 없습니다").
- **미배정 카드** — 담당자 placeholder.
- **overdue 카드** — 경고 색 강조(임박 D-카운트다운 관례).
- **아바타 +N** — 다수 겹침 처리(다중 담당자 확장 대비).
- **카드 과밀 컬럼** — 세로 스크롤 + 헤더 카운터.
- **읽기 전용(Viewer)** — 드래그 핸들 비활성 + 안내.

**화면 요구(공통).**
- 모든 인터랙티브 요소는 와이어드(죽은 컨트롤 0 — G1). 범위 밖 요소는 라벨/스텁으로 막다른 길 방지.
- 키보드 접근: 카드 이동(키보드 드래그 — 집기/이동/놓기), 필터 토글, 댓글 입력, 패널 열기/닫기 모두 포커스·키 조작 가능 + ARIA live region으로 드래그 상태 알림(데스크톱 우선이라 키보드 중요도 높음).
- 셀렉터 안정성: role·aria·텍스트 기반 식별(검증 spec이 포터블 셀렉터로 잡음).
- 데스크톱 1280px+에서 4~6컬럼이 가로 스크롤 없이 보이도록(컬럼당 카드 5~20 가정 — §11). 컬럼 초과 시 가로 스크롤.

## 8. 디자인 방향 (톤·레퍼런스·뷰포트·다크 모드)

- **톤.** 미니멀·차분(Linear류). 절제된 그레이 램프 + 포인트 컬러 1개. 장식 최소, 미세한 그레이 단계로 표면 elevation 구분(그림자/그라데이션 최소). 4px 스페이싱 그리드(시각 리듬). 텍스트/아이콘은 대부분 절제된 그레이(40~60% 톤), 풀채도 컬러는 상태/우선순위 신호·인터랙티브 요소·라벨에만.
- **레퍼런스.** Linear(보드 레이아웃·차분한 다크) — base + accent + contrast로 라이트/다크 생성. Trello(진입 단순함). 화려한 일러스트/그라데이션 지양.
- **뷰포트.** **데스크톱 우선(1280px+, 1440px 함께 검증).** 모바일은 1급 타깃 아님. 컬럼 가로 배치가 핵심 — 보드가 화면폭을 넘으면 가로 스크롤. /design 인테이크(00-flow)에서 대상 뷰포트로 1280px+를 명시.
- **색 모드(다크 모드).** **라이트 + 다크 둘 다(다크는 옵션이되 필수 산출).** 다크는 라이트의 단순 반전이 아니라 **별도 페어 스케일**로 설계(Linear도 ground-up 페어). 다크 표면은 **최소 4단 elevation**(base 배경 / 1차 elevated=카드·패널·사이드바 / 2차 elevated=중첩카드·hover / overlay=모달·툴팁·드롭다운).
- **토큰(표준 2단).** primitive 중립 그레이 램프를 **50~900 숫자 단계로 촘촘히(8~10단계)** — 미니멀 UI는 미세 그레이 차이로 elevation을 구분하므로 단계가 촘촘해야 함. accent 1개는 semantic role(`--color-primary`)로만 노출, **브랜드 단어로 토큰명 짓지 않음**. 따뜻한(blue-ish 회피) 그레이 경향. **라벨 색은 파운데이션 토큰이 아니라 별도 라벨 색 팔레트**(제품 데이터).
- **대비비 주의(근거 있는 결정).** 촘촘한 그레이 램프는 그레이끼리 가까워 **WCAG 위반 위험이 크다** — a11y 게이트에서 텍스트/표면/라벨칩 대비를 라이트·다크 양쪽 검증 필수(G4).
- **언어.** 한국어 UI. 모든 라벨·빈 상태·활동 문장·필터·placeholder 한국어. 활동 로그는 "홍길동 님이 In Progress로 이동함" 류 한국어 문장.

## 9. 비기능 요구사항

- **접근성(a11y).** 라이트·다크 양 모드 텍스트/주요 컨트롤/라벨칩 대비 WCAG **AA**(본문 4.5:1, 큰 글자/UI 3:1). **키보드 드래그&드롭**(집기/이동/놓기) + ARIA live region 상태 알림. 폼·체크박스·필터 토글에 적절한 라벨/aria.
- **데스크톱 레이아웃.** 1280px·1440px 무파손(G5). 컬럼 초과 시 가로 스크롤(겹침/잘림 0). 카드 상세 오버레이 무파손. 모바일은 비목표.
- **드래그&드롭 체감.** 즉각적 시각 피드백(real-time)으로 신뢰 형성 — 놓은 위치 삽입, 드롭존 하이라이트, 빈 컬럼 드롭존(핵심 체감 품질).
- **상태(개념).** 디자인 산출물은 목업 데이터로 상태 표현. 실제 영속·실시간 동기화는 범위 밖(handoff 대상 몫).
- **독립 실행.** atelier 규칙상 산출물은 외부 의존성(외부 라이브러리) 없는 독립 실행 HTML — 상태 변화·드래그용 인라인 JS 허용.
- **에러/방어.** 빈/공백 카드 제목 차단(F5). 긴 제목/라벨/댓글로 레이아웃 깨지지 않음(말줄임/접기). 카드 과밀·컬럼 과다에서 무파손.

## 10. 범위 외 (Out of Scope)

이번 MVP가 **명시적으로 하지 않는 것** — 착수자가 추측하지 않도록 못박는다.

- **실시간 동기화 · 계정/인증 백엔드** — 디자인 산출물 범위 밖. 멀티유저는 **개념적으로만**(아바타·담당자·활동 로그·읽기전용 상태) 표현(사용자가 닫음).
- **멀티 보드 / 워크스페이스 전환** — 단일 보드 MVP. 다음 단계(§4-2). 모델에 boardId만 남김.
- **WIP 한도 · 체크리스트/서브태스크 · 스윔레인 · 추정치 · 카드 우선순위** — 경쟁 도구에 흔하나 정보밀도 절제·단일 보드 경계로 제외(시장조사: 포지션에 맞게 범위 밖).
- **타임라인/간트 · AI/자동화 · 첨부 파일 관리 · 카드 의존성** — 미니멀 포지션 밖.
- **P1 다음 버전 후보(이번 제외)** — 멀티보드, WIP, 다중 담당자, 삭제 실행취소/아카이브, 우선순위 마커, 알림 센터(§4-2).
- **물리 데이터 스키마/저장 포맷 확정** — 개발 단계 결정(§5).
- **다국어** — 한국어 단일.
- **행동형 분석/KPI 수집** — 측정 인프라 없음(§2).
- **모바일 1급 지원** — 데스크톱 우선(반응 축소는 디자인 재량, 검증 비목표).

## 11. 미해결 이슈와 가정

> 시장조사로 근거 확보된 항목은 §1·§4·§5·§8에서 **근거 있는 결정**으로 본문에 반영했다. 아래는 (a) 사용자가 답한 전제, (b) 사용자가 답하지 않아 가정으로 닫은 것, (c) 여전히 추측이라 본문에서 단정하지 않은 것이다.

**A. 사용자 전제(인테이크 — 최우선, 본문 반영됨).**
- **A1.** 핵심 사용자 = 소규모 팀(3~10명). 플랫폼 = 데스크톱 우선(1280px+), 다크모드 옵션.
- **A2.** 비주얼 = 미니멀·차분(Linear류, 절제된 그레이 램프 + 포인트 컬러 1개).
- **A3.** MVP 협업 기능 = ① 드래그&드롭 이동 ② 담당자·아바타·멤버필터 ③ 댓글·활동로그 ④ 라벨·마감일·필터. 단일 보드 기준.
- **A4.** 실시간 동기화·계정/인증은 디자인 범위 밖(개념적 멀티유저로만 표현). 멀티보드/워크스페이스는 다음. 언어 = 한국어.

**B. 시장조사로 닫은 근거 있는 결정(가정 → 결정 승격).**
- **B1.** 4개 MVP 기능이 경쟁 도구 카드 칸반 핵심과 1:1 일치 — MVP 경계 타당, 충돌 없음(confidence grounded).
- **B2.** 댓글/활동은 카드 표면이 아니라 **상세 패널 하단 타임라인** — 모든 경쟁 도구 관례(grounded).
- **B3.** 카드 표면 정보 절제(제목·아바타·라벨점·마감·댓글수만), 나머지 상세 패널(grounded).
- **B4.** 드래그 위치 보존을 위한 **분수 인덱싱 position** 관례(grounded — 단, 물리 구현은 §5 범위 밖).
- **B5.** Linear가 실제 base+accent+contrast로 라이트/다크 생성 — 촘촘한 그레이 램프 + 단일 accent + 다크 별도 페어 + a11y 대비 게이트 필수(grounded).
- **B6.** 다크모드는 사용자 명시 옵션(A1)이며 Linear류 톤의 핵심 산출 — **테마 토글을 명세된 기능(F9)으로 둔다**(G4·S6·SC1·§8을 닫는 앵커). 라이트의 단순 반전이 아닌 별도 페어 스케일·4단 elevation·양모드 AA(grounded).
- **B7.** 카드 표면 댓글 표시는 **"댓글 수(total)"** 로 단순화 — MVP Comment 모델에 해결 상태가 없어 "미해결" 구분은 P1로 미룬다(미니멀 톤 부합, 모델-기능 정합). @멘션은 별도 참조 없이 body 파싱으로 해소(§5 노트).

**A-bis. 디자인 범위 확장(사용자 피드백 2026-06-26 — 1차 산출 이후).**
