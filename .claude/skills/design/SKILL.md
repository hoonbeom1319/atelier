# /design — 로컬 claude.ai/design 파이프라인

`claude.ai/design`을 로컬에서 재현하는 하니스다. 이 작업장(`ai-design/`)은 **여러 프로젝트를 담는 컨테이너**이고,
실제 작업은 `projects/<name>/` 안에서 일어난다. 형식·품질 규칙의 원본은 작업장 `CLAUDE.md`다. 충돌 시 그쪽을 따른다.

핵심 2단계: **(A) 어느 프로젝트인지 정한다 → (B) 그 프로젝트의 단계를 STATUS.md로 이어간다.**
한 번에 다 쏟지 말고, **단계마다 사용자 OK**를 받는다.

---

## A. 프로젝트 선택 (매번 먼저)

second 하다 first로 돌아올 수 있으므로, **현재 폴더 상태에 의존하지 말고** 명시적으로 고른다.

1. `projects/` 아래 폴더들을 훑고, 각 프로젝트의 `STATUS.md`에서 "현재 단계"를 읽어 **대시보드**로 보여준다:
   ```
   현재 작업장의 프로젝트:
   1) my-first-app    — 5 하이파이 진행 중
   2) my-second-app   — 3 흐름 테스트 (사용자 클릭 대기)
   3) my-third-app    — 1 인테이크
   + 새 프로젝트 만들기
   ```
2. 사용자가 이미 프로젝트명을 말했으면(예: "my-first-app 이어서") 그걸로 바로 간다.
3. 안 정해졌으면 **어느 프로젝트인지 묻는다.** 추측으로 시작하지 않는다.
4. **새 프로젝트**: `projects/<name>/`을 만들고 `PRD.md`(비어있으면 사용자에게 요청)와 `STATUS.md`(아래 템플릿)를 생성한 뒤 1단계로.

> `projects/`가 아예 없으면 만들고, 새 프로젝트 생성으로 안내한다.

### STATUS.md 템플릿

프로젝트 생성 시 `projects/<name>/STATUS.md`를 이렇게 만든다. **단계가 끝날 때마다 이 파일을 갱신**한다.

```markdown
# <name> — 진행 상태

현재 단계: 1-인테이크

- [ ] 1. 인테이크
- [ ] 2. 와이어프레임 (lo-fi)
- [ ] 3. 흐름 테스트 (사용자 클릭 확인)
- [ ] 4. 디자인 시스템
- [ ] 5. 하이파이 (hi-fi)
- [ ] 6. 인터랙션 테스트 (사용자 클릭 확인)
- [ ] 7. handoff

메모:
```

`[x]`는 **사용자가 OK한** 단계만. 특히 3·6번(테스트)은 사용자 확인 전엔 절대 체크하지 않는다.

## B. 선택한 프로젝트에서 단계 이어가기

`STATUS.md`의 "현재 단계"와 체크박스를 보고 해당 단계부터 진행한다. 모든 경로는 `projects/<name>/` 기준.

### 1. 인테이크
`PRD.md`를 읽고 `00-flow.md`로 정리·합의: **화면 목록 / 사용자 플로우(A→B·분기) / 핵심 상태(로딩·빈·에러·권한) / 열린 질문.**
톤·브랜드가 PRD에 없으면 3~4개만 묻는다(제품 성격 / 무드 / 핵심 컬러 / 다크모드). → 합의되면 STATUS 갱신.

### 2. 와이어프레임 (lo-fi) — 클릭되어야 한다
흑백 골격, 색·폰트·실데이터 없음, 빠르게.
- `wireframe/<screen>.html`, 첫 줄 `<!-- @dsCard group="Wireframe" -->`.
- **페이지끼리 진짜 `<a href="other.html">`로 링크** — 클릭하면 실제 이동.
- `wireframe/index.html`을 진입점으로. → 경로 알려주고 3단계로.

### 3. ▶ 흐름 테스트 (사용자가 한다)
**브라우저로 `wireframe/index.html` 열어 눌러보라**고 요청. 체크리스트: A→B 이동 / 빠진 화면 / 막다른 길 / 뒤로·취소 경로.
피드백은 **와이어에서 전부 수정.** 사용자가 "흐름 OK" 해야만 STATUS의 3번 체크 + 4단계로.

### 4. 디자인 시스템
- `foundation/tokens.css` — 색/타이포/스페이싱/라운드/그림자/모션을 CSS 변수로. 전 화면 공유. **(이건 메인이 직접. 토큰은 일관성의 근원이라 쪼개지 않는다.)**
- `foundation/colors.html`·`typography.html` (`group="Foundation"`), `components/<name>.html` (`group="Components"`, 모든 variant/size/state 나열).
- 컴포넌트가 많으면 **§C 에이전트 편성**을 적용할 수 있다(토큰 확정 후).

### 5. 하이파이 (hi-fi) — 여러 화면이면 §C 에이전트 편성
- 산출물: `screens/<screen>.html` (`group="Screens"`), `tokens.css` import, 와이어의 링크 구조 유지.
- 실제 디바이스 뷰포트(예: 390px) 안에서. 픽셀 완성도 + 상태(로딩/빈/에러) + **전환·애니메이션·마이크로 인터랙션.** `screens/index.html` 진입점.
- **화면이 여러 개면 직접 다 짜지 말고 §C로 편성**한다(브리프 분석 → 계획 제시 → 병렬 빌더 → 독립 검증). 1~2화면이면 메인이 직접.

### 6. ▶ 인터랙션 테스트 (사용자가 한다)
`screens/index.html` 눌러보라고 요청. *느낌* 확인: 전환 자연스러움 / 애니메이션 타이밍 / 호버·탭 피드백 / 상태 전이. OK면 STATUS 6번 체크 + 7단계로.

### 7. handoff
사용자가 대상 프로젝트(예: `../../dear`)로 넘기라 하면: **대상 CLAUDE.md를 먼저 읽고** 그 컨벤션으로 변환(예: `shared/ui` TSX), 토큰은 대상 시스템에 **매핑**(통째 복붙 금지), 무엇을 어디 넣었는지 요약 보고.

---

## C. 에이전트 편성 (생성 단계에서 동적으로)

생성량이 많은 단계(주로 5 하이파이, 때로 4 컴포넌트)에서, **브리프를 읽고 상황에 맞게** 서브에이전트를 편성한다.
고정 개수로 박지 않는다 — 안 쓸 에이전트는 안 띄우는 게 더 싸고 빠르다.

**원칙: 골격은 고정, 살은 동적.** 7단계 파이프라인은 그대로 두고, *단계 안에서 무엇을 몇 개 띄울지*만 브리프에 맞춰 정한다.

### 절차 (반드시 이 순서)

1. **브리프 분석** — `00-flow.md`/`PRD.md`를 읽고 두 축을 판단한다:
   - **규모(scale)** — 화면/컴포넌트 수 → 빌더 개수
   - **종류(kind)** — 브리프가 요구할 때만 전문 에이전트 추가:

   | 브리프가 이러면 | 띄울 에이전트 |
   |---|---|
   | 비주얼 방향 미정·탐색 필요 | **디자인 디렉션 N개** — 각자 다른 무드로 시안 → 비교(판정) |
   | 플로우 복잡·분기 많음 | **플로우 아키텍트** 1개 |
   | 데이터 많은 화면(리스트·대시보드) | **실데이터 콘텐츠** 에이전트 (lorem 방지) |
   | 상태(빈·에러·로딩) 많음 | **상태 커버리지** 에이전트 |
   | 단순 화면 여러 개 | 전문 에이전트 없이 **빌더 N개 병렬**만 |

2. **편성 계획 제시 (가드레일 ①)** — fan-out 하기 전에 **사용자에게 계획을 보여주고 OK를 받는다.**
   예: `"빌더 4 + 실데이터 콘텐츠 1 + 독립 검증 1로 갑니다. 이유: 화면 4개 중 2개가 리스트형. OK?"`
   에이전트는 토큰이 비싸다. 깜깜이로 늘리지 않는다.

3. **일관성 잠금 (가드레일 ②)** — 빌더에게 **반드시** 다음을 함께 넘긴다:
   - 확정된 `foundation/tokens.css` (모든 빌더가 같은 토큰 공유)
   - 화면별 스펙(`00-flow.md`에서 뽑은 "이 화면의 요소·상태")
   - 와이어의 링크 구조(같은 파일명·href 유지)
   이걸 안 먹이면 빌더마다 간격·네이밍·구조가 따로 놀아 디자인이 깨진다.

4. **병렬 빌더 fan-out** — 화면 1개 = 빌더 에이전트 1개. 각자 자기 화면 HTML만 만들어 **경로만 반환**(거대한 HTML이 메인 컨텍스트로 안 들어오게).

5. **독립 검증** — **HTML을 만들지 않은 별도 에이전트**에게 render-check를 맡긴다. 만든 놈이 자기 걸 검증하면 "괜찮은데?" 편향이 생긴다. 검증 에이전트는 각 산출물을 적대적으로 보고 thin/bad/variantsIdentical/off-brief 판정 + 수정 지시를 반환. 통과까지 빌더에게 되돌린다.

> **이 메커니즘을 쓰지 않는 곳:** 인테이크(1)·흐름 테스트(3)·인터랙션 테스트(6). 여긴 사용자와 대화해야 하므로 메인 스레드가 직접 한다. 와이어(2)도 보통 양이 적어 메인이 직접.

---

## render-check 루프 (모든 산출물에 적용)
만들 때마다 점검하고 걸리면 **통과까지** 고친다: **thin**(빈약) / **bad**(레이아웃 깨짐) / **variantsIdentical**(다 똑같음) / **off-brief**(brief 이탈). 애매하면 사용자에게 브라우저 확인 요청.
**§C로 편성한 단계에선 이 판정을 독립 검증 에이전트가 수행**한다(셀프 검증 금지).

## 원칙
- **항상 프로젝트 먼저 확정**, 그다음 단계. 폴더만 보고 프로젝트를 추측하지 않는다.
- 단계 사이엔 **항상 사용자 OK** + **STATUS.md 갱신.** 묻지 않고 직진 금지.
- 테스트 단계(3·6)는 사용자 확인 없이 통과 처리 금지.
- 생성량 많은 단계는 **§C로 동적 편성**(계획 제시 → 토큰·스펙 공유 → 병렬 빌더 → 독립 검증). 에이전트 개수를 고정으로 박지 않는다.
- 큰 PRD는 플로우 묶음 단위로 2~6단계 반복.
- 산출물은 외부 의존성 없는 독립 HTML. 브라우저로 바로 열려야 한다.
