# /design — 로컬 디자인 파이프라인

PRD를 받아 디자인 산출물을 단계별로 짓는 harness다. 이 작업장은 **여러 프로젝트를 담는 컨테이너**이고,
실제 작업은 `projects/<name>/` 안에서 일어난다. 형식·품질 규칙의 원본은 작업장 `CLAUDE.md`다. 충돌 시 그쪽을 따른다.

**이 스킬은 특정 제품에 묶이지 않는다.** 모바일·데스크톱·대시보드·웹뷰 무엇이든 같은 공정으로 처리한다.
아래의 모든 예시(화면명·뷰포트·컬러 등)는 *예시일 뿐*이다 — 실제 값은 그 프로젝트의 `00-flow.md`에서 정한다.

핵심 2단계: **(A) 어느 프로젝트인지 정한다 → (B) 그 프로젝트의 단계를 STATUS.md로 이어간다.**
한 번에 다 쏟지 말고, **단계마다(묶음으로 나눴으면 묶음마다) 사용자 OK**를 받는다.

---

## A. 프로젝트 선택 (매번 먼저)

second 하다 first로 돌아올 수 있으므로, **현재 폴더 상태에 의존하지 말고** 명시적으로 고른다.

1. `projects/` 아래 폴더들을 훑고, 각 프로젝트의 `STATUS.md`에서 "현재 단계"를 읽어 **대시보드**로 보여준다:
   ```
   현재 작업장의 프로젝트:
   1) my-first-app    — 5 하이파이 진행 중
   2) my-second-app   — 3 흐름·기능 검증 (묶음2 사람 OK 대기)
   3) my-third-app    — 1 인테이크
   + 새 프로젝트 만들기
   ```
2. 사용자가 이미 프로젝트명을 말했으면(예: "my-first-app 이어서") 그걸로 바로 간다.
3. 안 정해졌으면 **어느 프로젝트인지 묻는다.** 추측으로 시작하지 않는다.
4. **새 프로젝트**: `projects/<name>/`을 만들고 `PRD.md`(비어있으면 사용자에게 요청)와 `STATUS.md`(아래 템플릿)를 생성한 뒤 1단계로.

> `projects/`가 아예 없으면 만들고, 새 프로젝트 생성으로 안내한다.

### STATUS.md 템플릿

프로젝트 생성 시 `projects/<name>/STATUS.md`를 이렇게 만든다. **단계가 끝날 때마다(묶음으로 나눴으면 묶음마다) 이 파일을 갱신**한다.

```markdown
# <name> — 진행 상태

현재 단계: 1-인테이크

- [ ] 1. 인테이크
- [ ] 2. 와이어프레임 (lo-fi, 클릭+상태변화)
- [ ] 3. 흐름·기능 검증 (기계 green + 사람 판단)
- [ ] 4. 디자인 시스템
- [ ] 5. 하이파이 (hi-fi)
- [ ] 5.5 리디자인 (선택 — 발산 비주얼. 건너뛰어도 됨)
- [ ] 6. 인터랙션 검증 (기계 green + 사람 느낌)
- [ ] 7. handoff

후보 / chosen final:
- screens/ (원본) — (chosen?)

메모:
```

`[x]`는 **사용자가 OK한** 단계만. 특히 3·6번(검증)은 *기계 green* 과 *사람 확인* 이 **둘 다** 끝나기 전엔 체크하지 않는다.
**묶음으로 나눈 단계**는 메모에 묶음별 진행을 적는다(예: `2단계: 묶음1 OK / 묶음2 진행 중 / 묶음3 대기`).
**리디자인을 돌리면** 후보를 `후보 / chosen final`에 나열하고 사용자가 고른 하나를 `(chosen)`으로 표시한다 — **handoff는 이 chosen을 소비한다.**

## B. 선택한 프로젝트에서 단계 이어가기

`STATUS.md`의 "현재 단계"와 체크박스를 보고 해당 단계부터 진행한다. 모든 경로는 `projects/<name>/` 기준.

### 1. 인테이크
`PRD.md`를 읽고 `00-flow.md`로 정리·합의:
- **§화면 목록** / **§사용자 플로우(A→B·분기)** / **§화면별 구성요소·상태(빈·로딩·정상·에러)** / **§인터랙션 모델** / **§범위 표시** / **§열린 질문.**
- 톤·브랜드·**대상 뷰포트/플랫폼**(모바일 390px? 데스크톱? 대시보드?)이 PRD에 없으면 3~4개만 묻는다. → 합의되면 STATUS 갱신.

> `00-flow.md`의 **구성요소·상태**(§화면별)와 **인터랙션 모델**(§)은 장식이 아니다 — **3단계 기능 검증 spec이 여기서 파생된다.** 화면이 책임지는 동작·상태를 여기서 못박아야 나중에 검증할 게 생긴다.

### 2. 와이어프레임 (lo-fi) — 클릭되고, 상태가 변한다
흑백 골격, 색·폰트·실데이터 없음, 빠르게. **단, "정적 화면"이 아니다 — 상태는 진짜로 변한다.**
- `wireframe/<screen>.html`, 첫 줄 `<!-- @dsCard group="Wireframe" -->`. 진입점 `wireframe/index.html`.
- **페이지끼리 진짜 `<a href="other.html">`로 링크** — 클릭하면 실제 이동.
- **상태는 lo-fi에서도 실재한다(밋밋하게).** 사진 누르면 *선택됨*으로 보이고, 토글 끄면 *꺼진* 상태가 되고, 카운터가 *올라가고*, 미분류 수가 *−1* 된다. 색·전환·애니메이션은 없어도 되지만 **상태 자체는 변해야 흐름이 잡힌다.** 이를 위한 **최소 인라인 JS는 허용**된다 — 금지는 *외부 라이브러리 의존성*이지 JS가 아니다. (상태의 *느낌*은 5·6단계 hi-fi 몫.)
- **상태는 한 화면 안에서 바꾸는 게 기본.** 별도 페이지로 모델링(`loading.html`→`result.html`)하는 건 로딩→결과 같은 **큰 점프**에만 쓰는 예외다. 토글·선택·카운터 같은 건 그 화면 안에서.
- **죽은 컨트롤 금지 = 모든 인터랙티브 요소는 둘 중 하나다:** ① 자기 책임을 실제로 수행(실제 페이지로 이동하거나, 관측 가능한 상태 변화를 일으킴) 또는 ② `[범위 밖]`으로 **명시**. 살아있는 척하면서 아무것도 안 하는 컨트롤은 **버그**다(덜 만든 것과 구분되게 라벨링).
- **인터랙션 모델을 캡처한다(이건 구조다).** 단일 vs 다중 선택, 드래그, 인라인 편집, 일괄 액션 등 — 모델이 다르면 화면 구조(선택 모드·카운터·일괄 액션 바·시트)가 달라진다. *모양*이 아니라 *구조*다. 모호하면 `00-flow.md`로 거슬러 올라가 고친다.

### 3. ▶ 흐름·기능 검증 (기계 + 사람) — **검증은 3층이다**

| 층 | 주체 | 무엇을 | 언제 |
|---|---|---|---|
| **1. 내비게이션** | 기계 | A→B 링크 작동 / 막다른 길 0 / 범위 안 전 화면 도달 | 바닥. 깔고 시작 |
| **2. 기능 완결성** | 기계 | **각 화면이 책임지는 동작이 실제로 작동하나** + 죽은 컨트롤 0 + 상태가 실제로 변하나 | ★ 이 단계의 핵심 |
| **3. 느낌·판단** | 사람 | "개념적으로 옳은 흐름인가 / 빠진 건 없나" (3단계) · 전환·애니메이션 *느낌* (6단계) | 사람만 안다 |

**(a) 기계 — Playwright green까지 루프.** 검증 에이전트가 `00-flow.md`의 **§플로우(1층) + §구성요소·상태 + §인터랙션 모델(2층)** 에서 spec을 파생한다. 검증 항목:
- **1층:** 각 경로 A→B 이동 / 막다른 링크 부재 / 범위 안 도달성 (index에서 BFS 크롤).
- **2층:** 화면이 책임지는 인터랙션이 *완결*되나 — 예) 다중선택 → 카운터 증가 → 일괄 액션 → 결과 도달 / 미분류 항목 N개 지정의 **모든 경로**(맞는 분류에 넣기·새 분류 추가·직접 지정)가 완료에 도달 / 토글이 실제로 뒤집힘(`aria-pressed`) / 팔로우 등 관계 액션의 흐름이 *존재*하나. **모든 인터랙티브 요소를 열거해 "와이어드 됐나 vs 죽었나"를 assert**(죽은 컨트롤 0). 범위 밖 라벨 요소는 면제.
- **포터블하게 짠다:** 셀렉터는 `getByRole`·`aria-*`·텍스트로. (클래스·DOM 구조 X) → **같은 spec이 6단계 hi-fi에도 그대로 돌아간다.**
- **루프:** 실패 → 와이어 수정 → 재실행을 green 될 때까지. → 사전 준비·재사용 골격은 아래 **Playwright** 참고.

**(b) 사람 — 판단(3층).** green이 된 와이어를 사용자가 직접 눌러보며 *"이게 맞는 흐름인가 / 개념적으로 빠진 건 없나"* 를 본다. '옳은 흐름인가'는 사람만 안다.

**기계 green + 사람 OK 둘 다** 통과해야 STATUS 3번 체크 + 4단계로. **묶음으로 나눴으면 이 게이트는 묶음마다 건다**(아래 '묶음 진행 순서').

### 4. 디자인 시스템
- `foundation/tokens.css` — 색/타이포/스페이싱/라운드/그림자/모션을 CSS 변수로. 전 화면 공유. **(메인이 직접. 토큰은 일관성의 근원이라 쪼개지 않는다.)**
- **토큰 구조는 표준 2단으로 짠다 — 프리스타일 금지.** 브리프의 브랜드 단어("코랄")에 끌려 토큰 이름을 브랜드명으로 짓지 않는다.
  1. **primitive(원시값)** — 팔레트. `--coral-500`, `--gray-50…900`처럼. **중립/회색은 숫자 램프 필수**(거의 항상 여러 단계가 필요), 액센트는 *실제 쓰는 단계만*.
  2. **semantic(역할)** — primitive를 별칭. `--color-primary: var(--coral-500)` / `--color-text` / `--color-text-muted` / `--color-bg` / `--color-surface` / `--color-border` / `--color-on-primary` / 상태색(success·warn·danger). **컴포넌트·화면은 semantic만 쓴다**(primitive 직접 참조 금지).
  - **이름은 역할이지 브랜드가 아니다.** `--color-primary`(O) / `--c-coral`(X). 코랄은 *값*이다 — 리스킨해도 이름이 거짓말이 안 되게.
  - **타이포·간격·모서리·그림자·모션은 스케일/네임드 토큰**(`--fs-*`, `--sp-1..n`, `--r-*`, `--shadow-*`, `--dur/--ease`)으로. 이미 표준이면 유지.
  - **제품 전용 토큰은 파운데이션에 넣지 않는다.** 예) `--map-route`·`--map-pin` 같은 특정 컴포넌트 결정은 그 `components/<name>.html` 안으로. 토대엔 전역으로 재사용되는 것만.
  - **카고컬트 금지:** "1색 미니멀" 브리프에 50~900 9단계를 *억지로* 만들지 않는다. 규칙은 *중립 램프 필수 / 액센트는 쓰는 만큼 / semantic 역할층은 항상 / 제품색은 컴포넌트로*.
- `foundation/colors.html`·`typography.html` (`group="Foundation"`), `components/<name>.html` (`group="Components"`, 모든 variant/size/state 나열).
- 컴포넌트가 많으면 **§C 에이전트 편성**을 적용할 수 있다(토큰 확정 후).

### 5. 하이파이 (hi-fi) — 여러 화면이면 §C 에이전트 편성
- 산출물: `screens/<screen>.html` (`group="Screens"`), `tokens.css` import, **와이어의 링크 구조 + 상태 동작 그대로 유지**(여기선 거기에 *살*을 입힌다 — 색·타이포·전환·애니메이션·마이크로 인터랙션).
- 그 프로젝트의 대상 뷰포트(00-flow에서 정한 값) 안에서. 픽셀 완성도 + 상태(로딩/빈/에러) + **전환·애니메이션.** 진입점 `screens/index.html`.
- **화면이 여러 개면 직접 다 짜지 말고 §C로 편성**한다(브리프 분석 → 계획 제시 → 병렬 빌더 → 독립 검증). 1~2화면이면 메인이 직접.

> **한계 인지:** hi-fi는 와이어가 얼린 레이아웃에 *살*을 입히는 단계라, 와이어가 잘 됐을수록 "와이어 + 색"처럼 도약이 작게 느껴진다(정상이다). **진짜 비주얼 도약이 필요하면 5.5 리디자인으로.**

### 5.5 ▶ 리디자인 (선택) — 발산적 비주얼 재해석
hi-fi가 "와이어 + 색"에 머물러 **도약이 필요할 때만**. 건너뛰어도 되는 선택 단계다. *재색칠이 아니라 다른 아트디렉션으로 재해석.*
- **히스토리를 전부 읽는다:** `00-flow.md`(=계약), `wireframe/`(구조 의도), 현재 `screens/`(현 디자인), `tokens.css`(현 시스템). 맥락 없이 새로 그리지 않는다.
- **지키는 계약(불변):** 화면 목록 / 흐름(§2) / 기능 책임·상태(§3·§5). 즉 **무엇을 하는 앱인지는 안 바뀐다.**
- **깨도 되는 것(여기가 핵심):** 구도·레이아웃·정보 밀도·타이포 인격·색 시스템·모션 언어·이미지/일러스트·장식. **재색칠만 하면 같은 불만이 반복된다 — 명시적으로 "다른 시각".**
- **출력은 원본을 덮지 않는다.** 후보 폴더로: `screens-<variant>/`(예: `screens-editorial/`, `screens-bold/`). 토큰 변형이 필요하면 `foundation/tokens-<variant>.css`.
- **N개 발산이면 §C로 편성** — 디자인 디렉션 에이전트 N개가 각자 다른 무드/아트디렉션으로 후보 생성 → 비교. (계획 제시 → OK → 병렬 fan-out → 독립 검증.)
- **가드레일 = 기능 spec 재실행.** 각 후보는 3단계 기능 spec(2층)을 **그대로 통과**해야 한다. 완전히 달라 보여도 *기능 동등*이어야 후보 자격. (role·aria·text 셀렉터라 마크업 갈아엎어도 의미만 지키면 green.) green 아니면 탈락/수정.
- **선택(사람·3층):** 사용자가 후보들(+원본)을 브라우저로 비교 → **chosen final** 하나를 고른다. STATUS의 `후보 / chosen final`에 기록. '어느 게 제일 좋나'는 사람만 안다.

### 6. ▶ 인터랙션 검증 (기계 + 사람)
- **대상은 chosen final.** 리디자인 했으면 고른 후보, 안 했으면 `screens/`.
- **(a) 기계 —** 3단계의 **기능 spec(2층)을 chosen final에 그대로 재실행**한다. 폴리시/리디자인 과정에서 동작이 깨지지 않았는지(회귀) green까지. (포터블 셀렉터라 거의 그대로 돈다.)
- **(b) 사람 —** *느낌*(3층) 확인: 전환 자연스러움 / 애니메이션 타이밍 / 호버·탭 피드백 / 상태 전이의 매끄러움.
- 기계 green + 사람 OK → STATUS 6번 체크 + 7단계로.

### 7. handoff — chosen final을 *패키지 + 인계 프롬프트*로 묶는다 (직접 호출 가능)
파이프라인 끝 단계이자 **언제든 직접 호출 가능**(재-handoff·다중 대상). **항상 chosen final**(리디자인 했으면 고른 후보, 아니면 `screens/`)을 소비한다.

**핵심 산출물은 "받는 쪽 Claude Code를 구동하는 인계 프롬프트"다.** atelier는 대상 코드를 **직접 짜지 않는다** — **디자인 패키지를 만들고, 그걸 참고해 짜라는 프롬프트(`hand-off.md`)를 건넨다.** 대상 프로젝트의 Claude가 자기 컨벤션으로 구현한다. **특정 프레임워크·아키텍처에 묶이지 않는다 — 컨벤션은 전적으로 대상 CLAUDE.md가 정한다.**

`projects/<name>/handoff/`에 모은다:
1. **비주얼 handoff 문서** `handoff/index.html` — chosen final **스크린샷** + **흐름맵**(00-flow §2) + **화면별 스펙**(구성요소·상태·기능 책임). 클릭으로 훑게.
2. **토큰 매핑표** `handoff/tokens-map.md` — 우리 **semantic 토큰**(`--color-primary` 등) → **대상 시스템 토큰** 대응표. (통째 복붙 금지 — 매핑.)
3. **컴포넌트 인벤토리** `handoff/components.md` — variant/size/state 목록 + **각 컴포넌트가 쓰이는 화면** 매핑.
4. **제품 요구사항(PRD)** — `projects/<name>/PRD.md`(`/plan` 산출물)가 있으면 **그대로 인계에 포함**한다. 별도 복사 없이 hand-off.md의 "먼저 읽을 것"에서 경로로 참조한다(없으면 — PRD 없이 바로 `/design`으로 시작한 프로젝트면 — 이 항목은 생략하고 `00-flow.md`가 계약의 최상위 근거가 된다). **PRD는 "왜·무엇"(배경·목표·기능·데이터 모델·엣지케이스)의 근거**, `00-flow.md`는 "어떻게 흐르나"의 계약 — 둘은 보완 관계다.
5. ★ **인계 프롬프트 `handoff/hand-off.md`** — 받는 쪽 Claude Code에 **그대로 붙여넣는 브리프.** "어떤 파일을 참고해 / 무엇을 지키고 / 무엇을 만들지"를 못박는다. 골격:

```markdown

## 먼저 읽을 것

- 제품 요구사항 (있으면 — `/plan` 산출물):
  - `<atelier>/projects/<name>/PRD.md` — 배경·목표·기능 요구사항(동작·엣지케이스)·**데이터 모델**·범위 외 = **왜·무엇의 근거**. 흐름과 디자인이 *왜* 이렇게 됐는지 여기서 확인.
- 디자인 패키지 (atelier 산출물, 경로):
  - chosen final 화면 `<atelier>/projects/<name>/<chosen>/` — 최종 비주얼·상태·전환
  - `<...>/handoff/index.html` — 스크린샷 + 흐름맵 + 화면별 스펙
  - `<...>/handoff/tokens-map.md` — semantic 토큰 → 너의 토큰 매핑
  - `<...>/handoff/components.md` — 컴포넌트 variant/state + 사용 화면
  - `<...>/projects/<name>/00-flow.md` — 흐름(§2)·기능 책임(§3·§5) = **지킬 계약**

## 지킬 것 (계약 — 임의 변경 금지)

- 토큰은 매핑표대로 **대상 시스템 토큰에 매핑**(통째 복붙 금지).
```

## 묶음 진행 순서 (큰 PRD — 중요)

화면이 많아 묶음(청크)으로 나눌 때, **두 규칙이 서로 다른 높이에서 동시에 성립한다.**

**규칙 ① 단계(세로)로 자른다 — 한 묶음을 hi-fi까지 내리지 않는다.**
`tokens.css`는 제품 전체가 공유하는 토대다. 1묶음만 보고 만들면 뒤 묶음에서 새 컴포넌트가 나올 때마다 땜질하게 된다. **모든 묶음의 구조가 확정된 뒤 한 번에** 만든다.

**규칙 ② 단계 *안에서는* 묶음별로 게이트를 건다 — "한 번에 다 만들고 전체검증" 금지.**
와이어 단계에서 13화면을 한꺼번에 쏟고 "다 됐어, 전체 흐름검증해줘"로 가지 않는다. **묶음1 와이어 → 기계 green + 사람 OK → 묶음2 와이어 → … 묶음별로 끊어서** 내려간다. 묶음1의 구조 결정(인터랙션 모델·선택 패턴)이 검증된 뒤에 묶음2가 그걸 물려받아야 품질이 올라간다. 이게 사용자가 "스텝바이스텝으로 가자"고 한 이유다.

> 규칙 ①과 ②는 충돌이 아니다. ①은 *"묶음을 hi-fi까지 내리지 마라"*(세로 배리어), ②는 *"한 단계 안에서도 묶음마다 검증·확인하고 넘어가라"*. 다른 높이의 규칙이라 둘 다 지킨다.

**묶음 간 링크 처리:** 묶음N을 단독 검증할 때 아직 안 만든 다음 묶음으로 가는 링크는 **스텁**(예: `다음 묶음 →`이라 표시된 임시 페이지)으로 잇거나 `[범위 밖(다음 묶음)]`으로 명시한다 — 묶음N 안에 막다른 길이 없게. **전체 도달성 크롤(스텁 제거·묶음 간 링크 연결)은 와이어 단계 배리어에서 1회.**

```
[와이어 단계 — 묶음별 게이트]
  묶음1 와이어 → 기계 green(1·2층) + 사람 OK
  묶음2 와이어 → 기계 green(1·2층) + 사람 OK
  묶음3 와이어 → 기계 green(1·2층) + 사람 OK
        │
        ▼  전체 도달성 크롤 1회 (스텁 제거, 묶음 간 링크 연결)   ← 와이어 단계 배리어
  tokens.css : 1회 (전 묶음 구조 확정 후)                        ← 토큰은 공유 자산
        │
        ▼
  하이파이: 묶음별  →  인터랙션 검증: 묶음별  →  handoff
```

## Playwright (흐름·기능 자동 검증)

3·6단계의 *기계적* 검증(1층 내비게이션 + 2층 기능 완결성)을 자동화한다. **3층 느낌은 자동화하지 않는다 — 사람 몫.**

- **설치는 워크스페이스에 한 번** — `package.json` + `@playwright/test` + 브라우저. 프로젝트마다 다시 깔지 않는다. 공용 `playwright.config.js`는 `testDir: ./projects`, `testMatch: **/*.spec.js`. 와이어는 `file://` 상대 `<a href>`로 동작해 `webServer` 불필요.
- **실행은 프로젝트별 헬퍼로 — 증빙이 그 프로젝트 폴더에 남는다.** `node scripts/test-project.js <name> [스펙경로]` (예: `node scripts/test-project.js family-todo projects/family-todo/wire.spec.js`). 헬퍼가 `ATELIER_PROJECT` env를 세팅하면 공용 config가 리포트·결과를 **`projects/<name>/playwright-report` · `projects/<name>/test-results`** 로 라우팅한다. **`npx playwright test`를 맨손으로 돌리지 말 것** — 그러면 루트 한 곳에 쌓여 다른 프로젝트 증빙을 덮어쓴다(어느 프로젝트 결과인지 헷갈림). 프로젝트별 증빙은 `.gitignore`로 제외됨.
- **뷰포트는 프로젝트 값으로.** 00-flow에서 정한 뷰포트를 spec/실행에 쓴다(예: 모바일 390px, 데스크톱 1280px). config의 기본 디바이스와 어긋나면 spec에서 `page.setViewportSize`로 맞춘다.
- **스펙은 흐름 문서에서 파생** — `projects/<name>/*.spec.js`를 그 프로젝트 `00-flow.md`의 **§플로우 + §구성요소·상태 + §인터랙션 모델**에서 생성한다. 흐름 문서가 진실의 원천이다.
- **재사용 골격(매 프로젝트 재발명 금지):**
  - **1층 — 막다른 길 + 도달성:** `index.html`에서 BFS로 `a[href]`를 따라가며 ⓐ 링크 타깃 파일이 실제 존재(막다른 길 0) ⓑ 범위 안 필수 화면이 전부 방문됨(도달성)을 assert.
  - **2층 — 기능 완결성:** 화면별로 *책임 동작*을 assert. 패턴: 핵심 흐름 한 줄 통과 / 분기마다 결과 도달 / 토글·선택의 상태 변화(`aria-pressed`·`aria-selected`·카운터 텍스트) / 인터랙티브 요소 열거 → 각각 "클릭 후 URL 또는 DOM 상태가 변했나(= 와이어드)" 검사, 안 변하면 `[범위 밖]` 라벨 있는지 확인(없으면 = 죽은 컨트롤 = 실패).
  - 셀렉터는 `getByRole`·`aria-*`·텍스트(클래스·구조 X) → **3·6단계 양쪽 이식.**
  - **⚠ 넓은 `getByText(/단어/)`는 strict-mode 위반을 부른다.** 한 단어가 여러 요소에 걸리면(흔한 예: *상태 데모 버튼* "오프라인" + *오프라인 배너 텍스트*, 또는 같은 문구가 `#sub`·`#hint`처럼 두 군데) Playwright가 "resolved to 2 elements"로 실패한다. **좁혀라:** 특정 `#id` 로케이터 / `getByRole`+`name` / exact 텍스트 / `.locator(scope)` 안으로. 데모 컨트롤과 상태 표시가 단어를 공유하면 거의 항상 터지니, spec 작성 시 미리 스코프를 건다.
  - **strict-mode 실패 = spec 결함이지 산출물 버그가 아니다.** "2 elements"로 빨개지면 **와이어/hi-fi를 고치지 말고 셀렉터를 좁혀라.** (산출물 동작은 멀쩡한데 spec이 모호한 것.)
- **루프:** 실패 → 산출물 수정 → 재실행을 green까지. green 없이 사람에게 넘기지 않는다. **단, 실패 원인이 spec(셀렉터 모호·오타)인지 산출물(죽은 컨트롤·상태 안 변함)인지부터 가른다 — 전자는 spec을 고치고, 후자만 산출물을 고친다.**
- **증빙은 항상 남긴다(사람의 3층 판단용).** 공용 `playwright.config.js`가 `html` 리포터 + `screenshot:'on'` + `trace:'on'`로 설정돼 있어, 매 실행마다 **테스트별 pass/fail + 화면 스크린샷 + 클릭·상태변화 step-by-step 트레이스**가 그 프로젝트의 `projects/<name>/playwright-report`·`projects/<name>/test-results`에 남는다. 실행 후 **사용자에게 `npx playwright show-report projects/<name>/playwright-report`로 열어보라**고 안내한다 — 경로에 프로젝트명을 꼭 붙여라(맨손 `show-report`는 루트/직전 프로젝트 걸 열어 엉뚱한 결과를 보여준다). "무엇을 검증했고 어디서 멈췄나"가 한눈에 보인다.
- **터미널 "passed"만 보고 끝내지 않는다.** green이어도 리포트/스크린샷 경로를 사용자에게 알려, 사람이 실제 화면·흐름을 눈으로 확인하게 한다.
- (선택) 전 화면을 PNG로 빠르게 훑고 싶으면, `wireframe/*.html`을 glob해 한 장씩 캡처하는 작은 스크립트로 컨택트시트(`_shots/`)를 만들 수 있다 — 리포트 서버 없이 폴더에서 바로 눈으로.

## C. 에이전트 편성 (생성 단계에서 동적으로)

생성량이 많은 단계(주로 5 하이파이, 때로 4 컴포넌트)에서, **브리프를 읽고 상황에 맞게** 서브에이전트를 편성한다.
고정 개수로 박지 않는다 — 안 쓸 에이전트는 안 띄우는 게 더 싸고 빠르다.

**원칙: 골격은 고정, 살은 동적.** 7단계 파이프라인은 그대로 두고, *단계 안에서 무엇을 몇 개 띄울지*만 브리프에 맞춰 정한다.

### 절차 (반드시 이 순서)

1. **브리프 분석** — `00-flow.md`/`PRD.md`를 읽고 두 축을 판단한다:
   - **규모(scale)** — 화면/컴포넌트 수 → 빌더 개수
   - **종류(kind)** — 브리프가 요구할 때만 전문 에이전트 추가:

   | 브리프가 이러면 | 띄울 에이전트 |
   |---|---|
   | 비주얼 방향 미정·탐색 필요 | **디자인 디렉션 N개** — 각자 다른 무드로 시안 → 비교(판정) |
   | 플로우 복잡·분기 많음 | **플로우 아키텍트** 1개 |
   | 데이터 많은 화면(리스트·대시보드) | **실데이터 콘텐츠** 에이전트 (lorem 방지) |
   | 상태(빈·에러·로딩) 많음 | **상태 커버리지** 에이전트 |
   | 단순 화면 여러 개 | 전문 에이전트 없이 **빌더 N개 병렬**만 |

2. **편성 계획 제시 (가드레일 ①)** — fan-out 하기 전에 **사용자에게 계획을 보여주고 OK를 받는다.**
   예: `"빌더 4 + 실데이터 콘텐츠 1 + 독립 검증 1로 갑니다. 이유: 화면 4개 중 2개가 리스트형. OK?"`
   에이전트는 토큰이 비싸다. 깜깜이로 늘리지 않는다.

3. **일관성 잠금 (가드레일 ②)** — 빌더에게 **반드시** 다음을 함께 넘긴다:
   - 확정된 `foundation/tokens.css` (모든 빌더가 같은 토큰 공유)
   - 화면별 스펙(`00-flow.md`에서 뽑은 "이 화면의 구성요소·상태·책임 동작")
   - 와이어의 링크 구조 + **상태 동작**(같은 파일명·href·인터랙션 유지)
   이걸 안 먹이면 빌더마다 간격·네이밍·구조·동작이 따로 놀아 깨진다.

4. **병렬 빌더 fan-out** — 화면 1개 = 빌더 에이전트 1개. 각자 자기 화면 HTML만 만들어 **경로만 반환**(거대한 HTML이 메인 컨텍스트로 안 들어오게).

5. **독립 검증** — **HTML을 만들지 않은 별도 에이전트**에게 검증을 맡긴다. 만든 놈이 자기 걸 검증하면 "괜찮은데?" 편향이 생긴다. 검증 에이전트는 각 산출물을 적대적으로 보고 **render-check(thin/bad/variantsIdentical/off-brief) + 기능 완결성(죽은 컨트롤·상태 안 변함)** 판정 + 수정 지시를 반환. 통과까지 빌더에게 되돌린다.

> **이 메커니즘을 쓰지 않는 곳:** 인테이크(1) — 사용자와 대화해야 하므로 메인이 직접. 와이어(2)도 보통 양이 적어 메인이 직접.
> **검증 단계(3·6)는 절반만:** *기계적* 검증(1·2층 green 여부)은 검증 에이전트가 자동으로 돌리고, *판단*(3층: 옳은 흐름인가·느낌)은 사람이 한다.

---

## render-check 루프 (모든 산출물에 적용)
만들 때마다 점검하고 걸리면 **통과까지** 고친다:
- **thin**(빈약) / **bad**(레이아웃 깨짐) / **variantsIdentical**(다 똑같음) / **off-brief**(brief 이탈)
- **deadControl**(인터랙티브한데 와이어드 안 됨, 범위 밖 라벨도 없음) / **stateInert**(눌렀는데 상태가 안 변함)

애매하면 사용자에게 브라우저 확인 요청. **§C로 편성한 단계에선 이 판정을 독립 검증 에이전트가 수행**한다(셀프 검증 금지).

## 원칙
- **항상 프로젝트 먼저 확정**, 그다음 단계. 폴더만 보고 프로젝트를 추측하지 않는다.
- 단계 사이엔(묶음으로 나눴으면 묶음마다) **항상 사용자 OK** + **STATUS.md 갱신.** 묻지 않고 직진 금지.
- **검증은 3층이다:** 내비게이션(기계) + 기능 완결성(기계) + 느낌·판단(사람). "기계는 내비게이션만"은 틀렸다 — 화면이 책임지는 동작·상태 변화도 기계가 green까지 본다.
- **죽은 컨트롤은 버그다.** 모든 인터랙티브 요소는 와이어드 되거나 `[범위 밖]`이거나 둘 중 하나.
- **lo-fi도 상태는 진짜 변한다**(밋밋하게, 인라인 JS 허용). *느낌*만 hi-fi로 미룬다.
- **hi-fi가 "와이어 + 색"에 머물면 5.5 리디자인**으로 발산(흐름·기능은 지키고 비주얼 언어만 재해석). 후보는 원본을 덮지 않고, **기능 spec(2층)을 통과**해야 후보 자격. 사용자가 **chosen final**을 고르고, **6·7단계는 그 chosen을 소비**한다.
- **handoff의 산출물은 코드가 아니라 "인계 프롬프트 `hand-off.md`" + 디자인 패키지**다. atelier는 대상 코드를 직접 짜지 않는다 — 받는 쪽 Claude가 그걸 보고 **자기 컨벤션**으로 짠다. (특정 프레임워크·아키텍처에 묶이지 않음.)
- 검증 단계(3·6)는 *기계 green* 과 *사람 확인* 둘 다 없이 통과 처리 금지.
- 생성량 많은 단계는 **§C로 동적 편성**(계획 제시 → 토큰·스펙·상태동작 공유 → 병렬 빌더 → 독립 검증). 에이전트 개수를 고정으로 박지 않는다.
- **큰 PRD는 ①단계(세로)로 자르고 ②단계 안에선 묶음마다 게이트.** "한 번에 다 만들고 전체검증" 금지. 한 묶음을 hi-fi까지 내리지 않는다.
- 산출물은 외부 의존성(외부 라이브러리) 없는 독립 HTML. 상태용 인라인 JS는 허용. 브라우저로 바로 열려야 한다.
- 이 스킬은 **범용**이다 — 예시(화면명·뷰포트·컬러)에 끌려가지 말고, 실제 값은 그 프로젝트 `00-flow.md`에서 정한다.
