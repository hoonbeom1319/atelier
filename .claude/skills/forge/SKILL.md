# /forge — 무인 파이프라인 오케스트레이터 (아이디어 → hi-fi)

`/plan`(기획)과 `/design`(디자인)을 **사람 게이트 없이** 이어 붙여, 아이디어 한 줄에서 **최종 hi-fi**까지 자율로 내려가는 harness다.
`/plan`·`/design`을 **대체하지 않는다 — 그 공정을 무인 모드로 구동하는 상류 오케스트레이터**다. 단계별 *절차*의 원본은 여전히 두 SKILL이고, 이 문서는 *"언제 사람을 빼고, 무엇으로 그 자리를 메우나"* 의 원본이다.

```
/forge  (시작 1회 질문)  →  [무인] 시장조사(웹검색)→ /plan 공정 → PRD.md(lint-prd green) + plan-decisions.md
                          →  [무인] /design 공정 → 와이어(Playwright green) → tokens(a11y green) → hi-fi(axe green)
                          →  트렌드 전문가(웹검색) 감정 → (미달 시 자동 리디자인 1회) → design-critique.md
                          →  최종 hi-fi + 가정·근거·감정 요약 전달
```

형식·작업장 규칙의 원본은 `CLAUDE.md`, 공정 절차의 원본은 `plan/SKILL.md`·`design/SKILL.md`다. 충돌 시 그쪽을 따른다.
이 스킬은 **범용**이다 — 웹·앱·대시보드·내부툴 무엇이든 같은 무인 공정.

---

## 0. 무인 계약 — 무엇을 빼고, 무엇으로 메우나 (이 스킬의 존재 이유)

`/plan`·`/design`은 **두 종류의 게이트**를 가진다. /forge는 둘을 **다르게** 다룬다:

| 게이트 종류 | 예 | /forge에서 |
|---|---|---|
| **자동 게이트**(스크립트가 본다) | `lint-prd` · Playwright 1·2층(내비·기능 완결성·죽은 컨트롤 0) · a11y(`defineTokenContrastTest`·`defineAxeTest`) | **전부 유지.** green 없이 다음 단계로 안 넘어간다. 품질 바닥이다. |
| **사람 게이트**(사람만 안다) | 단계마다 "OK?" · 검증 3층(옳은 흐름인가·느낌) · 5.5 리디자인 후보 선택 | **건너뛴다.** 그 자리를 아래 두 장치로 메운다. |

**사람을 뺀 자리를 메우는 장치들:**
1. **시작 1회 인테이크(§A-2)** — 사람 접점이 여기 하나뿐이라 *최대한 자세히* 한 번에 받아(왕복 없이) 추측 리스크를 죽인다.
2. **가정 원장(assumption ledger)** — 그 외 에이전트가 사용자 대신 내린 모든 결정을 PRD §11과 STATUS, 그리고 최종 전달 요약에 **명시**한다. 사실처럼 단정하지 않는다 — "내가 이렇게 가정했다"로.
3. **전문가 에이전트(웹검색) — 추측을 근거로 승격:**
   - **시장조사 에이전트(§B-0)** — plan의 세세한 질문에 *웹 근거*로 답한다. 가정을 *근거 있는 결정*으로 올리고 `plan-decisions.md`에 출처와 함께 남긴다.
   - **디자인 트렌드 전문가(§C-6)** — hi-fi 스샷을 *brief 방향 안에서의* 현행 비주얼 기준과 대조해 점수화. 임계값 미달이면 자동 리디자인 1회. 빠진 3층(미감)의 *부분* 프록시이지 완전 대체가 아니다.

> **무인의 한계를 정직하게 안다:** 자동 게이트는 *"동작하나·접근성 통과하나"* 까지만 본다. *"이게 진짜 옳은 제품/흐름인가, 비주얼이 좋게 느껴지나"* 는 보지 못한다. /forge의 hi-fi는 **"기능적으로 완결되고 접근성 통과한 1차 시안"** 이지 사람이 검수·선택을 마친 최종이 아니다. 이 점을 전달 시 분명히 한다(§D).

## A. 시작 — 프로젝트 확정 + 1회 질문 (유일한 사람 접점)

### A-1. 프로젝트 선택
`/plan`·`/design`과 같은 작업장을 공유한다. **현재 폴더 상태에 의존하지 말고** 명시적으로 정한다.
- 사용자가 프로젝트명/아이디어를 줬으면 그걸로 **새 프로젝트** `projects/<name>/`을 만든다(이름이 모호하면 아이디어에서 짧은 kebab-case로 짓고, 그 이름을 질문 묶음에서 함께 확인).
- 기존 프로젝트를 무인으로 *이어서* 돌려달라는 경우엔 그 `STATUS.md`의 현재 단계부터 무인으로 잇는다(아래 B·C는 어느 단계서든 진입 가능).
- `_`로 시작하는 폴더(`_smoke` 등)는 내부 픽스처 — 건드리지 않는다.
- `STATUS.md`는 `/plan` 템플릿(0번 기획 포함)으로 생성하고, **메모에 `모드: /forge (무인)`** 을 박는다.

### A-2. 시작 1회 인테이크 (한 번만 — 대신 **충분히 자세히**)
이게 **유일한 사람 접점**이다. 그래서 가볍게 2~3개만 묻고 마는 게 아니라, **이 한 번에 최대한 길어올린다** — 여기서 받은 1정보가 뒤에서 추측 100개를 막는다. 단, **여러 번 끊어 묻지 않는다(왕복 금지).** 카테고리로 묶은 질문지를 **한 번에** 던지고, 사용자가 답한 만큼만 박고 나머지는 가정으로 닫는다.

`AskUserQuestion`을 쓰되 한 라운드에 카테고리별로 묶어 묻는다(아이디어에서 이미 명확한 항목은 빼고, 모호한 것만). **각 질문엔 "왜 묻는지(이게 뒤 어느 산출물을 가르는지)"를 한 줄 붙여** 사용자가 판단해 답하게 한다. 질문 은행 — 상황에 맞게 고른다:

**① 사용자·문제 (PRD §1·§3의 뿌리)**
- 핵심 사용자는 *구체적으로* 누구이고 어떤 상황·빈도·기기로 쓰나? ("일반 사용자"는 결과를 흐린다 — 직업·맥락까지)
- 지금 그 사람은 이 문제를 *무엇으로* 해결하나(현재 대안), 그게 왜 불편한가?
- 1순위로 풀 **단 하나의 문제**는? (여러 개면 이번 MVP의 메인 하나)

**② 범위·우선순위 (PRD §4·§10 — MVP 경계)**
- 이번에 **반드시 들어갈 기능 3~5개** / **빼도 되는 것** / **명시적으로 범위 밖**.
- "있으면 좋지만 이번엔 아닌 것"을 *미리* 받아야 에이전트가 범위를 안 부풀린다.

**③ 데이터·연동 (PRD §5·§9)**
- 다루는 핵심 데이터(엔터티)와 대략 규모는? 로그인·계정 개념이 있나?
- 외부 연동(결제·지도·알림·소셜로그인 등) 필요 여부 — *있으면 화면·흐름이 바뀐다.*

**④ 플랫폼·뷰포트 (디자인 산출물의 형태를 가른다 — 모바일이 기본 아님)**
- 모바일(390px)? 데스크톱(1280px)? 대시보드? 반응형? 다크모드 필요?

**⑤ 비주얼 방향 (없으면 비주얼을 통째로 추측하게 된다)**
- 톤(미니멀/대담/따뜻함/전문적 …)과 **닮고 싶은 서비스 1~2개**(반례 — 안 닮고 싶은 것도 환영).
- 브랜드 컬러·로고·기존 시스템이 있나? 한국어/영어/다국어?

**⑥ 제약 (PRD §6)**
- 마감·일정, 만들 사람의 스택/역량, 절대 피해야 할 것.

답을 받으면 **답한 것은 전제로 박고**, 안 답한 항목·디테일은 전부 **가정 원장**에 "가정: …(근거)"로 적으며 무인으로 직진한다. **여기서부터 끝까지 OK를 묻지 않는다.** (질문지가 길어도 라운드는 1회 — 추가로 되묻지 않는다.)

> **완전 무인 폴백:** 사용자가 "질문도 알아서 답해"라고 하면 이 단계도 생략하고, ①④⑤의 핵심을 *가정*으로 박아 §11과 전달 요약 맨 위에 크게 올린다.

## B. 무인 기획 (/plan 공정을 자율로)

`plan/SKILL.md`의 7단계·PRD 11섹션·`lint-prd` 게이트를 **그대로** 따르되, **사용자에게 캐묻는 대신 에이전트가 시니어 PM 페르소나로 자문자답한다.** 단 자문자답을 *허공의 추측*으로 두지 않는다 — **시장조사 에이전트가 웹검색으로 근거를 길어와 답을 받친다.**

### B-0. 시장조사 에이전트 (자문자답을 근거화 — 추측을 죽이는 핵심 장치)
PRD를 쓰기 *전에*, 도메인을 웹으로 조사하는 서브에이전트를 띄운다(`WebSearch`/`WebFetch` 사용). 무인의 최대 약점인 *"PRD가 에이전트 추측"* 을 **실제 시장 근거**로 메운다.
- **규모에 맞게 편성** — 보통 1~3개. 축을 나눠 병렬로(예: ⓐ 경쟁/유사 제품과 핵심 기능·차별점 ⓑ 실사용자 불만·리뷰·니즈 ⓒ 도메인 관행·표준·규제/통화·데이터 모델 관례). 단순 도메인이면 1개로 충분.
- **각 에이전트는 `/plan`의 세세한 질문에 답한다** — 7단계 각 질문(목표 지표 수준·MVP 경계·우선순위 근거·데이터 모델·엣지케이스)에 대해 **출처와 함께** 근거 있는 답/권고를 반환한다(구조화 결과 + URL 인용). 추측이면 "추측"이라고 표시.
- **A-2 사용자 답이 최우선** — 조사는 그 *위에서* 빈칸을 채우고 디테일을 벼린다. 조사가 사용자 답과 충돌하면 사용자 답을 따르되, 충돌 사실을 plan-decisions.md에 적는다(사용자가 검수).
- **▶ 산출물 `projects/<name>/plan-decisions.md`** — *"에이전트가 plan 질문에 어떻게 답했나"* 의 기록. 행마다: **질문 → 조사 근거(출처 URL) → 내가 내린 답/권고 → 신뢰도(근거 있음/추측)**. 이게 가정 원장의 *증거판*이다 — §11 가정과 교차참조(근거 확보된 항목은 "가정"에서 "근거 있는 결정"으로 승격).

### B-1. PRD 작성 (조사 근거 위에서)
- **7단계를 머릿속(또는 짧은 작업 노트)으로 통과**한다 — 문제정의→목표·지표→타깃→핵심기능·우선순위(MVP)→4-1 개념 데이터 모델→사용자 흐름·화면→제약→체크리스트. 각 단계에서 *시니어 PM이라면 어떤 질문을 했을지*를 던지고, **A-2 답 + 합리적 기본값**으로 답한다.
- **가정 원장을 채운다.** 사용자에게 못 물은 모든 결정(우선순위 근거·뺀 기능·데이터 모델 선택·엣지케이스 처리)은 PRD **§11 미해결 이슈와 가정**에 "가정: …(근거)" 형태로 적는다. **본문에서 사실처럼 단정 금지** — 가정은 가정 섹션에.
- **PM 규율은 유지한다(자문자답이라도):** "다 중요"로 도망가지 않고 MVP 경계를 *직접 긋는다* / 각 기능에 "해결하는 사용자 문제 / 없으면 어떻게 되나"를 *직접 답한다* / 데이터 관점에서 무리한 기능을 *직접 짚는다*. 무인이라고 범위를 임의로 부풀리지 않는다.
- `projects/<name>/PRD.md`에 11섹션 정식 PRD를 쓴다(개발자가 추가 질문 없이 착수 가능한 깊이).
- **▶ 게이트:** `node scripts/lint-prd.js <name>` → **error 0(green)까지 루프.** 빠진 섹션·깊이 결함을 채우고 다시 돌린다. green 없이 디자인으로 안 넘어간다.
- STATUS `0. 기획` + `→ PRD.md 작성 완료` 체크, `현재 단계: 1-인테이크`로 갱신. **사용자에게 알리되 OK를 기다리지 않고** 바로 C로.

## C. 무인 디자인 (/design 공정을 자율로)

`design/SKILL.md`의 7단계를 **순서대로** 따르되, **단계 사이 사람 OK를 자동 green으로 대체**한다. 자동 검증은 전부 유지, 3층(느낌·판단)만 생략.

### C-1. 인테이크 → `00-flow.md`
PRD를 읽어 `00-flow.md`(§화면 목록·§플로우·§화면별 구성요소·상태·§인터랙션 모델·§범위·§열린 질문)로 정리한다. 톤·뷰포트는 **A-2 답을 그대로** 쓴다(다시 묻지 않는다). 열린 질문은 가정으로 닫고 §열린 질문에 명시. → STATUS 갱신, 바로 다음.

> §화면별 구성요소·상태와 §인터랙션 모델은 **3단계 spec의 출처**다 — 무인이라도 여기서 화면 책임·상태를 못박아야 검증할 게 생긴다. 대충 넘기지 않는다.

### C-2. 와이어프레임 (lo-fi, 클릭+상태변화)
`design/SKILL.md` §2 그대로 — 진짜 `<a href>` 링크 + **실재하는 상태 변화**(인라인 JS 허용) + **죽은 컨트롤 0**(와이어드 또는 `[범위 밖]`) + 인터랙션 모델 캡처. 색·폰트 없음.

### C-3. ▶ 흐름·기능 자동 검증 (사람 3층은 생략)
**1·2층 자동만 돌린다.** `00-flow.md`에서 spec을 파생해 `scripts/lib/`(`crawl.js`·`controls.js`·`selectors.js`)를 `require`하고 프로젝트 고유 기능 assert를 더한다.
- 실행: `node scripts/test-project.js <name> <spec경로>` → **green까지 루프.**
- **실패 원인을 먼저 가른다** — spec 결함(셀렉터 모호·strict-mode "2 elements")이면 셀렉터를 좁히고, 산출물 결함(죽은 컨트롤·상태 안 변함)이면 와이어를 고친다.
- **3층(옳은 흐름인가) 사람 판단은 생략** — 그 자리는 *PRD/00-flow 계약 준수*로 대신한다(spec이 계약에서 파생됐으므로 green = 계약 충족). green이면 STATUS 3번 체크, 바로 다음.

### C-4. 디자인 시스템 (tokens) + a11y 자동
`foundation/tokens.css`를 **표준 2단(primitive→semantic)** 으로 메인이 직접 짠다(브랜드 단어로 토큰명 금지). 색은 A-2 톤/레퍼런스에서 도출.
- **▶ 게이트:** `defineTokenContrastTest(...)`로 semantic 쌍 WCAG AA를 **토큰 확정 즉시** 검사 → 미달이면 *여기서* 토큰 수정(색 다 입힌 뒤가 아니라). green까지.
- `colors.html`·`typography.html`·`components/*.html` 생성.

### C-5. 하이파이 (hi-fi) + 렌더 a11y 자동 — §C 에이전트 편성
와이어의 링크·상태 동작을 유지한 채 살(색·타이포·전환·마이크로 인터랙션)을 입힌다. A-2 뷰포트 안에서. 진입점 `screens/index.html`.
- **화면이 여럿이면 `design/SKILL.md` §C로 편성하되, 가드레일 ①(사용자에게 편성 계획 OK)은 무인이라 생략** — 대신 **편성 결정을 로그로 남긴다**(예: "빌더 4 + 실데이터 1 + 독립 검증 1, 이유: 4화면 중 2개 리스트형"). **가드레일 ②(토큰·스펙·상태동작 공유)와 독립 검증(만든 놈 셀프검증 금지)은 유지.**
- **▶ 게이트:** `defineAxeTest(...)`로 렌더된 각 화면 a11y(대비·역할·포커스) serious+ 0까지.

### C-6. 디자인 트렌드 전문가 (무인의 3층 *비주얼 느낌*을 자동 프록시)
hi-fi가 axe green이 되면, **HTML을 만들지 않은 별도 전문가 에이전트**에게 비주얼 품질을 적대적으로 감정시킨다. 무인 모드에서 빠지는 3층(*좋게 느껴지나*)을 **부분적으로** 자동화하는 장치다. (사람의 미감을 완전히 대체하진 못한다 — 전달 시 그렇게 말한다.)
- **입력:** 렌더 스크린샷(Playwright `screenshot:'on'` 증빙 또는 전용 캡처) + `00-flow.md`/PRD §8 디자인 방향 + `tokens.css`.
- **웹검색으로 현행 기준 대조** — `WebSearch`/`WebFetch`로 **이 brief의 방향**(예: 미니멀 금융앱) 안에서의 현행 UX·비주얼 베스트프랙티스를 조사해 대조한다.
- **⚠ 함정 = off-brief.** 전문가는 *"제일 핫한 트렌드인가"* 가 아니라 ***"이 brief가 택한 방향(00-flow·PRD §8) 안에서 현행 수준에 부합하나"*** 를 본다. 브리프의 의도(예: 의도적 미니멀)를 트렌드 추종으로 배신하지 않는다. 트렌드는 *방향 안에서의 완성도 기준*이지 *방향 교체 사유*가 아니다.
- **루브릭 점수(0~100)** — 위계·여백·타이포 리듬·색 운용·컴포넌트 일관성·상태 표현·모바일 적합성 등. **임계값(기본 80) 미달이면 자동 5.5 리디자인 1회.**
- **자동 5.5 리디자인(임계값 1회):** 전문가의 구체 지적을 받아 디자인 디렉션 에이전트가 후보를 `screens-<variant>/`로 생성(원본 안 덮음). **불변 계약: 화면 목록·흐름·기능 책임. 가드레일: 기능 spec(2층)을 그대로 통과**해야 후보 자격(role·aria·text 셀렉터라 마크업 갈아엎어도 green). **재감정 후 더 높은 점수를 chosen final로.** (사람 선택 대신 *전문가 점수*가 chosen을 정한다 — 무인 프록시. 1회로 제한해 비용·무한루프 차단.)
- **▶ 산출물 `projects/<name>/design-critique.md`** — 화면별 점수·지적·근거(출처 URL)·리디자인 여부·최종 chosen. STATUS의 `후보 / chosen final`에도 기록.

### 정지점 — hi-fi(+전문가 감정)에서 멈춘다
- **5.5 리디자인은 C-6 전문가가 임계값 미달일 때만 자동 1회.** 그 외 추가 발산·후보 선택은 사람 몫(원하면 따로 트리거).
- **6 인터랙션 검증의 자동(2층 회귀 + a11y)은 돌린다**(사람 없이 가능) — 폴리시/리디자인 후 회귀를 잡아 chosen final 신뢰도를 올린다. 단 **6의 사람 느낌·7 handoff는 멈춤.**
- STATUS를 `현재 단계: 5-하이파이 완료 (무인 — 전문가 감정 통과, 사람 검수 대기)`로 갱신.

### 큰 PRD = 묶음(청크)도 무인으로
화면이 많으면 `design/SKILL.md` "묶음 진행 순서"를 따른다 — **①단계(세로)로 자르고(한 묶음을 hi-fi까지 내리지 않음, tokens는 전 묶음 확정 후 1회) ②단계 안에선 묶음마다 게이트.** 무인에선 **묶음 게이트가 "사람 OK"가 아니라 "자동 green(1·2층)"** 이 된다. 묶음 간 링크는 스텁/`[범위 밖]`으로 막고 와이어 배리어에서 1회 연결.

## D. 전달 — 최종 hi-fi + 가정 요약 (사람이 처음 보는 결과)

무인 파이프라인이 끝나면 사용자에게 **간결한 1회 보고**를 한다. 과정을 길게 늘어놓지 않고, 사람이 검수에 바로 쓸 것만:

1. **진입점** — `projects/<name>/<chosen>/index.html`(리디자인 했으면 그 후보, 아니면 `screens/`)을 브라우저로 열어 흐름을 클릭.
2. **에이전트가 사용자 대신 내린 핵심 가정 (검수 우선순위)** — 가정 원장에서 *결과에 영향 큰 것* 위주로 5~8개. *"이 가정이 틀렸으면 여기를 바꾸세요"* 가 되게. (전체는 PRD §11, 근거는 `plan-decisions.md`.)
3. **근거·감정 기록** — `plan-decisions.md`(시장조사가 plan 질문에 어떻게·무슨 근거로 답했나) + `design-critique.md`(트렌드 전문가 점수·지적·리디자인 여부). 검수의 출발점.
4. **자동 검증 증빙** — 통과한 게이트(lint-prd green / Playwright 1·2층 green / a11y 토큰·axe green / 전문가 점수 ≥ 임계값)와 리포트 경로: `npx playwright show-report projects/<name>/playwright-report` (경로에 프로젝트명 필수).
5. **무인의 한계 한 줄** — "동작·접근성·트렌드 부합은 자동 통과했지만 *옳은 제품인가·최종 미감*은 사람 미검수다. 검수 후 ① 가정 수정 → 재생성 ② 추가 리디자인 ③ handoff로 인계 중 택일."

> 터미널 "passed"만 보고 끝내지 않는다 — 항상 리포트·진입점 경로를 줘서 사람이 실제 화면을 눈으로 보게 한다.

## 사람이 다시 들어오는 법 (무인은 일방통행이 아니다)
STATUS와 모든 산출물이 남아 있어, 사용자는 언제든 **수동 모드로 갈아탈 수 있다**:
- `/plan <name>` — PRD의 가정을 사람이 캐물어 다시 다진다(가정 원장이 출발점).
- `/design <name>` — 멈춘 단계(보통 5 하이파이)부터 사람 OK·3층 판단을 넣어 이어간다. 5.5 리디자인·6 느낌검수·7 handoff가 여기서 살아난다.
- 무인 산출물은 **수동 공정의 1차 초안**으로 그대로 호환된다(같은 폴더·STATUS·spec).

## 원칙 (무인 계약 — 어기지 않는다)
- **사람 접점은 시작 1회뿐.** A-2 이후 끝까지 OK를 묻지 않는다. 못 물은 결정은 *가정 원장*에 명시(사실처럼 단정 금지).
- **자동 게이트는 전부 유지** — `lint-prd` · Playwright 1·2층(죽은 컨트롤 0) · a11y(토큰 대비·axe). green 없이 다음 단계·전달로 가지 않는다. 이게 사람 판단을 부분적으로 대신하는 *품질 바닥*이다.
- **사람 게이트·3층(느낌·옳은 흐름)만 생략.** 자동이 못 보는 영역임을 알고, 전달 시 *미검수*라고 정직하게 말한다.
- **추측은 전문가 웹검색으로 근거화한다** — 기획은 시장조사 에이전트(§B-0)가 plan 질문을 *출처 있는 답*으로 받치고(`plan-decisions.md`), 디자인은 트렌드 전문가(§C-6)가 hi-fi를 *brief 방향 안의* 현행 기준으로 감정(`design-critique.md`)한다. **트렌드는 brief 방향 안에서의 완성도 기준 — 방향 교체 사유가 아니다(off-brief 금지).**
- **hi-fi(+전문가 감정)에서 정지.** 자동 5.5 리디자인은 트렌드 전문가 임계값 미달 시 **1회**만(기능 spec 통과 필수). 그 외 추가 발산·후보 선택·6 느낌검수·7 handoff는 사람이 트리거. (6의 자동 회귀는 돌림.)
- **PM·디자인 규율은 무인이라도 유지** — MVP 경계 직접 긋기 / 죽은 컨트롤 0 / 토큰 2단 / 빌더에 토큰·스펙·상태동작 공유 + 독립 검증(셀프검증 금지). 무인이라고 품질 루프(thin·bad·variantsIdentical·off-brief·deadControl·stateInert)를 건너뛰지 않는다.
- **공정 절차는 베끼지 않는다** — 단계별 상세는 `plan/SKILL.md`·`design/SKILL.md`가 원본. 이 문서는 *"사람을 어디서 빼고 무엇으로 메우나"* 만 정의한다.
- 산출물은 외부 의존성 없는 독립 HTML(상태용 인라인 JS 허용), A-2에서 정한 뷰포트, 첫 줄 `@dsCard` 마커 — `CLAUDE.md` 형식 규칙 그대로.
