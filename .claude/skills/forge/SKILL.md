# /forge — 무인 파이프라인 오케스트레이터 (아이디어 → hi-fi)

`/plan`(기획)과 `/design`(디자인)을 **사람 게이트 없이** 이어 붙여, 아이디어 한 줄에서 **최종 hi-fi**까지 자율로 내려가는 harness다.
`/plan`·`/design`을 **대체하지 않는다 — 그 공정을 무인 모드로 구동하는 상류 오케스트레이터**다. 단계별 *절차*의 원본은 여전히 두 SKILL이고, 이 문서는 *"언제 사람을 빼고, 무엇으로 그 자리를 메우나"* 의 원본이다.

```
/forge  (시작 1회 질문)  →  [Workflow: forge-plan] 시장조사(축 분할 근거화) → PRD 초안
                                → N명 prd-critic 다측면 비평(다수결) → (미달 시 개정 1회)
                                → PRD.md(lint-prd + lint-prd-review green) + plan-decisions.md + prd-review.md
                          →  [무인] /design 공정 → 와이어(Playwright green) → tokens(a11y green)
                          →  [Workflow: forge-design] (빌드 전)비주얼 레퍼런스 브리프 → 빌드 fan-out(anti-wireframe)
                                → 렌더 스크린샷(PNG) → 화면당 N명 적대적 검증(렌더 기반·다수결, wireframey 포함)
                                → 트렌드 점수(렌더 기반) → (미달 시 리디자인 1회) → design-verify.md·design-critique.md
                          →  [무인] lint-verify green + axe/2층 회귀 green → handoff 패키지 → lint-handoff green
                          →  최종 인계 패키지 + 가정·근거·감정 요약 전달
```

형식·작업장 규칙의 원본은 `CLAUDE.md`, 공정 절차의 원본은 `plan/SKILL.md`·`design/SKILL.md`다. 충돌 시 그쪽을 따른다.
이 스킬은 **범용**이다 — 웹·앱·대시보드·내부툴 무엇이든 같은 무인 공정.

## 0. 무인 계약 — 무엇을 빼고, 무엇으로 메우나 (이 스킬의 존재 이유)

`/plan`·`/design`은 **두 종류의 게이트**를 가진다. /forge는 둘을 **다르게** 다룬다:

| 게이트 종류 | 예 | /forge에서 |
|---|---|---|
| **자동 게이트**(스크립트가 본다) | `lint-prd`(형식) · `lint-prd-review`(PRD 실질 비평) · Playwright 1·2층(내비·기능 완결성·죽은 컨트롤 0) · a11y(`defineTokenContrastTest`·`defineAxeTest`) · `lint-verify`(독립 검증 산출물) · `lint-handoff` | **전부 유지.** green 없이 다음 단계로 안 넘어간다. 품질 바닥이다. |
| **사람 게이트**(사람만 안다) | 단계마다 "OK?" · 검증 3층(옳은 흐름인가·느낌) · 5.5 리디자인 후보 선택 | **건너뛴다.** 그 자리를 아래 두 장치로 메운다. |

**사람을 뺀 자리를 메우는 장치들:**
1. **시작 1회 인테이크(§A-2)** — 사람 접점이 여기 하나뿐이라 *최대한 자세히* 한 번에 받아(왕복 없이) 추측 리스크를 죽인다.
2. **가정 원장(assumption ledger)** — 그 외 에이전트가 사용자 대신 내린 모든 결정을 PRD §11과 STATUS, 그리고 최종 전달 요약에 **명시**한다. 사실처럼 단정하지 않는다 — "내가 이렇게 가정했다"로.
3. **전문가/검증 에이전트 — 추측을 근거로, 셀프검증을 독립 검증으로:** (`.claude/agents/`에 정의, 무인 기획·디자인 구간은 `forge-plan`·`forge-design` Workflow가 코드로 엮는다)
   - **시장조사 `market-researcher`(§B)** — plan의 세세한 질문에 *웹 근거*로 답한다. 가정을 *근거 있는 결정*으로 올리고 `plan-decisions.md`에 출처와 함께 남긴다.
   - **PRD 비평가 `prd-critic`(§B)** — PRD를 *작성자가 아닌 별도 컨텍스트*가 적대적으로 비평. 형식이 아니라 실질(측정가능·정당화·정합·일관성·가정노출·범위규율(부풀림)·표준 화면 골격 완결성(빈약함) 8차원)을 본다. workflow에선 N명 다수결이라 셀프검증 불가능. `prd-review.md` → `lint-prd-review` 게이트.
   - **렌더러 `screen-shooter`(§C-5)** — 빌드 직후 hi-fi를 헤드리스 크로미움으로 *실제 렌더*해 PNG로 떨군다(`scripts/shoot.js`). 검증·트렌드가 *소스가 아니라 픽셀*을 보게 하는 픽셀맹 해소 장치.
   - **독립 검증자 `design-verifier`(§C-5)** — hi-fi를 *빌더가 아닌 별도 컨텍스트*가 **렌더 PNG 기준으로** 적대적 render-check(7항목 — `wireframey`="와이어 + 색" 수준 포함). workflow에선 화면당 N명 다수결이라 셀프검증이 구조적으로 불가능. `design-verify.md` → `lint-verify` 게이트.
   - **디자인 트렌드 전문가 `design-trend-expert`(§C-5·6)** — 두 모드: (빌드 전)현행 비주얼 기준을 근거화해 빌더에 줄 **레퍼런스 브리프**, (빌드 후)hi-fi를 *brief 방향 안에서* **렌더 PNG 기준으로** 점수화. 임계값 미달이면 자동 리디자인 1회. 빠진 3층(미감)의 *부분* 프록시이지 완전 대체가 아니다.

**안정성 계약 (무인이라 사람이 못 멈춘다 — 멈춤·은폐를 코드/규칙으로 막는다):**
- **게이트 재시도 상한 = 2회.** 자동 게이트(lint·Playwright·a11y)가 red면 workflow 결과의 지적을 반영해 **최대 2회**까지 재실행/보정한다. **그래도 red면 더 돌리지 말고 멈춘다** — STATUS와 §D 전달에 `미통과 게이트: <이름> (N회 시도) — 사람 개입 필요`로 보고하고 그 단계에서 정지한다. *만족 불가능한 게이트에 무한 재실행(토큰 폭주·행)하지 않는다.* 이게 무인 최대 리스크다.
- **조용한 저하(degraded)는 반드시 표면화한다.** workflow는 부품 실패(시장조사 전멸·렌더 실패·검증자 사망·빌드 실패)에 견고하게 *degrade* 하고 그 목록을 `degraded`로 반환한다. 사람은 과정 로그를 안 보므로 **메인은 그 `degraded`를 §D 전달의 "⚠ 자동 저하 항목"에 그대로 노출**한다(은폐 금지). 비면 "없음".

> **무인의 한계를 정직하게 안다:** 자동 게이트는 *"동작하나·접근성 통과하나"* 까지만 본다. *"이게 진짜 옳은 제품/흐름인가, 비주얼이 좋게 느껴지나"* 는 보지 못한다. /forge의 hi-fi는 **"기능적으로 완결되고 접근성 통과한 1차 시안"** 이지 사람이 검수·선택을 마친 최종이 아니다. 이 점을 전달 시 분명히 한다(§D).

## A. 시작 — 프로젝트 확정 + 1회 질문 (유일한 사람 접점)

### A-1. 프로젝트 선택
`/plan`·`/design`과 같은 작업장을 공유한다. **현재 폴더 상태에 의존하지 말고** 명시적으로 정한다.
- 사용자가 프로젝트명/아이디어를 줬으면 그걸로 **새 프로젝트** `projects/<name>/`을 만든다(이름이 모호하면 아이디어에서 짧은 kebab-case로 짓고, 그 이름을 질문 묶음에서 함께 확인).
- 기존 프로젝트를 무인으로 *이어서* 돌려달라는 경우엔 그 `STATUS.md`의 현재 단계부터 무인으로 잇는다(아래 B·C는 어느 단계서든 진입 가능).
- `_`로 시작하는 폴더(`_smoke` 등)는 내부 픽스처 — 건드리지 않는다.
- `STATUS.md`는 `/plan` 템플릿(0번 기획 포함)으로 생성하고, **메모에 `모드: /forge (무인)`** 을 박는다.

### A-2. 시작 1회 인테이크 (한 번만 — 대신 **충분히 자세히**)
이게 **유일한 사람 접점**이다. 그래서 가볍게 2~3개만 묻고 마는 게 아니라, **이 한 번에 최대한 길어올린다** — 여기서 받은 1정보가 뒤에서 추측 100개를 막는다. 단, **여러 번 끊어 묻지 않는다(왕복 금지).** 카테고리로 묶은 질문지를 **한 번에** 던지고, 사용자가 답한 만큼만 박고 나머지는 가정으로 닫는다.

`AskUserQuestion`을 쓰되 한 라운드에 카테고리별로 묶어 묻는다(아이디어에서 이미 명확한 항목은 빼고, 모호한 것만). **각 질문엔 "왜 묻는지(이게 뒤 어느 산출물을 가르는지)"를 한 줄 붙여** 사용자가 판단해 답하게 한다. 질문 은행 — 상황에 맞게 고른다:

**① 사용자·문제 (PRD §1·§3의 뿌리)**
- 핵심 사용자는 *구체적으로* 누구이고 어떤 상황·빈도·기기로 쓰나? ("일반 사용자"는 결과를 흐린다 — 직업·맥락까지)
- 지금 그 사람은 이 문제를 *무엇으로* 해결하나(현재 대안), 그게 왜 불편한가?
- 1순위로 풀 **단 하나의 문제**는? (여러 개면 이번 MVP의 메인 하나)

**② 범위·우선순위 (PRD §4·§10 — MVP 경계)**
- 이번에 **반드시 들어갈 기능 3~5개** / **빼도 되는 것** / **명시적으로 범위 밖**.
- "있으면 좋지만 이번엔 아닌 것"을 *미리* 받아야 에이전트가 범위를 안 부풀린다.

**③ 데이터·연동 (PRD §5·§9)**
- 다루는 핵심 데이터(엔터티)와 대략 규모는? 로그인·계정 개념이 있나?
- 외부 연동(결제·지도·알림·소셜로그인 등) 필요 여부 — *있으면 화면·흐름이 바뀐다.*

**④ 플랫폼·뷰포트 (디자인 산출물의 형태를 가른다 — 모바일이 기본 아님)**
- 모바일(390px)? 데스크톱(1280px)? 대시보드? 반응형? 다크모드 필요?

**⑤ 비주얼 방향 (없으면 비주얼을 통째로 추측하게 된다)**
- 톤(미니멀/대담/따뜻함/전문적 …)과 **닮고 싶은 서비스 1~2개**(반례 — 안 닮고 싶은 것도 환영).
- 브랜드 컬러·로고·기존 시스템이 있나? 한국어/영어/다국어?

**⑥ 제약 (PRD §6)**
- 마감·일정, 만들 사람의 스택/역량, 절대 피해야 할 것.

**⑦ 진입·구조 / 표준 화면 골격 (PRD §7·§10 — 화면 표면이 빈약해지는 걸 막는다)**
- 이 제품의 **아키타입**(SaaS 웹앱·모바일 소비자앱·내부툴·대시보드·마켓플레이스·콘텐츠사이트 등)은? → 그 유형의 **table-stakes 화면**을 default-IN으로 깐다.
- 진입 경험에 **로그인/온보딩**이 있나? **홈/대시보드**(핵심 툴과 별개의 착지점)는? **설정/프로필·멤버/팀 관리**는? — 각각 *이번 MVP에 IN/OUT*. (왜 묻나: 이걸 안 받으면 "핵심 툴 한 화면"만 그려져 서비스로 안 보인다 — `collab-kanban`이 정확히 이 누락이었다.)
- ⚠ "인증/실시간 *백엔드*는 범위 밖"과 "로그인·진입 *화면 UI*를 안 그림"은 다르다. 백엔드만 범위 밖이고, **디자인 산출물이므로 진입 화면은 그린다.**

답을 받으면 **답한 것은 전제로 박고**, 안 답한 항목·디테일은 전부 **가정 원장**에 "가정: …(근거)"로 적으며 무인으로 직진한다. **특히 ⑦ 표준 골격은 안 받았어도 아키타입에서 *default-IN으로 가정*해 §7에 깔고, "이 화면들을 포함 가정함(아키타입 table-stakes)"을 §11·전달 요약에 명시한다 — 빠뜨리는 게 아니라 "포함했다"를 가정으로 노출한다.** **여기서부터 끝까지 OK를 묻지 않는다.** (질문지가 길어도 라운드는 1회 — 추가로 되묻지 않는다.)

> **완전 무인 폴백:** 사용자가 "질문도 알아서 답해"라고 하면 이 단계도 생략하고, ①④⑤의 핵심을 *가정*으로 박아 §11과 전달 요약 맨 위에 크게 올린다.

## B. 무인 기획 (/plan 공정을 자율로) — **`forge-plan` Workflow로 코드 강제**

무인 기획의 *근거화·작성·비평*은 산문 지시가 아니라 **`Workflow` 도구로 `forge-plan` 스크립트(`.claude/workflows/forge-plan.js`)를 띄워** 돌린다. 시장조사(축 분할 병렬 근거화) → PRD 초안 작성(PM 페르소나) → **N명(기본 3) prd-critic 다측면 적대적 비평(다수결)** → 미달 시 개정 1회 → `PRD.md`·`plan-decisions.md`·`prd-review.md` 기록까지 *코드가* **근거화·셀프검증 금지·비평 누락 0**을 강제한다. (작성자↔비평가가 구조적 별도 프로세스라 **셀프검증 불가능** — "PRD가 에이전트 추측"이라는 무인 최대 약점이 *근거(시장조사) + 적대적 비평*으로 두 겹으로 닫힌다.)

- **호출:** `Workflow`(name: `forge-plan`)에 args를 넘긴다 — `{ project, idea, intake, researchAxes:[…], critics:3, paths:{ prd, status } }`. `idea`·`intake`는 A-1/A-2에서 받은 것을 그대로. 단순 도메인이면 `researchAxes`를 1개로 줄여도 된다.
- **사람을 뺀 자리:** 사용자에게 캐묻는 대신 시장조사가 *근거*를 대고, PM 페르소나가 7단계·11섹션·가정 원장 규율로 초안을 쓰고(미검증은 본문 단정 금지 → §11 가정), prd-critic이 *작성자가 아닌* 컨텍스트에서 8차원을 적대적으로 본다. PM 규율("다 중요" 거절·기능 정당화·데이터 관점·범위 안 부풀림 + **표준 화면 골격 누락 방지** — 부풀림 거절 ≠ 기본기 누락 허용)은 코드 경로에서도 유지.
- **A-2 사용자 답이 최우선** — 시장조사는 그 *위에서* 빈칸만 채운다. 충돌하면 사용자 답을 따르고 충돌 사실을 `plan-decisions.md`에 남긴다.
- **▶ 반환 후 게이트(메인이 인라인):** workflow가 끝나면 ① `node scripts/lint-prd.js <name>`(형식 — 11섹션·깊이) ② `node scripts/lint-prd-review.js <name>`(실질 — 독립 비평 산출물 green) 둘 다 green을 확인한다. red면 workflow 결과의 지적을 반영해 재실행하거나 해당 섹션만 보정 — **단 재시도 상한 2회(§0 안정성 계약). 그래도 red면 멈추고 §D에 '미통과 게이트 — 사람 개입 필요'로 보고.** green 없이 C(디자인)로 안 넘어간다. **workflow가 반환한 `degraded`는 §D에 노출한다.**
- STATUS `0. 기획` + `→ PRD.md 작성 완료` 체크, `현재 단계: 1-인테이크`로 갱신. **사용자에게 알리되 OK를 기다리지 않고** 바로 C로.

> **수동 `/plan`과의 관계:** 대화형 `/plan`에선 7단계 질문을 *사람이* 몰고(질문의 질이 핵심), 비평만 PRD 완료 직전 인라인 단일 `prd-critic`으로 띄운다. workflow는 *무인(forge)* 전용 — 같은 부품(`market-researcher`·`prd-critic`)을 코드로 엮어 다수결·개정 루프를 강제할 뿐이다.

## C. 무인 디자인 (/design 공정을 자율로)

`design/SKILL.md`의 7단계를 **순서대로** 따르되, **단계 사이 사람 OK를 자동 green으로 대체**한다. 자동 검증은 전부 유지, 3층(느낌·판단)만 생략.

### C-1. 인테이크 → `00-flow.md`
PRD를 읽어 `00-flow.md`(§화면 목록·§플로우·§화면별 구성요소·상태·§인터랙션 모델·§범위·§열린 질문)로 정리한다. 톤·뷰포트는 **A-2 답을 그대로** 쓴다(다시 묻지 않는다). 열린 질문은 가정으로 닫고 §열린 질문에 명시. → STATUS 갱신, 바로 다음.

> §화면별 구성요소·상태와 §인터랙션 모델은 **3단계 spec의 출처**다 — 무인이라도 여기서 화면 책임·상태를 못박아야 검증할 게 생긴다. 대충 넘기지 않는다.

### C-2. 와이어프레임 (lo-fi, 클릭+상태변화)
`design/SKILL.md` §2 그대로 — 진짜 `<a href>` 링크 + **실재하는 상태 변화**(인라인 JS 허용) + **죽은 컨트롤 0**(와이어드 또는 `[범위 밖]`) + 인터랙션 모델 캡처. 색·폰트 없음.

### C-3. ▶ 흐름·기능 자동 검증 (사람 3층은 생략)
**1·2층 자동만 돌린다.** `00-flow.md`에서 spec을 파생해 `scripts/lib/`(`crawl.js`·`controls.js`·`selectors.js`)를 `require`하고 프로젝트 고유 기능 assert를 더한다.
- 실행: `node scripts/test-project.js <name> <spec경로>` → **green까지 루프(단, 산출물 수정 재시도 최대 2회 — §0 안정성 계약). 2회 후에도 red면 멈추고 §D에 '미통과 게이트 — 사람 개입 필요'로 보고.**
- **실패 원인을 먼저 가른다** — spec 결함(셀렉터 모호·strict-mode "2 elements")이면 셀렉터를 좁히고(이건 횟수에 안 셈 — 산출물 버그 아님), 산출물 결함(죽은 컨트롤·상태 안 변함)이면 와이어를 고친다.
- **3층(옳은 흐름인가) 사람 판단은 생략** — 그 자리는 *PRD/00-flow 계약 준수*로 대신한다(spec이 계약에서 파생됐으므로 green = 계약 충족). green이면 STATUS 3번 체크, 바로 다음.

### C-4. 디자인 시스템 (tokens) + a11y 자동
`foundation/tokens.css`를 **표준 2단(primitive→semantic)** 으로 메인이 직접 짠다(브랜드 단어로 토큰명 금지). 색은 A-2 톤/레퍼런스에서 도출.
- **▶ 게이트:** `defineTokenContrastTest(...)`로 semantic 쌍 WCAG AA를 **토큰 확정 즉시** 검사 → 미달이면 *여기서* 토큰 수정(색 다 입힌 뒤가 아니라). green까지.
- `colors.html`·`typography.html`·`components/*.html` 생성.

### C-5 ~ C-6. 하이파이 + 적대적 검증 + 트렌드 감정 — **`forge-design` Workflow로 코드 강제**
무인의 *생성·검증 핵심*은 산문 지시가 아니라 **`Workflow` 도구로 `forge-design` 스크립트(`.claude/workflows/forge-design.js`)를 띄워** 돌린다. **(빌드 전)비주얼 레퍼런스 브리프** → 빌드 fan-out(anti-wireframe 바·브리프 입력) → **렌더 스크린샷(`screen-shooter` → PNG)** → **화면당 N명(기본 3) 적대적 검증(렌더 기반·다수결)** → 트렌드 점수(렌더 기반) → 임계값(기본 80) 미달 시 발산 리디자인 1회 → `design-verify.md`·`design-critique.md` 기록·STATUS chosen 표시까지 *코드가* **순서·독립성·검증 누락 0·픽셀맹 0**을 강제한다. (빌더↔검증자가 구조적 별도 프로세스라 **셀프검증이 불가능**; 검증·트렌드가 *소스가 아니라 렌더 PNG*를 보므로 "와이어풍"이 무인에서도 잡힌다 — 첫 칸반이 와이어 수준으로 샌 구멍이 닫힌다.)

- **선행(메인이 인라인):** C-4 토큰 확정(전 묶음 공유, a11y 대비 green) + 와이어/2층 spec이 있어야 한다. workflow는 *생성+감정*만 맡고, 토큰·Playwright 게이트는 메인이 건다(토큰은 fan-out 전에 1회 확정 — 묶음 규칙). **▶ 렌더 선행 점검(F-S4):** `node scripts/shoot.js _smoke screens 1280x900`(또는 대상 1화면)으로 헤드리스 렌더가 도는지 확인한다 — 실패하면 Playwright/브라우저 미설치(`npx playwright install chromium`)다. 점검 없이 돌리면 렌더가 조용히 실패해 **픽셀맹으로 회귀**한다(그땐 workflow `degraded`/`pixelChecked:false`로 잡혀 §D에 뜬다).
- **호출:** `Workflow`(name: `forge-design`)에 args를 넘긴다 — `{ project, screens:[{name, spec}], paths:{ tokensCss, flow, prd, screensDir, wireDir }, viewport, trendThreshold:80, verifiers:3 }`. `screens`·`spec`은 `00-flow.md` §화면별 구성요소·상태에서 뽑는다. (Workflow는 백그라운드로 돌고 완료 시 결과를 반환한다.)
- **사람 게이트 대체:** 편성 계획 OK(가드레일 ①)는 생략하되 workflow가 phase 로그를 남긴다. 가드레일 ②(토큰·스펙 공유)·독립 검증(셀프검증 불가)·5.5 리디자인 선택(= *트렌드 점수 비교*가 chosen 결정)은 코드로 유지. 자동 리디자인은 **1회**로 제한(비용·무한루프 차단). 트렌드 전문가(`design-trend-expert`)는 **brief 방향 안의 완성도만** 본다 — off-brief(방향 교체 권고) 금지.
- **▶ 픽셀맹 해소(D1) — 검증·트렌드는 *렌더 PNG*를 본다.** workflow가 빌드 직후 `screen-shooter`(Bash+Playwright)로 각 화면을 헤드리스 렌더해 `_shots/`에 PNG를 떨구고, `design-verifier`·`design-trend-expert`가 그 PNG를 Read(이미지)로 열어 *소스가 아니라 픽셀*을 판정한다. render-check에 **`wireframey`**("와이어 + 색" 수준)가 추가돼, 깨지지도 빈약하지도 않은데 hi-fi가 아닌 화면을 잡는다. 빌드 전엔 `design-trend-expert` 레퍼런스 모드가 현행 기준을 근거화해 빌더에 먹인다(추측 빌드 방지 — plan market-researcher의 디자인판).
- **▶ 반환 후 게이트(메인이 인라인):** workflow가 끝나면 산출물에 대해 ① `node scripts/lint-verify.js <name>`(독립 검증 산출물 green — workflow가 쓴 `design-verify.md`를 *스크립트가* 재확인) ② `defineAxeTest`/2층 회귀 Playwright(chosen final 렌더 a11y serious+ 0 · 동작 회귀 green)를 돌린다. red면 workflow 결과의 지적을 반영해 재실행하거나 해당 화면만 보정 — **단 재시도 상한 2회(§0). 그래도 red면 멈추고 §D에 '미통과 게이트 — 사람 개입 필요'로 보고.** (6 인터랙션 검증의 *자동*은 여기서 돈다 — 6의 *사람 느낌*만 멈춘다.)
- **▶ workflow `degraded`·`belowThreshold`·`pixelChecked`를 §D에 반영(F-S2/S3/S4):** 반환의 `degraded[]`(시장조사/렌더/검증/빌드 저하)는 §D "⚠ 자동 저하 항목"에 그대로, `belowThreshold:true`(리디자인 후에도 트렌드<임계값)·`pixelChecked:false`(렌더 실패 → 소스 폴백)는 §D 맨 위에 크게 — 무인이라 사람이 로그를 안 본다.
- **점수가 chosen을 정한다 = 무인 프록시이지 사람 미감의 완전 대체가 아니다** — §D 전달 시 명시한다.

> **수동 모드와의 관계:** `/design`(대화형·사람 게이트)에선 이 구간을 workflow로 돌리지 않고 §C step 5의 인라인 단일 검증자로 간다(단계마다 사람 OK가 필요하므로). workflow는 *무인(forge)* 전용 — 같은 부품(`screen-builder`·`design-verifier`·`design-trend-expert`)을 코드로 엮을 뿐이다.

### C-7. handoff — 인계 패키지 + hand-off.md (무인의 *최종 산출물*)
**이게 atelier의 존재 이유다 — hi-fi는 중간 산출물이고, 받는 쪽 Claude Code에 넘길 *인계 패키지*가 진짜 최종이다.** handoff는 *사람 판단이 필요 없는 패키징*이고 **`lint-handoff.js`라는 자동 게이트**가 있어 무인으로 돌린다. `design/SKILL.md §7`을 그대로 따르되 chosen final을 소비한다.
- `projects/<name>/handoff/`에 4종을 묶는다: ① **비주얼 문서 `index.html`**(chosen 화면 전부 임베드 + 흐름맵 + 화면별 스펙) ② **토큰 매핑표 `token-mapping.md`**(semantic 토큰 → 대상 토큰, primitive 제외) ③ **컴포넌트 인벤토리 `component-inventory.md`**(variant/state + 쓰이는 화면) ④ ★ **인계 프롬프트 `hand-off.md`**(받는 쪽에 그대로 붙여넣는 브리프 — 먼저 읽을 것·지킬 계약·만들 순서). PRD가 있으면 `hand-off.md`의 "먼저 읽을 것"에서 `PRD.md` 경로로 참조.
- **무인 특이점:** 대상 프로젝트가 미정이라 `hand-off.md`엔 `<대상>` 같은 **placeholder 토큰을 쓰지 않는다** — "대상 프로젝트"를 일반 텍스트로 쓰고, 경로는 atelier 저장소 기준 실경로(ASCII)로 채운다. (lint-handoff가 안 채운 placeholder를 에러로 잡는다.)
- **▶ 게이트:** `node scripts/lint-handoff.js <name>` → **error 0(green)까지.** 필수파일·placeholder 잔존·chosen 화면 임베드 누락·semantic 토큰 누락·컴포넌트 누락을 본다.

### 정지점 — handoff(인계 패키지)에서 멈춘다
- **무인의 최종 산출물 = `handoff/` 패키지 + `hand-off.md`.** 여기까지 자동으로 만든다. (5.5 리디자인은 C-6 임계값 미달 시 자동 1회 — 그 외 *추가* 발산·후보 재선택, 6의 *사람 느낌* 검수만 사람 몫.)
- STATUS를 `현재 단계: 7-handoff 완료 (무인 — 사람 검수/실인계 대기)`로 갱신.

### 큰 PRD = 묶음(청크)도 무인으로
화면이 많으면 `design/SKILL.md` "묶음 진행 순서"를 따른다 — **①단계(세로)로 자르고(한 묶음을 hi-fi까지 내리지 않음, tokens는 전 묶음 확정 후 1회) ②단계 안에선 묶음마다 게이트.** 무인에선 **묶음 게이트가 "사람 OK"가 아니라 "자동 green(1·2층)"** 이 된다. 묶음 간 링크는 스텁/`[범위 밖]`으로 막고 와이어 배리어에서 1회 연결.

## D. 전달 — 최종 인계 패키지 + 가정 요약 (사람이 처음 보는 결과)

무인 파이프라인이 끝나면 사용자에게 **간결한 1회 보고**를 한다. 과정을 길게 늘어놓지 않고, 사람이 검수에 바로 쓸 것만:

0. **⚠ 자동 저하·미통과 (있으면 맨 위에 — 무인이라 사람이 로그를 안 본다, F-S1~S4).** workflow가 반환한 `degraded[]`를 *그대로* 나열한다(시장조사 전멸·렌더 실패→픽셀맹·검증자 사망·빌드 실패 등). 추가로: **재시도 상한(2회) 후에도 미통과한 게이트**(`미통과 게이트: <이름> — 사람 개입 필요`), **`belowThreshold`**(리디자인 후에도 트렌드<임계값 → "비주얼 미흡, 사람 검수 권장"), **`pixelChecked:false`**(렌더 실패 → 소스로만 검증돼 wireframey 놓쳤을 수 있음)를 크게 적는다. **저하가 하나도 없으면 "자동 저하 없음 — 전 게이트 정상 통과"라고 명시**(침묵≠통과). 상세는 `design-critique.md`/STATUS 메모.
1. **최종 산출물 = 인계 패키지** — `projects/<name>/handoff/index.html`(chosen 화면 임베드 + 흐름맵 + 스펙)을 열어 훑고, **`handoff/hand-off.md`** 가 대상 Claude Code에 붙여넣을 브리프다. 화면만 보려면 `projects/<name>/<chosen>/index.html`.
2. **에이전트가 사용자 대신 내린 핵심 가정 (검수 우선순위)** — 가정 원장에서 *결과에 영향 큰 것* 위주로 5~8개. *"이 가정이 틀렸으면 여기를 바꾸세요"* 가 되게. (전체는 PRD §11, 근거는 `plan-decisions.md`.)
3. **근거·감정 기록** — `plan-decisions.md`(시장조사가 plan 질문에 어떻게·무슨 근거로 답했나) + `design-critique.md`(트렌드 전문가 점수·지적·리디자인 여부). 검수의 출발점.
4. **자동 검증 증빙** — 통과한 게이트(lint-prd green / Playwright 1·2층 green / a11y 토큰·axe green / 전문가 점수 ≥ 임계값)와 리포트 경로: `npx playwright show-report projects/<name>/playwright-report` (경로에 프로젝트명 필수).
5. **무인의 한계 한 줄** — "동작·접근성·트렌드 부합·인계 완결성은 자동 통과했지만 *옳은 제품인가·최종 미감*은 사람 미검수다. 검수 후 ① 가정 수정 → 재생성 ② 추가 리디자인 ③ `hand-off.md`를 대상 프로젝트에 붙여넣어 구현 중 택일."

> 터미널 "passed"만 보고 끝내지 않는다 — 항상 리포트·진입점 경로를 줘서 사람이 실제 화면을 눈으로 보게 한다.

## 사람이 다시 들어오는 법 (무인은 일방통행이 아니다)
STATUS와 모든 산출물이 남아 있어, 사용자는 언제든 **수동 모드로 갈아탈 수 있다**:
- `/plan <name>` — PRD의 가정을 사람이 캐물어 다시 다진다(가정 원장이 출발점).
- `/design <name>` — 멈춘 단계부터 사람 OK·3층 판단을 넣어 이어간다. 추가 5.5 발산·6 느낌검수·재-handoff가 여기서 살아난다.
- 무인 산출물은 **수동 공정의 1차 초안**으로 그대로 호환된다(같은 폴더·STATUS·spec).

## 원칙 (무인 계약 — 어기지 않는다)
- **사람 접점은 시작 1회뿐.** A-2 이후 끝까지 OK를 묻지 않는다. 못 물은 결정은 *가정 원장*에 명시(사실처럼 단정 금지).
- **자동 게이트는 전부 유지** — `lint-prd`(형식) · `lint-prd-review`(PRD 실질 비평) · Playwright 1·2층(죽은 컨트롤 0) · a11y(토큰 대비·axe) · `lint-verify`(하이파이 독립 검증) · `lint-handoff`. green 없이 다음 단계·전달로 가지 않는다. 이게 사람 판단을 부분적으로 대신하는 *품질 바닥*이다.
- **사람 게이트·3층(느낌·옳은 흐름)만 생략.** 자동이 못 보는 영역임을 알고, 전달 시 *미검수*라고 정직하게 말한다.
- **안정성: 멈추지 말고 막히지 말고 숨기지 마라(§0 안정성 계약).** ① 게이트 red는 **최대 2회** 재시도 후 멈추고 §D에 '사람 개입 필요'로 보고(무한 재실행·토큰 폭주 금지). ② 부품 실패는 견고히 degrade하되 workflow가 `degraded[]`로 반환하고 메인이 **§D 맨 위에 노출**(조용한 품질 저하 은폐 금지 — 침묵≠통과). ③ 렌더(hi-fi 픽셀 검증)는 선행 점검하고 실패 시 `pixelChecked:false`로 §D에 띄운다(픽셀맹 회귀 방지).
- **추측은 근거화하고, 산출물은 작성자 아닌 컨텍스트가 적대적으로 본다** — 기획은 `market-researcher`가 plan 질문을 *출처 있는 답*으로 받치고(`plan-decisions.md`) `prd-critic`이 PRD 실질을 8차원으로 비평(`prd-review.md` — 범위 부풀림(scoped)+표준 화면 골격 빈약(surfaceComplete) 양방향), 디자인은 `screen-shooter`가 hi-fi를 *렌더(PNG)*하고 `design-verifier`가 그 픽셀로 render-check(7항목 — `wireframey` 포함, `design-verify.md`)하며 `design-trend-expert`가 빌드 전 레퍼런스 브리프 + 빌드 후 *brief 방향 안의* 현행 기준으로 렌더 기반 감정(`design-critique.md`)한다. 무인 구간은 `forge-plan`·`forge-design` Workflow가 다수결·개정/리디자인 루프를 코드로 강제(셀프검증 불가). **트렌드는 brief 방향 안에서의 완성도 기준 — 방향 교체 사유가 아니다(off-brief 금지).**
- **handoff(인계 패키지)에서 정지 — 그게 atelier의 최종 산출물이다.** 7 handoff는 자동(`lint-handoff` 게이트)으로 `hand-off.md`까지 만든다. 자동 5.5 리디자인은 트렌드 전문가 임계값 미달 시 **1회**만(기능 spec 통과 필수). 그 외 *추가* 발산·후보 재선택·6 느낌검수만 사람이 트리거. (6의 자동 회귀는 돌림.)
- **PM·디자인 규율은 무인이라도 유지** — MVP 경계 직접 긋기 / 죽은 컨트롤 0 / 토큰 2단 / 빌더에 토큰·스펙·상태동작 공유 + 독립 검증(셀프검증 금지). 무인이라고 품질 루프(thin·bad·variantsIdentical·off-brief·deadControl·stateInert)를 건너뛰지 않는다.
- **공정 절차는 베끼지 않는다** — 단계별 상세는 `plan/SKILL.md`·`design/SKILL.md`가 원본. 이 문서는 *"사람을 어디서 빼고 무엇으로 메우나"* 만 정의한다.
- 산출물은 외부 의존성 없는 독립 HTML(상태용 인라인 JS 허용), A-2에서 정한 뷰포트, 첫 줄 `@dsCard` 마커 — `CLAUDE.md` 형식 규칙 그대로.
