# 🎨 Atelier

> 터미널 안의 아틀리에 — 아이디어 한 줄에서 개발에 넘길 디자인까지, 손으로 짓는다.

**Atelier**(아틀리에)는 **아이디어부터 개발에 넘길 디자인까지**를 터미널에서 짓는 작업장이다.
아이디어를 캐물어 **PRD**로 굳히고 → 그 PRD에서 와이어프레임 → 디자인 시스템 → 하이파이까지 단계별로 짓고 →
완성본을 *인계 패키지*로 묶어 다른 Claude Code 프로젝트(앱 레포)에 넘긴다.

여기서 만드는 건 "최종 앱 코드"가 아니라 **기획 산출물(PRD)과 개발에 넘길 디자인 산출물**이다.

---

## 두 개의 harness

**아틀리에**(이 작업장)는 **상·하류로 물린 두 harness**로 돌아간다. 같은 `projects/`를 공유하고, 한쪽의 출력이 다른 쪽의 입력이 된다.

```
아이디어 ──/plan──▶ PRD.md ──/design──▶ 와이어 → 검증 → 디자인 시스템 → 하이파이 → 검증 → handoff
          (기획·PM)            (디자인)
```

| harness | 역할 | 입력 → 출력 |
|---|---|---|
| **`/plan`** | 기획·PM (25년차 시니어처럼 캐묻는다) | 아이디어 → **`PRD.md`** |
| **`/design`** | 디자인 빌드·검증 | `PRD.md` → 디자인 산출물 + **handoff 패키지** |

`/plan`이 상류, `/design`이 하류다. 아이디어부터면 `/plan`으로 시작하고, 이미 PRD가 있으면 `/design`으로 바로 들어간다.

---

## 왜 만들었나

좋은 기획·디자인 산출물은 **모델만으로 나오지 않는다.** 무엇을 왜 만드는지 캐묻는 기획 공정, 만들고 → 렌더로 검증하고 → 다시 고치는 루프, 단계마다 멈춰 합의받는 게이트 — 이 **"생성 방식(harness)"** 이 결과의 품질을 만든다.

Atelier는 그 harness를 터미널에 담았다. 설치 한 번으로, 아이디어부터 handoff까지 로컬에서 끝낸다.

---

## `/plan` — 기획 (아이디어 → PRD)

**바로 기획서를 써주지 않는다.** 25년차 PM의 인격으로, 핵심 질문부터 던져 전제·목표를 단단히 잡는다.

- **모호한 말을 그냥 넘기지 않는다** — "사용자 친화적인", "깔끔한"은 구체적으로 무엇인지 되묻는다.
- **"다 중요하다"는 거절한다** — MVP에 무엇이 들어가고 무엇이 빠지는지 분명히 한다.
- 각 기능을 **사용자 문제 + 데이터 관점** 둘 다로 검증한다("이게 없으면 사용자는 어떻게 되나").
- 7단계로 진행하고, 정리되면 정식 PRD를 `projects/<name>/PRD.md`로 쓴다:

```
1 문제 정의 → 2 목표·성공 지표 → 3 타깃 사용자 → 4 핵심 기능·우선순위(MVP)
→ 4-1 개념 데이터 모델 → 5 사용자 흐름·화면 → 6 제약 → 7 착수 전 체크리스트
```

> 개념 데이터 모델까지만 — 물리 스키마(컬럼 타입·인덱스·마이그레이션)는 개발 단계의 결정이라 손대지 않는다.

---

## `/design` — 디자인 (PRD → handoff)

`/design` 스킬 하나가 디자인 파이프라인 전체를 구동한다.

```
PRD ─▶ ① 인테이크 ─▶ ② 와이어(클릭됨) ─▶ ③ ▶흐름·기능 검증
                                                   │
        ⑦ handoff ◀─ ⑥ ▶인터랙션 검증 ◀─ ⑤ 하이파이 ◀─ ④ 디자인 시스템
```

- **단계마다(묶음으로 나눴으면 묶음마다) 사용자 OK**를 받고 넘어간다. 한 번에 다 쏟지 않는다.
- **와이어는 진짜 클릭되고 상태가 변한다** — 페이지끼리 `<a href>`로 잇고, 선택·토글·카운터 같은 상태도 실제로 변한다(밋밋하게). *느낌*만 하이파이 몫.
- **검증은 3층** — 내비게이션(기계)·화면이 책임지는 **기능 완결성**(기계, 죽은 컨트롤 0)·느낌과 "옳은 흐름인가"(사람). 기계 검증은 ③·⑥ 두 군데서 green까지 루프(Playwright).
- **에이전트 동적 편성** — 화면이 많은 단계는 브리프를 읽고 빌더를 병렬로 띄우고, **만들지 않은 독립 에이전트**가 적대적으로 검증한다.

## 구조

Atelier는 단일 프로젝트가 아니라 **여러 프로젝트를 담는 작업장**이다.

```
atelier/                            # (레포. 로컬 폴더명은 달라도 무관)
├── CLAUDE.md                       # 작업장 공통 규칙 (형식·품질의 원본)
├── .claude/skills/
│   ├── plan/                       # /plan   — 기획 harness (→ PRD)
│   └── design/                     # /design — 디자인 harness (PRD → handoff)
├── scripts/test-project.js         # 프로젝트별 Playwright 실행기(검증 증빙 라우팅)
└── projects/
    ├── my-first-app/
    │   ├── PRD.md                  # /plan 산출물 = /design 입력
    │   ├── STATUS.md               # 이 프로젝트가 지금 어느 단계인지 (0 기획 ~ 7 handoff)
    │   ├── 00-flow.md              # 화면·플로우 합의본
    │   ├── wireframe/              # 클릭되는 lo-fi (*.html)
    │   ├── foundation/             # tokens.css + colors/typography
    │   ├── components/             # *.html
    │   ├── screens/                # 하이파이 (*.html)
    │   └── handoff/                # 인계 패키지 (hand-off.md + 비주얼 문서·매핑표·인벤토리)
    ├── my-second-app/
    └── my-third-app/
```

- **프로젝트는 서로 독립.** 토큰·산출물을 공유하지 않는다.
- **`STATUS.md`가 프로젝트의 현재 위치.** `0 기획 → 1 인테이크 → … → 7 handoff`를 한 줄로 추적한다. 파일만으로는 알 수 없는 것(예: "흐름 검증을 사람이 통과시켰는가")을 기록한다. `/plan`이 `0`을 채우고, `/design`이 `1`부터 이어받는다.

## 쓰는 법

> 🚀 **처음이라면 [GETTING-STARTED.md](./GETTING-STARTED.md)** — 설치부터 첫 프로젝트, 검증, handoff까지 단계별 가이드.

작업장 폴더에서 새 세션을 열고, 가진 게 무엇이냐에 따라:

```
/plan      # 아이디어만 있을 때 — 캐물어 PRD까지
/design    # PRD가 있을 때 — 인테이크부터 handoff까지
```

- 두 스킬 모두 **먼저 프로젝트를 고른다** — `projects/`의 각 `STATUS.md`를 읽어 대시보드로 보여주고, 멈춘 단계부터 이어간다. 폴더만 보고 추측하지 않는다.
- 처음이면 → 새 프로젝트 생성부터 안내한다.
- PRD는 `/plan`으로 만들거나, 이미 있으면 `projects/<name>/PRD.md`로 넣고 `/design`을 부른다.

## 산출물 형식

- 화면·컴포넌트마다 **외부 의존성 없는 독립 실행 HTML 1개.** 브라우저로 바로 열린다. (상태 변화용 인라인 JS는 허용 — 금지는 외부 라이브러리지 JS가 아니다.)
- 디자인 토큰은 `foundation/tokens.css`에 **표준 2단**으로: primitive(`--blue-500`·중립 램프) → semantic 역할(`--color-primary`·`--color-text`…). 컴포넌트는 semantic만 참조.
- 각 파일 첫 줄에 카드 마커: `<!-- @dsCard group="Wireframe|Foundation|Components|Screens" -->`
- 이 마커로 산출물을 그룹(Wireframe/Foundation/Components/Screens)별로 분류·탐색한다.

## 품질 기준 (render-check)

산출물을 만들 때마다 스스로 점검하고, 걸리면 통과할 때까지 고친다:

| 항목 | 본다 |
|---|---|
| **thin** | 내용이 빈약한가 (상태·variant·실데이터가 충분한가) |
| **bad** | 레이아웃이 깨졌나 (정렬·간격·오버플로우·반응형) |
| **variantsIdentical** | variant·화면이 사실상 똑같아 보이나 |
| **off-brief** | `00-flow.md`/PRD에서 벗어났나 |
| **deadControl** | 인터랙티브한데 와이어드 안 됨(범위 밖 라벨도 없음) |
| **stateInert** | 눌렀는데 상태가 안 변함 |

## handoff

완성된 디자인(chosen final)을 앱 레포로 넘길 때 — **atelier는 코드를 직접 짜지 않는다.** 대신 *디자인 패키지 + 인계 프롬프트*를 만들고, 받는 쪽 Claude Code가 그걸 보고 **자기 컨벤션**으로 구현한다(특정 프레임워크·아키텍처에 안 묶임):

1. `handoff/` 패키지 — 비주얼 문서(화면 임베드·흐름맵·화면별 스펙)·토큰 매핑표·컴포넌트 인벤토리 (+ PRD가 있으면 참조로 포함).
2. **`hand-off.md`** — 받는 쪽에 붙여넣는 브리프: 어떤 파일을 참고하고 / 무엇을 지키고(흐름·기능·모션·데이터 계약) / 무엇을 만들지.
3. 토큰은 대상의 토큰 시스템에 **매핑**한다. 통째 복붙하지 않는다.
